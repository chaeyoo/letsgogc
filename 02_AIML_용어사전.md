## 0. 큰 그림
> 먼저 숲을 본다. 아래 흐름이 이 문서 전체의 뼈대다.

```
데이터 → (학습) → 모델 → (추론) → 예측/생성
                          │
          [LLM의 경우]     ▼
   사전학습(방대한 텍스트) → 정렬(SFT/RLHF) → 프롬프트로 제어
                          │
        문제: 모델은 "최신·사내 지식"을 모른다 + 환각(hallucination)
                          │
              ┌───────────┴───────────┐
              ▼                        ▼
        [RAG]                     [Agent]
   외부 지식을 검색해            도구(tool)를 호출해
   context에 넣어줌             행동을 수행함
              │                        │
              └──────────┬─────────────┘
                         ▼
                    [MCP]
        모델 ↔ 도구/데이터를 잇는 표준 규격
        (Hey.GC 2.0가 이 구조)
```

**핵심 서사 3줄 (면접용):**
1. LLM은 강력하지만 **사내 최신 지식이 없고 환각을 낸다** → 그래서 **RAG**로 지식을 붙이고 **Agent**로 실제 업무 도구를 붙인다.
2. 그 도구 연결을 **매번 커스텀으로 짜면 확장이 안 되므로** → **MCP**라는 표준 규격으로 붙인다. (= GC의 Hey.GC 2.0)
3. FDE는 이 스택으로 **현업의 반복업무를 실제로 자동화**하는 사람이다.

## 1. 기초 계층
> AI ⊃ ML ⊃ DL ⊃ (Transformer/LLM). 포함 관계를 정확히 말할 수 있어야 한다.

| 용어 | 정의 | 핵심 포인트 |
|---|---|---|
| **AI (인공지능)** | 인간의 지능적 행동을 기계가 수행 | 가장 큰 우산. 규칙기반(if-else)도 넓게는 AI |
| **ML (머신러닝)** | 데이터에서 **패턴(규칙)을 스스로 학습** | "명시적으로 규칙을 코딩하지 않는다"가 핵심 |
| **DL (딥러닝)** | 여러 층의 **신경망(neural net)** 으로 학습 | 특징(feature)을 사람이 안 만들고 모델이 스스로 추출 |
| **생성형 AI (GenAI)** | 새로운 콘텐츠(텍스트·이미지·코드)를 **생성** | 분류/예측(판별형)과 대비되는 개념 |
| **LLM (거대 언어 모델)** | 방대한 텍스트로 학습한 초대형 신경망 | GPT, Claude, Gemini 등. Transformer 기반 |
| **파운데이션 모델(Foundation Model)** | 대규모로 사전학습되어 다양한 작업에 적응 가능한 범용 모델 | LLM은 그 하위 개념. 하나 학습해 여러 태스크에 재활용 |

💬 **판별형(Discriminative) vs 생성형(Generative):**
- 판별형 = "입력이 A냐 B냐" (스팸/정상 분류). 경계선을 학습.
- 생성형 = "그럴듯한 새 데이터를 만든다" (문장 생성). 데이터의 분포를 학습.

## 2. 머신러닝 핵심
> 공고의 "ML/DL 이론 이해"에 직결. 개념의 **정의 + 왜 중요한가**를 말할 수 있어야.

### 2.1 학습 방식 3분류
| 분류 | 정의 | 예시 |
|---|---|---|
| **지도학습(Supervised)** | 정답(label)이 있는 데이터로 학습 | 분류(classification), 회귀(regression) |
| **비지도학습(Unsupervised)** | 정답 없이 구조·군집을 발견 | 클러스터링, 차원축소, **임베딩** |
| **강화학습(RL)** | 보상(reward)을 최대화하는 행동을 학습 | 게임 AI, **RLHF**(LLM 정렬) |
| (준지도/자기지도) | **Self-Supervised**: 데이터 자체에서 정답을 만들어 학습 | LLM 사전학습이 이 방식(다음 단어 맞히기) |

### 2.2 반드시 아는 개념 💬
- **특징(Feature) / 라벨(Label):** 입력 변수 / 정답 값.
- **학습·검증·테스트 분할(Train/Validation/Test split):** 모델을 만들고(train), 튜닝하고(val), 최종 평가(test). 테스트셋은 절대 학습에 안 쓴다.
- **과적합(Overfitting):** 학습 데이터는 잘 맞추는데 **새 데이터는 못 맞춤**. 모델이 데이터를 "외워버림". → 대책: 더 많은 데이터, 정규화(Regularization), 드롭아웃, 조기종료.
- **과소적합(Underfitting):** 모델이 너무 단순해 패턴 자체를 못 잡음.
- **편향-분산 트레이드오프(Bias-Variance):** 너무 단순하면 편향↑(과소적합), 너무 복잡하면 분산↑(과적합). 그 사이 균형점을 찾는 게 ML의 본질.
- **손실함수(Loss Function):** 예측이 정답과 얼마나 틀렸는지 수치화. 이걸 **최소화**하는 방향으로 학습. (예: MSE, Cross-Entropy)
- **경사하강법(Gradient Descent):** 손실을 줄이는 방향(기울기 반대)으로 파라미터를 조금씩 이동. **learning rate(학습률)** = 한 걸음의 크기.
- **하이퍼파라미터(Hyperparameter):** 학습 전에 사람이 정하는 값(학습률, 배치크기, 층 개수). 모델이 학습하는 파라미터(가중치)와 구분.

### 2.3 평가 지표 💬 (제약·헬스케어에선 특히 중요)
| 지표 | 의미 | 언제 중요 |
|---|---|---|
| **Accuracy(정확도)** | 전체 중 맞춘 비율 | 클래스 불균형이면 함정 |
| **Precision(정밀도)** | 양성이라 한 것 중 진짜 양성 | 거짓양성 비용 클 때 (오진 경보) |
| **Recall(재현율)** | 진짜 양성 중 잡아낸 비율 | 놓치면 안 될 때 (질병 검출) |
| **F1 Score** | Precision·Recall의 조화평균 | 둘의 균형 |
| **ROC-AUC** | 임계값 전반의 판별력 | 이진분류 종합 성능 |

> 💊 **헬스케어 맥락:** 질병 스크리닝은 보통 **Recall(놓치면 안 됨)** 우선, 그다음 Precision. "정확도 99%"가 무의미할 수 있다(99%가 건강한 사람이면 다 '정상'이라 찍어도 99%). → 면접에서 이 감각을 보이면 좋다.

## 3. 딥러닝 핵심
> LLM을 이해하려면 신경망의 기본기가 필요.

- **인공신경망(Neural Network):** 입력층-은닉층-출력층으로 구성. 각 연결에 **가중치(weight)** 가 있고 학습으로 조정됨.
- **뉴런/노드:** 입력의 가중합에 **활성화함수**를 적용해 출력.
- **활성화함수(Activation):** 비선형성을 부여해 복잡한 패턴 학습 가능하게 함. (ReLU, GELU, Sigmoid, Softmax)
  - **Softmax:** 출력을 확률분포(합=1)로 변환. LLM이 "다음 단어 확률"을 낼 때 사용.
- **역전파(Backpropagation):** 출력의 오차를 뒤로 전파하며 각 가중치를 얼마나 바꿀지 계산. 딥러닝 학습의 엔진.
- **에폭/배치(Epoch/Batch):** 전체 데이터 1회 학습 = 1 epoch. 데이터를 쪼갠 묶음 = batch.
- **파라미터(Parameter) 규모:** "7B, 70B"의 B = billion(십억). 파라미터=학습된 가중치 수. 많을수록 표현력↑, 비용↑.

**주요 아키텍처 계보 (맥락만):**
- **CNN**(합성곱): 이미지. 지역 패턴 인식.
- **RNN/LSTM**: 순차 데이터(과거). 길어지면 앞 내용을 잊음 → Transformer가 대체.
- **Transformer**: 현재 LLM의 표준. ↓ 다음 장.

## 4. Transformer & LLM 내부
> 💬 면접에서 "Transformer가 뭐냐"는 충분히 나올 수 있다. Attention 한 문장으로 설명할 수 있어야.

### 4.1 왜 Transformer인가
2017년 논문 **"Attention Is All You Need"**. 이전 RNN은 순서대로 처리해 **느리고 긴 문맥을 잊었다.** Transformer는 **모든 토큰을 한 번에 보고, 서로의 관련도를 계산**해서 이 문제를 풀었다. → 병렬처리 가능(학습 빠름) + 긴 문맥 처리.

### 4.2 핵심 부품
| 용어 | 한 문장 설명 |
|---|---|
| **토큰(Token)** | 텍스트를 쪼갠 최소 단위(단어·subword). 모델은 토큰 단위로 처리. "먹었다"→"먹","었다" 식 |
| **토크나이저(Tokenizer)** | 텍스트↔토큰 변환기. (BPE 등). **토큰 수 = 비용·context 한도의 단위** |
| **임베딩(Embedding)** | 토큰을 의미를 담은 숫자 벡터로 변환 (7장 참조) |
| **어텐션(Self-Attention)** ⭐ | 각 토큰이 **다른 어떤 토큰에 주목할지** 가중치를 계산. "그 약은 그것에 효과적"에서 '그것'이 뭘 가리키는지 연결 |
| **Query·Key·Value (Q·K·V)** | 어텐션의 계산 재료. Q(질문)·K(색인)의 유사도로 V(내용)를 가중합 |
| **멀티헤드 어텐션** | 여러 어텐션을 병렬로 = 문법·의미 등 여러 관계를 동시에 포착 |
| **위치 인코딩(Positional Encoding)** | 순서 정보가 없는 어텐션에 "몇 번째 토큰인지"를 주입 |
| **디코더(Decoder-only)** | GPT·Claude류 구조. 앞 토큰들로 **다음 토큰을 예측**하며 생성 |

### 4.3 반드시 아는 3개념 💬
- **컨텍스트 윈도우(Context Window):** 모델이 한 번에 볼 수 있는 토큰 최대치(입력+출력). 초과하면 잘림. RAG·긴 대화의 제약 요인. (요즘 수십만 토큰까지)
- **자기회귀 생성(Autoregressive):** 한 토큰씩 → 그걸 다시 입력에 붙여 → 다음 토큰. 그래서 긴 출력은 느리고, 앞의 실수가 뒤에 누적될 수 있음.
- **환각(Hallucination):** 사실이 아닌데 그럴듯하게 지어냄. **원인:** 모델은 "사실 DB"가 아니라 "다음 토큰 확률기"라서. → **RAG가 이걸 완화하는 핵심 이유.**

## 5. LLM 학습 파이프라인
> "어떻게 이런 모델이 만들어지나"의 3단계. RLHF·SFT는 면접 단골.

```
① 사전학습(Pre-training)  ── 인터넷 규모 텍스트로 "다음 단어 맞히기"
        ▼                    → 언어·세상 지식을 습득 (self-supervised)
② 지도 미세조정(SFT)       ── 좋은 질문-답변 예시로 "지시를 따르게" 훈련
        ▼                    → Instruction tuning
③ 정렬(RLHF/DPO)          ── 사람 선호(좋은 답 vs 나쁜 답)로 미세조정
                             → 도움되고·안전하고·정직하게
```

| 용어 | 설명 |
|---|---|
| **사전학습(Pre-training)** | 방대한 텍스트로 기본 언어능력·지식 습득. 가장 비쌈. **여기서 배운 지식이 곧 모델의 "지식 컷오프(knowledge cutoff)"** |
| **미세조정(Fine-tuning)** | 사전학습 모델을 특정 데이터/작업에 맞게 추가 학습 |
| **SFT (Supervised Fine-Tuning)** | 사람이 만든 모범 답안으로 "지시 따르기"를 학습 |
| **RLHF** | Reinforcement Learning from Human Feedback. 사람이 답변에 순위를 매긴 걸 보상모델로 만들어 강화학습. ChatGPT를 "쓸만하게" 만든 기술 |
| **DPO** | Direct Preference Optimization. RLHF보다 간단하게 선호를 직접 최적화하는 최신 대안 |
| **지식 컷오프(Knowledge Cutoff)** | 학습 데이터가 어느 시점까지인지. 그 이후·사내 정보는 모름 → **RAG 필요성의 근거** |

### 5.1 파라미터를 안 건드리는 미세조정 (실무 중요) 🧪
전체 재학습은 비싸다. 그래서:
- **PEFT (Parameter-Efficient Fine-Tuning):** 일부만 효율적으로 조정.
- **LoRA / QLoRA:** 원본은 얼려두고 작은 어댑터 행렬만 학습. 저비용 커스터마이징의 표준.
- **양자화(Quantization):** 가중치를 저정밀도(예: 16bit→4bit)로 줄여 메모리·속도 확보. 정확도 약간 희생.
- **증류(Distillation):** 큰 모델(교사)의 행동을 작은 모델(학생)이 모방 → 경량화.

💬 **면접 감각:** "사내 도메인 적응, Fine-tuning할까 RAG할까?" → **먼저 RAG**(빠르고·업데이트 쉽고·출처추적 가능), 스타일·형식 고정이 필요하면 그때 Fine-tuning. FDE 관점에선 대개 RAG+프롬프트가 먼저.

## 6. LLM 추론과 제어
> 학습된 모델을 "쓸 때" 조절하는 손잡이들. API 파라미터로 직접 만짐. 🧪

### 6.1 디코딩 파라미터
| 파라미터 | 역할 | 실무 팁 |
|---|---|---|
| **Temperature** | 무작위성. 0=결정적/일관, 높을수록 창의·발산 | 사내 사실검색·추출은 **0~0.3**, 브레인스토밍은 0.7+ |
| **Top-p (Nucleus)** | 누적확률 p까지의 후보에서만 샘플링 | temperature와 함께 다양성 제어 |
| **Top-k** | 상위 k개 후보에서만 샘플링 | |
| **Max tokens** | 최대 출력 길이 | 비용·잘림 방지 |
| **Stop sequence** | 이 문자열 나오면 생성 중단 | 구조화 출력 경계 제어 |
| **Seed** | 같은 seed면 재현 가능(가능한 API 한정) | 테스트 재현성 |

### 6.2 프롬프트 엔지니어링 💬
| 기법 | 설명 |
|---|---|
| **System / User / Assistant** | 역할 구분. System=페르소나·규칙, User=요청, Assistant=응답 |
| **Zero-shot** | 예시 없이 지시만 |
| **Few-shot** | 예시 몇 개를 함께 줘서 형식·품질 유도 (In-context learning) |
| **Chain-of-Thought (CoT)** | "단계적으로 생각해봐" → 추론 정확도↑ |
| **In-context Learning** | 파라미터 변경 없이, **프롬프트 안의 예시만으로** 즉석 학습하는 능력 |
| **구조화 출력(Structured Output)** ⭐ | JSON 등 정해진 스키마로 답하게 강제. Function Calling·PydanticAI의 기반 |
| **Grounding** | 답변을 **주어진 근거(context)에 기반**하게 함 = RAG의 목적 |

- **컨텍스트 엔지니어링(Context Engineering):** 프롬프트 문구뿐 아니라 **모델에 무엇을(RAG 결과·도구정의·이력) 어떤 순서로 채워 넣을지** 설계하는 상위 개념. 요즘 FDE 실무의 핵심어.

## 7. 임베딩과 벡터 검색
> ⭐ RAG의 심장. "왜 벡터로 검색하나"를 설명할 수 있어야.

| 용어 | 설명 |
|---|---|
| **임베딩(Embedding)** | 텍스트를 **의미를 담은 고차원 벡터**(예: 1536차원)로 변환. 의미가 비슷하면 벡터도 가깝다 |
| **의미 검색(Semantic Search)** | 키워드 일치가 아니라 **의미 유사도**로 검색. "감기약"으로 "해열제"도 찾음 |
| **코사인 유사도(Cosine Similarity)** | 두 벡터의 방향 유사도(-1~1). 임베딩 근접도 측정의 표준 |
| **벡터 데이터베이스(Vector DB)** | 임베딩을 저장·고속 유사검색. 예: Pinecone, Weaviate, Qdrant, Chroma, pgvector, FAISS |
| **ANN (근사최근접이웃)** | 수백만 벡터에서 "정확히"가 아니라 "거의 가까운" 걸 빠르게 찾는 알고리즘(HNSW 등) |
| **인덱싱(Indexing)** | 검색 빠르게 하려고 벡터를 미리 구조화(HNSW, IVF) |

💬 **키워드 검색(BM25) vs 벡터 검색:** 키워드는 정확한 용어·고유명사(약품코드)에 강하고, 벡터는 의미·동의어에 강하다. → 실무에선 **하이브리드**(둘 다)가 정답인 경우가 많다(8.4 참조).

## 8. RAG 심화 ⭐
> **공고 1순위 요구.** RAG는 "임베딩 넣고 검색" 그 이상이다. **최적화 포인트**를 말할 수 있어야 차별화된다.

### 8.1 RAG란
**Retrieval-Augmented Generation.** 질문에 답하기 전에 **외부 지식을 검색해 프롬프트에 넣어주는** 방식.
- **왜:** ①환각 감소(근거 기반) ②최신·사내 지식 반영(학습 안 해도 됨) ③**출처 추적 가능**(제약·규제 산업에 필수) ④Fine-tuning보다 싸고 빠름.

### 8.2 파이프라인 (2단계)
```
[인덱싱 단계 — 미리]                  [질의 단계 — 실시간]
문서 → 청킹 → 임베딩 → 벡터DB저장       질문 → 임베딩 → 검색(retrieve)
                                        → 리랭킹 → context 조립
                                        → LLM에 질문+근거 전달 → 답변(+출처)
```

### 8.3 각 단계 세부 용어 🧪
| 단계 | 용어 | 핵심 |
|---|---|---|
| 로딩 | **Document Loader** | PDF·HWP·DB 등에서 원문 추출(OCR 포함) |
| 분할 | **청킹(Chunking)** | 문서를 검색 단위로 쪼갬. **크기·겹침(overlap)** 이 품질 좌우 |
| 분할 | **청크 전략** | 고정크기 / 문장·문단 / 의미기반(semantic) / 구조기반(제목단위) |
| 임베딩 | **임베딩 모델** | 문서·질문을 같은 공간의 벡터로 |
| 저장 | **Vector DB + 메타데이터** | 벡터 + 출처·날짜·부서 등 필터용 메타 |
| 검색 | **Top-k Retrieval** | 유사도 상위 k개 청크 회수 |
| 검색 | **메타데이터 필터링** | "2025년·규제팀 문서만" 같은 조건 결합 |

### 8.4 RAG "최적화" (여기가 면접 차별화) ⭐⭐
- **하이브리드 검색(Hybrid Search):** 벡터(의미) + 키워드(BM25)를 결합. 고유명사·코드에 강해짐.
- **리랭킹(Re-ranking):** 1차로 많이(예: 50개) 회수 후, **Cross-Encoder 리랭커**로 진짜 관련순 재정렬해 상위 몇 개만 LLM에. 정밀도↑.
  - *Bi-Encoder(임베딩)=빠르지만 덜 정밀 / Cross-Encoder(리랭커)=느리지만 정밀* → 2단계로 장점만.
- **쿼리 변환(Query Transformation):** 질문을 다시 씀. **HyDE**(가상의 답을 생성해 그걸로 검색), **Multi-query**(질문을 여러 각도로 확장), **Query decomposition**(복합질문 분해).
- **청킹 튜닝:** 너무 작으면 문맥 손실, 너무 크면 노이즈. 겹침(overlap)으로 경계 손실 방지.
- **Contextual Retrieval:** 각 청크에 "이 청크가 문서 어디 맥락인지" 요약을 붙여 검색 정확도↑ (Anthropic이 제안한 기법).
- **컨텍스트 압축(Context Compression):** 회수한 청크에서 관련 문장만 추려 토큰 절약.
- **RAG 평가 지표:**
  - **Retrieval:** Recall@k, Precision@k, MRR, NDCG (관련 문서를 잘·상위에 회수했나)
  - **Generation:** **Faithfulness(근거 충실도=환각 여부)**, **Answer Relevancy(질문 적합성)**, **Context Precision/Recall**
  - 도구: **RAGAS**, LangSmith 등.

### 8.5 확장 개념 💬
- **GraphRAG:** 벡터 대신/함께 **지식그래프**(엔티티-관계)로 검색. 복잡한 관계 질의에 강함.
- **Agentic RAG:** 에이전트가 "검색할지·다시 검색할지·어느 도구 쓸지" **스스로 판단**하는 RAG (9장과 결합).
- **RAG vs Fine-tuning vs Long-context:** RAG=지식 주입/출처추적, FT=행동·형식 고정, Long-context=문서를 통째로 프롬프트에(짧은 문서엔 편하나 비싸고 산만).

> 💊 **GC 연결점:** "규제문서 검색, SOP·실험프로토콜 요약, 논문·부작용 정보 검색" → 전형적 RAG 유스케이스. 출처추적이 필수인 제약산업에서 RAG의 가치가 특히 크다.

## 9. Agentic AI ⭐
> **공고 2순위.** "LLM이 생각만 하는 것"에서 "행동하는 것"으로. Agentic Workflow·Function Calling·Subagent가 여기.

### 9.1 핵심 개념
| 용어 | 설명 |
|---|---|
| **에이전트(Agent)** | LLM이 **목표를 받아, 스스로 계획하고, 도구를 호출해, 관찰하고, 반복**하며 문제를 푸는 시스템 |
| **툴/도구(Tool)** | 에이전트가 부를 수 있는 외부 기능(검색·API·DB·계산·이메일 발송 등) |
| **Function Calling** ⭐ | LLM이 "이 함수를 이런 인자로 불러줘"라고 **구조화된(JSON) 호출**을 출력. 실제 실행은 외부 코드가. LLM↔코드의 다리 |
| **Tool Use** | Function Calling의 넓은 이름. "모델이 도구를 쓴다" |
| **에이전트 루프** | **관찰(Observe)→생각(Think)→행동(Act)** 반복. 목표 달성/한계까지 |
| **ReAct** | Reasoning + Acting. "추론↔행동"을 번갈아 (생각→도구호출→결과보고 다시 생각). 대표 에이전트 패턴 |
| **Planning** | 목표를 하위 단계로 분해해 계획 수립 |
| **Memory** | 단기(대화 맥락) + 장기(과거 상호작용을 벡터DB 등에 저장·회수) |
| **Reflection / Self-critique** | 자기 결과를 스스로 검토·수정 |
| **Human-in-the-loop (HITL)** | 위험한 행동(이메일 발송·삭제) 전 **사람 승인**을 받는 관문. 엔터프라이즈 필수 |

### 9.2 멀티 에이전트 & Subagent ⭐ (공고 명시)
- **Subagent(서브에이전트):** 큰 문제를 쪼개 **하위 전문 에이전트**에게 위임. 각자 독립 컨텍스트로 일하고 결과만 보고 → 상위(orchestrator)가 취합.
- **Orchestrator-Worker 패턴:** 총괄 에이전트가 작업을 분배·조율.
- **왜 나누나:** ①컨텍스트 한도 회피(각자 자기 맥락만) ②전문화(검색용/코딩용/검증용) ③병렬 처리.
- **Workflow vs Agent 💬:** **Workflow**=경로가 코드로 정해진 파이프라인(예측가능·안정). **Agent**=경로를 LLM이 런타임에 결정(유연·비결정적). 실무는 **"가능한 한 workflow, 필요한 곳만 agent"** 가 신뢰성의 정석.

### 9.3 Agentic Workflow 대표 패턴 💬
- **Prompt Chaining:** 단계를 순차 연결(추출→요약→번역).
- **Routing:** 입력 유형에 따라 다른 경로로 분기(문의 분류 후 담당 로직).
- **Parallelization:** 여러 하위작업 동시 실행 후 취합.
- **Evaluator-Optimizer:** 생성→평가→개선 루프.
- **Orchestrator-Workers:** 9.2 참조.

> 💊 **GC 연결점:** "회의실 예약·방문관리·이메일·캘린더를 에이전트가 자율 처리"(Hey.GC 2.0) = 전형적 Agentic Workflow + Function Calling + HITL. 이 문서 9·10장이 곧 그 제품의 원리.

## 10. MCP ⭐⭐ (GC 핵심)
> **Hey.GC 2.0의 구조가 바로 MCP.** 공고·회사분석 양쪽에 등장 → 여기를 가장 확실히 이해할 것.

### 10.1 MCP란
**Model Context Protocol.** Anthropic이 제안한 **개방형 표준**. LLM(에이전트)과 외부 도구·데이터를 잇는 **"USB-C 규격"** 이라 비유된다.
- **왜 필요한가 💬:** 도구가 N개, 모델·앱이 M개면 연결이 **N×M**로 폭발한다. MCP라는 공통 규격을 두면 **N+M**로 준다. 한 번 MCP 서버로 만든 도구를 어느 MCP 클라이언트(Claude, Cursor, 사내 에이전트)든 재사용.
- **표준화 이전:** 툴마다 커스텀 통합 코드 → 유지보수 지옥. MCP가 이걸 규격화.

### 10.2 구조
```
[MCP Host/Client]  ←→  [MCP Server]  ←→  실제 시스템
 (Claude, Cursor,      (도구·데이터를      (이메일, 캘린더,
  사내 에이전트)         MCP 규격으로 노출)   DB, 사내 API)
```

| 구성요소 | 역할 |
|---|---|
| **Host / Client** | LLM 앱 쪽. MCP 서버에 연결해 도구를 사용 |
| **Server** | 도구·데이터를 MCP 규격으로 제공하는 쪽. **네가 FDE로서 만들 것** |
| **Transport** | 통신 방식(stdio=로컬 프로세스, HTTP/SSE=원격) |

### 10.3 MCP가 노출하는 3가지 (꼭 암기) ⭐
| Primitive | 뜻 | 예 |
|---|---|---|
| **Tools** | 에이전트가 **실행**할 수 있는 함수(행동) | 이메일 보내기, 회의실 예약, DB 조회 |
| **Resources** | 읽을 수 있는 **데이터/문맥** | 파일, 문서, DB 레코드 |
| **Prompts** | 재사용 가능한 **프롬프트 템플릿** | "이 형식으로 회의록 요약" |

### 10.4 FastMCP ⭐ (공고 명시 스택) 🧪
- **FastMCP:** 파이썬으로 **MCP 서버를 빠르게** 만드는 프레임워크(FastAPI 감성). 데코레이터로 함수를 도구화.
- 감 잡는 예:
  ```python
  from fastmcp import FastMCP
  mcp = FastMCP("사내 도우미")

  @mcp.tool()
  def search_sop(query: str) -> str:
      """사내 SOP 문서를 검색한다."""   # ← 이 docstring을 LLM이 읽고 언제 쓸지 판단
      return vector_db.search(query)

  @mcp.resource("employee://{emp_id}")
  def get_employee(emp_id: str) -> dict:
      return hr_system.lookup(emp_id)
  ```
- 💬 **핵심 감각:** 도구의 **이름·docstring·타입힌트**가 곧 LLM에게 주는 "설명서"다. 잘 쓴 docstring = 에이전트가 도구를 잘 고름. (FDE의 실력이 드러나는 지점)

> 🎯 **너의 프로젝트(계획서 3장)가 정확히 이것:** "사내 문서 RAG + 툴을 MCP로 연결한 Agentic 어시스턴트" = 10장 + 8장 + 9장의 결합. 이 문서를 프로젝트 README 이론 배경으로 재활용 가능.

## 11. 실무 프레임워크 스택
> 공고 자격요건에 나열된 도구들. **각각 "무엇을 푸는가"** 를 한 줄로.

| 도구 | 한 줄 정체 | 언제 |
|---|---|---|
| **LangChain** | LLM 앱 부품(로더·프롬프트·체인·툴)의 표준 라이브러리 | RAG·체인 빠르게 조립 |
| **LangGraph** ⭐ | **그래프(노드·엣지·상태)** 로 에이전트 워크플로우를 짜는 프레임워크 | 순환·분기·HITL 있는 **상태기반 에이전트**. "workflow 명시적 제어"에 강함 |
| **PydanticAI** ⭐ | **타입 안전**하게 LLM 앱을 짜는 프레임워크(Pydantic 기반) | 구조화 출력·의존성 주입·검증이 중요할 때. FastAPI 쓰던 사람에게 친숙 |
| **Pydantic** | 파이썬 데이터 검증·스키마 라이브러리 | 구조화 출력 스키마 정의의 기반 |
| **FastAPI** ⭐ | 고성능 파이썬 웹/API 프레임워크 | 에이전트·RAG를 API로 서빙(백엔드) |
| **FastMCP** ⭐ | MCP 서버 제작 프레임워크 (10.4) | 사내 시스템을 도구로 노출 |
| **Enterprise LLM API** | Claude·OpenAI 등 상용 모델 API | 실제 추론 엔진 |
| **Vector DB** | Chroma·Qdrant·pgvector 등 | RAG 저장소 |
| **LangSmith / Langfuse** | LLM 앱 관측·추적·평가 도구 | 디버깅·품질측정(12장) |
| **Claude Code / Cursor** | AI 페어프로그래밍 도구 | 개발 생산성 어필(공고 명시) |

💬 **LangGraph vs PydanticAI (골라 쓰는 감):**
- **LangGraph** = 복잡한 **다단계·상태·순환** 워크플로우 제어에 강함(그래프로 흐름을 눈에 보이게).
- **PydanticAI** = **타입 안전·구조화 출력·검증** 중심, 가볍고 파이썬스러움.
- FDE 관점: 흐름이 복잡하면 LangGraph, 견고한 단일 에이전트/구조화 응답이면 PydanticAI. 정답은 없고 **문제에 맞게 고른다**고 말하면 좋다.

## 12. LLMOps · 평가
> 만드는 것보다 **믿을 수 있게 운영**하는 게 엔터프라이즈의 핵심. 제약산업은 신뢰성·추적성이 특히 중요.

| 용어 | 설명 |
|---|---|
| **관측성(Observability) / Tracing** | 에이전트가 어떤 도구를 왜 호출했는지 **전 과정 추적**(LangSmith 등). 디버깅·감사 필수 |
| **평가(Evaluation / Eval)** | 출력 품질을 체계적으로 측정. 오프라인(테스트셋) + 온라인(실사용) |
| **LLM-as-a-Judge** | 다른 LLM으로 출력 품질을 채점. 사람 평가 보완 |
| **가드레일(Guardrails)** | 입력·출력을 규칙/모델로 검증(개인정보·유해·형식 위반 차단) |
| **환각 완화** | RAG·인용 강제·"모르면 모른다고" 프롬프트·검증단계 |
| **프롬프트 인젝션(Prompt Injection)** | 악의적 입력이 시스템 지시를 덮어쓰는 공격. 신뢰경계·도구권한 최소화로 방어 |
| **비용·지연 관리** | 토큰=돈. 캐싱·모델 라우팅(쉬운 건 작은 모델)·컨텍스트 압축 |
| **프롬프트 캐싱(Prompt Caching)** | 반복되는 앞부분(시스템·문서)을 캐시해 비용·지연 절감 |
| **재현성·버저닝** | 프롬프트·모델·데이터 버전 관리(규제 대응) |

> 💊 **헬스케어 규제 감각:** 답변의 **출처·근거·처리이력이 추적 가능**해야 하고(감사), **HITL로 위험행동 승인**, **개인정보(환자·직원) 가드레일**이 필수. 면접에서 "제약사라서 신뢰성·추적성·승인관문을 이렇게 설계하겠다"고 말하면 강력하다.

## 13. 치트시트
> 마감 전날 30초 복습용. 한 줄로 못 말하면 다시 본문으로.

- **AI⊃ML⊃DL⊃Transformer/LLM** — 포함관계.
- **과적합** = 학습데이터만 잘 맞춤 → 정규화·데이터↑로 대응.
- **Transformer** = "모든 토큰이 서로 얼마나 관련있나(Attention)"로 언어 이해.
- **토큰/컨텍스트윈도우** = 처리·비용·한도의 단위.
- **환각** = 사실 아닌 그럴듯한 생성 → RAG로 완화.
- **사전학습→SFT→RLHF** = LLM이 유용·안전해지는 3단계.
- **Temperature** = 낮으면 일관, 높으면 창의.
- **임베딩** = 의미를 담은 벡터, 가까우면 비슷함 → 의미검색.
- **RAG** = 검색해서 근거를 넣고 답함(출처추적·최신지식).
- **RAG 최적화** = 청킹·하이브리드검색·리랭킹·쿼리변환·평가(RAGAS).
- **Function Calling** = LLM이 JSON으로 "이 함수 불러줘".
- **Agent** = 관찰→생각→행동(도구호출) 반복. **ReAct**가 대표패턴.
- **Subagent** = 문제를 쪼개 전문 하위 에이전트에 위임.
- **MCP** = 모델↔도구의 표준규격(USB-C). **Tools/Resources/Prompts** 노출. **FastMCP**로 서버 제작.
- **LangGraph** = 그래프 상태기반 에이전트. **PydanticAI** = 타입안전 LLM 앱. **FastAPI** = 서빙.
- **HITL·가드레일·Tracing** = 엔터프라이즈 신뢰성 3종.

## 14. 면접 예상 질문
> 각 항목: **질문 → 답변 뼈대(키워드)**. 실제론 네 프로젝트 경험으로 살을 붙일 것.

1. **"RAG가 뭐고 왜 쓰나요?"**
   → 외부지식 검색해 프롬프트에 주입 → 환각↓·최신/사내지식·출처추적·FT보다 저렴. 제약산업은 출처추적이 필수라 특히 적합.

2. **"RAG 성능이 안 나오면 어디를 손보겠어요?"** (차별화 질문)
   → 단계별로: 청킹 전략(크기·겹침) → 하이브리드 검색 → **리랭킹** → 쿼리 변환(HyDE·멀티쿼리) → 평가(RAGAS: faithfulness/context recall)로 어느 단계가 문제인지 진단.

3. **"Agent와 Workflow 차이는?"**
   → Workflow=경로가 코드로 고정(안정·예측), Agent=경로를 LLM이 런타임 결정(유연·비결정). **가능한 workflow, 필요한 곳만 agent** 가 신뢰성 원칙.

4. **"Function Calling을 설명해보세요."**
   → LLM이 실행은 안 하고, "이 도구를 이 인자로"라는 구조화(JSON) 호출을 출력 → 외부 코드가 실행 → 결과를 다시 LLM에. LLM과 실제 시스템의 다리.

5. **"MCP가 뭔가요? 왜 중요하죠?"** (GC 저격)
   → 모델↔도구 연결의 개방표준. N×M 통합을 N+M으로. Tools/Resources/Prompts를 노출. **Hey.GC 2.0가 MCP 구조** → 한 번 만든 사내 도구를 여러 에이전트가 재사용. FastMCP로 서버를 만든다.

6. **"환각을 어떻게 줄이죠?"**
   → RAG로 근거 제공 + 인용 강제 + "근거 없으면 모른다고" + 검증단계(LLM-as-judge/가드레일) + 낮은 temperature.

7. **"Fine-tuning vs RAG, 언제 뭘?"**
   → 지식·최신성·출처추적=RAG(먼저), 말투·형식·특정행동 고정=Fine-tuning. 대개 RAG+프롬프트로 시작.

8. **"제약사에서 AI 신뢰성을 어떻게 확보하겠어요?"** (헬스케어 저격)
   → 출처추적(RAG+인용), HITL 승인관문, 개인정보 가드레일, 전과정 Tracing/감사로그, 프롬프트·모델 버저닝. "정확도"보다 **Recall·추적성** 중심.

9. **"ML에서 과적합이 뭐고 어떻게 막나요?"** (기초 확인)
   → 학습데이터만 과하게 맞춰 일반화 실패. 데이터 증강·정규화·드롭아웃·조기종료·교차검증.

10. **"LangGraph를 왜 쓰나요?"**
    → 분기·순환·상태·HITL 있는 복잡 에이전트 흐름을 그래프로 명시적·안정적으로 제어. (단순 구조화 응답이면 PydanticAI로 충분하다고 비교까지.)
