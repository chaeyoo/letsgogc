## Part 1. 검색 정확도 0.875 → 1.0 의 원리

### 1-1. 평가 체계 — `project/eval/evaluate.py` + `eval/qa_dataset.json`

**정답은 미리 정해져 있다 (골드 라벨).** 평가셋은 시험 문제 + 정답지다:

```json
{
 "question": "신약 품목허가 심사는 며칠 이내에 처리되나요?",  ← 문제
 "relevant_doc_id": "REG-001",                              ← 정답 문서 (사람이 라벨링)
 "keywords": ["120", "근무일"],                              ← 답변 재료 핵심어
 "hard_negative": true (일부 문항)                           ← 헷갈리는 오답 문서가 존재하는 어려운 문항 표시
}
```

- **32문항**, 그중 **하드네거티브 14문항**. RA·PV 도메인 질문으로 구성.
- **Hit@1** ("힛 앳 원"): 정답 문서가 검색 결과 **1위**로 나온 비율. `_hit_rank()`가 순위 판정.
- **Hit@3** ("힛 앳 쓰리"): 정답 문서가 **상위 3개 안에** 들어온 비율.
- **MRR**: 정답의 첫 등장 순위의 역수 평균 (1위=1.0, 2위=0.5, 3위=0.33…).
- **ContextRecall**: 검색기가 돌려준 문단들 안에 keywords 가 존재하는 비율.
  - "검색된 문단" = 정답 여부와 무관하게 **검색기가 상위로 돌려준 문단들**.
  - 의미: 생성(LLM)이 정답을 말할 **재료**를 검색이 제공했는가 → 답변 신뢰도(faithfulness)의 상한.
  - 검색의 주체: 평가 때는 평가 스크립트가 검색기(retriever)를 직접 호출. 운영 때는 에이전트가 MCP 도구(`search_regulations`)를 통해 같은 검색기를 호출.
- **0.875의 뜻**: ①벡터 단독 모드의 Hit@1 = 32문항 중 **28개 정답 (28/32 = 0.875)**. 즉 4문항 실패 — 이 4개(및 이후 단계의 잔여 실패)를 분석한 것이 개선 루프의 출발점.

### 1-2. 단계별 비교 (①→④) — 이 과정의 이름

`evaluate.py main()`이 같은 문항을 4가지 구성으로 풀게 해 비교한다:

| 단계 | 구성 | 잡는 실패 유형 |
|---|---|---|
| ① 벡터만 | TF-IDF 코사인 단독 | (베이스라인 0.875) |
| ② 하이브리드 | + BM25 키워드 검색 | 코드·고유명사 등 **정확 용어를 못 잡는 실패** |
| ③ +리랭킹 | + 4신호 재정렬 | **어휘가 겹치는 오답 문서(하드네거티브)가 1위를 뺏는 실패** |
| ④ +질의확장 | + 동의어 사전 | 사용자 구어와 문서 용어가 **아예 안 겹치는 실패 (어휘 불일치)** |

**이 과정의 공식 명칭: ablation(애블레이션) / ablation study — 구성요소 절제 실험.**
구성요소를 하나씩 빼거나 더해 각각의 기여도를 수치로 분리하는 실험 방법. 면접에서는 "구성요소별 기여를 분리한 ablation 비교" 또는 쉽게 "단계별 성능 비교 실험"이라 말하면 된다.
중요한 건 순서 암기가 아니라 **"실패 문항 분석 → 원인별 처방 → 재측정" 반복 루프** — 자소서의 "반복마다 코드와 평가 수치를 직접 분석하며 원리를 소화"가 이 루프다.

### 1-3. 하이브리드 검색 — `src/rag/retriever.py` `_hybrid()` (234행)

**BM25와 벡터 검색은 둘 다 "문서 순위를 매기는 검색 알고리즘"이다:**

- **BM25 (키워드 검색)**: 질의 단어가 문서에 몇 번 나오는지(TF) × 그 단어가 얼마나 희귀한지(IDF)로 점수화. 단어가 **정확히 일치**해야 점수가 남. Elasticsearch 등의 기본 랭킹.
- **벡터 검색 (의미 검색)**: 질의·문서를 숫자 벡터로 바꿔 **방향 유사도(코사인)**를 봄. 단어가 달라도 의미가 비슷하면 점수가 나는 대신, 코드·고유명사에 둔함.

**왜 0~1 정규화 후 0.5 가중합인가:**
두 점수는 **스케일(자릿수)이 다르다.** 코사인은 대략 0~1, BM25 는 상한 없는 값(수 점~수십 점). 날것으로 더하면 BM25 가 결과를 독식한다 — 100점 만점 과목과 10점 만점 과목 점수를 그냥 더하는 것과 같다. 그래서 각자 min-max 로 0~1 로 맞춘 뒤(`_minmax()`), alpha=0.5 로 **두 신호를 동등하게 신뢰**해 평균낸다. (0.5는 스윕으로 재보정 가능한 하이퍼파라미터.)
정규화는 반드시 "버전 필터를 통과한 후보군 안에서" 수행 — 전체 코퍼스 기준으로 하면 스케일이 왜곡된다(240행 주석).

### 1-4. 리랭킹 — `_rerank_score()` (288행) + `retrieve()` (388행)

**잡는 실패 = 하드네거티브.** 코퍼스에는 의약품 규정과 화장품 표시·광고 규정이 함께 있는데, 화장품 규정 안의 **"의약품 표시기재와의 차이" 섹션**이 의약품 어휘를 통째로 인용한다. 의약품 표시기재를 물으면 정답(의약품 문서) 대신 이 화장품 문서 섹션이 어휘 겹침으로 1위를 뺏는다. 이런 "정답처럼 보이는 오답"이 하드네거티브.

**4가지 신호의 가중합** — 옆의 소수점은 각 신호의 중요도 비율(합 1.0):

```
재점수 = 0.55×coverage + 0.20×phrase + 0.15×section + 0.10×title
```

- **coverage 0.55** — "청크(문단)가 질의 토큰을 얼마나 덮는가" = **질의를 단어(토큰)로 쪼갰을 때, 그중 몇 %가 이 문단 안에 존재하는가** (희귀 단어일수록 IDF 로 더 크게 침).
  예: 질의 토큰 {신약, 품목허가, 심사, 기간} 중 문단에 3개 존재 → 대략 0.75 (실제로는 IDF 가중 비율).
- **phrase 0.20** — 질의 원문이 통째로 문단에 등장하는가 (0 또는 1).
- **section 0.15** — 질의 토큰이 **섹션 제목**에 있는가 (같은 문서 안에서 정답 섹션 고르기).
- **title 0.10** — **문서 제목**이 질의와 정합하는가. 하드네거티브의 반증 신호: 본문 어휘는 겹쳐도 제목("화장품 표시·광고")은 질의와 안 겹친다.

**+ 섹션 타입 prior (370~385행):** "~와의 차이" 대조 섹션은 질의가 비교를 묻지 않는 한 감점(-0.3), "목적/개요" 서두 섹션은 정의를 묻지 않는 한 감쇠(-0.055). **질의 의도로 게이트** — 비교 질문에는 대조 섹션이 정답이므로 페널티를 끈다. 가중치 재조정만으로는 하드네거티브가 안 뒤집혀 **신호를 추가**해 푼 사례(261~265행 주석).

**+ 1차 점수와 9:1 블렌딩 (423~431행):**
`최종 = 0.9 × (4신호 재점수) + 0.1 × (1차 하이브리드 점수)` — **9가 리랭커 재점수, 1이 1차 점수.** 순수 재정렬만 쓰면 쉬운 질의가 오히려 떨어지는 일이 있어, 1차 점수를 사전 신뢰(prior)로 섞어 안정화한다. 실무 Cross-Encoder 리랭커도 같은 관행.

### 1-5. 질의확장 — `src/rag/synonyms.py`

**잡는 실패 = 어휘 불일치.** 사용자는 "부작용이 심각하게…"라 묻지만 문서는 "중대한 이상사례"라 쓴다. 질의 단어가 문서에 하나도 없으면 ①②③이 전부 무력하다.

**로직:** 사전(`SYNONYMS`)의 표제어가 질의에 등장하면 정식 용어를 질의 **뒤에 덧붙인다**(append — 원 질의는 그대로 보존).
`"부작용이 심각한가요?"` → `"부작용이 심각한가요? 이상사례 이상반응 중대한"`

**설계 규율 3가지 (풀어서):**

1. **단방향 (구어→정식 용어만).**
   사전이 "부작용→이상사례"만 있고 "이상사례→부작용"은 없다. 양방향이나 연쇄(부작용→이상사례→이상반응→…)를 허용하면 확장어가 눈덩이처럼 붙어 질의가 원래 주제에서 멀어진다 = **주제 표류(topic drift)**. 사용자 말은 다양하지만 문서 용어는 고정이므로, 한 방향이면 충분하다.

2. **회수/정밀 차등 가중.**
   확장어는 어디까지나 시스템의 **추측**이다. 1단계(넓게 회수)에서는 전 가중으로 써서 그물을 넓히되, 2단계(정밀 재정렬)에서는 **절반 가중의 보조 신호**로만 쓴다 — 확장어가 사용자의 실제 단어와 동급이면, 추측 단어만 맞춘 문서가 실제 단어를 맞춘 문서를 이길 수 있기 때문(정밀도 희석). 그렇다고 리랭킹에서 아예 빼면, 질의 단어가 문서에 전무한 문항에서 리랭커가 판별력을 잃는다.
   근거 수치: **aux(보조 신호) 유무 = Hit@1 0.969 vs 0.906** (retriever.py 408행). 이런 "기능 하나 껐다 켰다 비교"가 ablation.

3. **매칭 경계 — 질의 속에서 표제어를 '어떻게 찾을 것인가'의 규칙.**
   - **영문 표제어는 단어 경계 필수**: 부분 문자열로 찾으면 "audit **find**ing" 속에서 "ind"(임상시험계획 IND)가, "PVC 포장재" 속에서 "pv"(약물감시)가 잘못 발화해 엉뚱한 확장이 붙는다.
   - **한글 표제어는 부분 문자열이 정답**: 한국어는 활용·조사가 직결되므로 "심각**하게**"에서 "심각"이 발화해야 한다. 완전 일치를 요구하면 활용형마다 놓친다.
   - **한글끼리는 최장 일치 우선**: "라벨링" 질의에서 "라벨링"(→표시기재)이 발화한 구간 안의 부분 문자열 "라벨"(→표시기재, 첨부문서)까지 또 발화하면 라벨링과 무관한 "첨부문서"가 실려 주제가 흐려진다. 더 구체적(긴) 표제어가 이미 설명한 구간에서 짧은 표제어는 침묵.
   - → 언어 특성에 따라 경계 규칙을 다르게 가져가야 한다는 것이 Part 2 "한국어 표기 경계" 주제의 예고편.

## Part 2. 검증 계층과 마스킹 — "한국어 표기 경계의 교훈 전파"

### 2-1. 검증 계층 — `src/verify/verifier.py`

검색 평가는 오프라인 측정이지만, LLM 답변은 매 요청 새로 생성된다. 답변 속 "15일"이 "30일"로 바뀌면 문장이 유창해도 컴플라이언스 사고 → **모든 응답에 런타임으로 도는 결정론적(규칙 기반) 검증 계층**을 둔다. 4가지 검증:

1. **수치 클레임 존재 대조** — 답변의 숫자+단위(15일, 120 근무일), 고유어 수사(보름), 날짜를 추출해 신뢰 소스에 실존하는지 확인
2. **방향 한정어 검증** — 수치가 근거에 있어도 "15일 **이내**"→"15일 **이후**" 방향 뒤집기는 별도 경고 (이내/이하/까지 vs 이상/이후/초과 — 닫힌 어휘라 기계 검증 가능)
3. **날짜 역할 검증** — 인지일↔마감일을 맞바꿔도 존재 대조는 통과하므로, 역할 키워드에 직접 붙은 날짜를 도구 출력의 역할 라벨(deadline_date 등)과 대조
4. **인용 버전 검증** — 폐지(superseded)된 문서 인용 감지

**핵심 설계 = 신뢰 소스 정의:** `검색된 근거 문단 ∪ 결정론적 도구의 출력 − 질문 에코`.
도구가 계산한 값(인지일+15일=마감일)은 신뢰하고, **사용자 질문은 제외** — 포함하면 틀린 전제로 물었을 때 맞장구가 통과하는 구멍(전제의 승격).

**설계 문답:**
- 왜 LLM 재검증이 아닌 규칙? → 검증자가 확률적이면 '검증의 검증'이 필요한 순환. 결정론은 같은 답변에 항상 같은 판정 = 감사 가능.
- 검증 실패 시? → 차단이 아니라 **경고 부착** (검증기도 오탐 가능, 원칙은 "사람의 최종 확정을 빠르게"). 단 경고는 본문·API·UI 배지에 모두 노출(시끄러운 실패).

### 2-2. 정규식 기초 — \w 와 \b

정규식(regex) = 텍스트에서 패턴을 찾는 검색 문법.

- **\w** = 단어를 이루는 글자 (영문자·숫자·밑줄, **한글 포함**). 공백·문장부호는 \w 아님.
- **\b** = 단어 경계. 글자가 아니라 **\w 글자와 비-\w 글자가 만나는 지점**.
  - `cat\b`: `"my cat!"` 매칭 ✅ ('t' 다음 '!'는 비-\w → 경계 있음) / `"my cats"` 매칭 ❌ ('t' 다음 's'는 \w → 경계 없음)

**한국어에서 \b 가 무너지는 이유:**

```
영어:   "deadline is 2026-07-25."   → '5' 다음 '.' (비-\w) → 경계 성립 → 매칭 ✅
한국어: "마감일은 2026-07-25입니다"  → '5' 다음 '입' — 한글도 \w → 경계 불성립 → 날짜 전체 매칭 실패 ❌
```

한국어는 조사·어미("~입니다", "~로", "~까지")가 **띄어쓰기 없이 직결**되는 언어라 이 표기가 예외가 아니라 최빈이다. \b 를 요구하는 패턴은 이 표기를 전부 놓친다.
**해결 = 룩어라운드**: "\b 필요" 대신 "앞뒤에 숫자만 아니면 됨" — `(?<![\d-])(\d{4}-\d{2}-\d{2})(?![\d-])` (verifier.py `_DATE_RE`, 120행). 한글이 붙어 있어도 경계로 인정된다.

이 결함의 무서운 점: **답변·근거 양쪽에서 똑같이 추출이 안 되니 오탐조차 없어 아무 신호가 없었다.** "수집되지 않는 클레임의 비검증은 통과처럼 보인다" — 조용한 사각지대.

### 2-3. 교훈의 전파 — `src/pv/redactor.py` (점검이력.md 발견 C)

점검 축: "검증기에서 배운 교훈이 다른 안전장치에도 적용됐는가?" → 마스킹에서 **같은 \b 사각지대** 발견:

- **주민번호·전화번호**: "010-1234-5678**로** 연락", "주민번호는 900101-1234567**입니다**" — 조사 직결 표기에서 매칭 전체가 빠짐 = **유출**. 검증기와 같은 숫자 룩어라운드로 수정.
  실패 비용 비대칭(핵심): **"검증기의 비검출은 경고 누락이지만, 마스킹의 비검출은 곧 유출"** (redactor.py 27~28행 주석).
- **이름 마스킹**: "홍길동님**이** 호소" — 호칭 뒤 조사도 같은 사각. 조사 첫 글자 열거([이가은는…])로 열었더니 조사는 열린 집합이라("~님으로부터", "~님께서") **열거는 다음 반례에서 항상 진다** → 룩어헤드 제거 + 오탐은 스톱리스트 방어로 설계 반전. 원칙: **"마스킹은 과탐이 미탐보다 싸다"** (과탐=값이 더 가려질 뿐, 미탐=유출).
- **이메일은 반대 방향**: `[\w.+-]` 가 한글을 삼켜 "hong@x.com**입니다**"의 '입니다'까지 마스킹 → ASCII 한정으로 수정. **과잉 매칭도 경계 결함** (감사 리포트의 원문 범위 왜곡).

**면접용 한 줄:** "정규식 \b 는 한글이 \w 라서 조사 직결 표기에서 경계가 안 됩니다. 검증기 날짜 추출에서 이걸 배웠고, 같은 결함이 다른 안전장치에도 있을 거라 보고 마스킹을 점검했더니 주민번호·전화번호가 조사 붙은 표기에서 통째로 새고 있었습니다. 마스킹의 누락은 유출이라 실패 비용이 더 커서, 룩어라운드로 봉합하고 회귀 테스트로 고정했습니다."

## Part 3. 용어 Q&A (스터디 기록)

| 질문 | 답 |
|---|---|
| Hit@1 발음? | "힛 앳 원". Hit@3 = "힛 앳 쓰리" = 상위 3개 안 정답 비율 |
| 정답이 미리 정해져 있나? | 예. qa_dataset.json 에 문항별 정답 문서 ID(golden label)를 사람이 라벨링 |
| 0.875 는 뭐? | ①벡터 단독의 Hit@1 = 28/32. 4문항 실패가 개선 루프의 출발점 |
| ①→④ 과정의 이름? | **ablation (구성요소 절제 실험)** / 쉽게는 "단계별 성능 비교 실험" |
| ContextRecall 은? | 검색기가 돌려준 문단들 안에 정답 핵심어(keywords)가 존재하는 비율. 생성이 정답을 말할 '재료'를 검색이 줬는가 = faithfulness 의 상한 |
| "검색된 문단"은 누가 검색? | 평가 때는 평가 스크립트가 검색기를 직접 호출, 운영 때는 에이전트가 MCP 도구(search_regulations)로 같은 검색기 호출 |
| BM25/벡터 검색은 검색 방법? | 예, 둘 다 문서 순위를 매기는 알고리즘. BM25=정확 단어 일치 기반, 벡터=의미 유사도 기반 |
| 왜 0~1 정규화 후 0.5 가중합? | 두 점수의 스케일이 달라(코사인 0~1, BM25 상한 없음) 날것으로 더하면 BM25 독식. 같은 자로 잰 뒤 동등 신뢰(0.5)로 평균 |
| 0.55/0.20/0.15/0.10 은? | 리랭커 4신호의 중요도 비율(합 1.0). 재점수 = 가중합 |
| "청크가 질의 토큰을 덮는다"? | 질의를 단어로 쪼갰을 때 그중 몇 %가 문단에 존재하는가(IDF 가중). 4개 중 3개 존재 ≈ 0.75 |
| 9:1 블렌딩의 9는? | 4신호 가중합 재점수(리랭커) 쪽. 1이 1차 하이브리드 점수(prior) |
| 하드네거티브란? | 어휘가 겹쳐 정답처럼 보이는 오답 문서. 예: 화장품 규정 속 "의약품 표시기재와의 차이" 섹션이 의약품 질의에서 정답 문서를 이기는 현상 |
| ablation 이란? | 기능을 껐다 켜서 기여도를 수치로 분리하는 실험 (예: aux 유무 = 0.969 vs 0.906) |

## Part 4. 예상 면접 질문 → 준비된 답

1. **"Hit@1 이 1.0 이면 과적합 아닌가요?"**
   → "맞습니다. 32문항은 실패 분석→개선에 쓴 개발셋이라 1.0은 일반화 성능이 아닙니다. 그래서 Wilson 95% 신뢰구간을 병기하고, 튜닝에 쓰지 않은 홀드아웃셋을 분리해 갭을 그대로 공개했습니다. 1.0은 '완벽'이 아니라 '이 표본에서 실패 관측 0'이라는 뜻입니다." (evaluate.py 161~188행)

2. **"왜 하이브리드 검색인가요?"**
   → "벡터는 동의어에 강하고 BM25 는 고유명사·코드의 정확 일치에 강합니다. 규제문서는 조항 번호·문서 코드처럼 정확 매칭이 필요한 토큰이 많아 둘을 결합했고, 스케일이 다르므로 정규화 후 가중합했습니다."

3. **"리랭커 가중치·페널티 값(0.3, 0.055)은 어디서 나왔나요?"**
   → "감이 아니라 eval/sweep.py 하이퍼파라미터 스윕으로 찾은 플랫 구간의 값입니다. 재조정만으로 안 뒤집히는 하드네거티브는 신호 추가(섹션 타입 prior)로 풀었습니다."

4. **"왜 LLM 질의 재작성이 아니라 동의어 사전인가요?"**
   → "결정론적이라 감사 가능하고 저지연·무비용입니다. 실무에서는 현업 용어집(SOP 용어 정의, 신입 교육자료)에서 사전을 추출해 관리하는 확장 지점입니다."

5. **"검증을 왜 LLM 이 아니라 규칙으로 하나요?"**
   → "검증자가 확률적이면 '검증의 검증'이 필요한 순환이 생깁니다. 결정론적 검증은 같은 답변에 항상 같은 판정이라 감사 가능하고, 그 자체를 평가셋으로 측정할 수 있습니다. 규칙이 못 잡는 것(부정문, 수치 없는 의미 왜곡)은 경계를 명시하고 LLM judge 2차 검증의 자리로 남겼습니다."

6. **"컨텍스트 엔지니어링이 뭔가요?"** (문항2 연계)
   → "AI 는 매번 프로젝트를 처음 보는 상태라, 프로젝트 구조·컨벤션·주의사항을 가이드 문서(CLAUDE.md)로 정리하고 참조할 예시를 지정해, 별도 설명 없이도 코드베이스 규칙에 맞는 결과가 나오게 만든 작업입니다. AI 결과물을 고치는 시간이 줄어 개발 속도가 올라갑니다."

7. **"페르소나 활용은 왜 효과가 있나요?"** (문항2 연계)
   → "AI 는 질문한 것만 답하는데, 페르소나를 주면 그 역할의 전문가가 챙기는 기준으로 먼저 지적해 줍니다. 대고객 UI 경험이 부족해 뭘 물어야 할지도 모르는 상태에서, 시니어 프론트엔드 페르소나의 리뷰로 몰라서 놓쳤을 것들을 잡았습니다."
