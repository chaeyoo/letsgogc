> **필수 포함:** 수행기간 / 지원자 역할 / 성과 / (실패 시) 실패로 느낀 점 / **객관적 증빙 가능**한 경험

## ✍️ 문단 ① (500자 완성본) — 5년간 헬스케어 현장에서 쌓은 FDE 역량

> **수행기간:** 2021~현재 (5년) / **역할:** 풀스택 개발 (마크로젠 → 에임메드 → KB헬스케어)
> **증빙:** 경력증명서 / 각 시스템 운영 실적
> **STAR 매핑:** S=헬스케어 3사에서 현업 부서 곁의 풀스택 / T=정제되지 않은 현업 요구를 작동하는 시스템으로 / A=담당자 인터뷰→화면·데이터 구조로 번역→작게 빨리 배포→피드백 반영 / R=보고서 자동화·챗봇 빌더 등 반복업무의 시스템 대체 (본인이 직접 주도한 성과만 사용)
> ⚠️ LIMS 해외법인 독립운영은 신입 시절 지시받아 수행한 업무 일부 → 성과 주장에서 제외, 나열(경험 목록)에만 LIMS 포함
> ⬜ 성과 수치(보고서 작성 시간 단축, 사용 규모 등) 확보되면 "그 결과" 문장에 보강

지난 5년간 유전자 분석(마크로젠), 디지털 헬스케어(에임메드), 건강검진(KB헬스케어) 기업에서 풀스택 개발자로 일하며, 현업 부서의 반복업무를 시스템으로 바꾸는 일을 해왔습니다. 실험실 정보관리 시스템(LIMS), 직무스트레스 검사 보고서 자동 생성, 비개발자 담당자가 직접 대화 흐름을 설계하는 챗봇 빌더, 임직원 건강관리 어드민, 건강검진 예약 시스템까지 모두 현업의 '이게 필요하다'는 요청에서 시작됐습니다. 요구가 정제된 문서로 오는 경우는 드물었기에, 실험실·운영 담당자를 직접 만나 업무 흐름을 묻고 화면과 데이터 구조로 번역한 뒤, 작게 빨리 배포하고 피드백으로 다듬는 방식으로 일했습니다. 그 결과 수작업이던 직무스트레스 검사 보고서는 자동 생성 시스템으로, 개발자를 거쳐야 했던 챗봇 시나리오 수정은 현업이 직접 설계하는 빌더로 대체됐습니다. 현장에 들어가 문제를 정의하고 코드로 답하는 것이 FDE의 일이라면, 저는 이미 헬스케어 현장에서 그 방식으로 역량을 쌓아왔습니다.

## ✍️ 문단 ② (500자 완성본) — AI 개발도구 협업으로 생성형 AI 스택 학습: 학습 사이트 + RAG·MCP 에이전트

> **수행기간:** 2026.07 (2주) / **역할:** 문제 정의·요구사항 제시·결과 검증 (구현은 Claude Code 에이전트에 위임하는 협업 방식 — 공고 주요업무 'AI 기반 개발도구 활용 개발'을 정직하게 그대로 기술)
> **증빙:** GitHub 리포지토리(pytest 195케이스·CI·평가 스크립트 5종·판정 기준 문서) + 배포된 학습 사이트 URL
> **STAR 매핑:** S=AX를 향한 풀스택 개발자의 출발 / T=공고의 'AI 개발도구 활용 개발' 방식 직접 검증 / A=AI에 구현 위임 + 매 반복 코드·수치 분석으로 원리 소화 + 검증 계층 자체의 반복 정합성 점검 / R=Hit@1 1.0 개선 과정 + 검증 계층의 사각지대 제거 경험 + AI 협업 워크플로우 체득

풀스택 개발자로 일하며 현업의 업무를 AI로 바꾸는 AI Transformation의 주체가 되고자 생성형 AI 학습을 시작했습니다. 직무 핵심 개념 16가지(RAG·Agentic Workflow·MCP 등)를 학습·퀴즈 웹사이트로 만들어 익힌 뒤, 공고의 'AI 개발도구를 활용한 개발'을 직접 검증하고자 Claude Code 에이전트와 함께 제약 규제업무(RA·PV)를 돕는 RAG·MCP 에이전트를 2주간 개발했습니다. 문제 정의와 결과 검증은 제가, 구현은 AI가 맡는 협업 방식을 실험했습니다. 첫 검색 정확도(Hit@1)가 0.87에 그치자 AI에 개선을 지시했고, 반복마다 생성된 코드와 평가 수치를 직접 분석하며 하이브리드 검색·리랭킹·질의확장이 정확도를 1.0으로 끌어올린 원리를 소화했습니다. 나아가 답변을 근거와 대조해 환각을 거르는 검증 계층은 그 자체를 반복 점검해, 한국어 표기 경계로 새던 개인정보 마스킹 누락 같은 사각지대를 찾아 메우는 과정까지 경험했습니다. AI가 코드를 짜는 시대에 AI가 만든 코드도 설명하지 못하면 내 것이 아니며, 엔지니어의 역량은 문제 정의·검증·원리 이해에 있음을 체득했습니다.

## 🗂 (보관) 이전 초안 — 챗봇 빌더 / LIMS 개별 경험안

> 문단①에 흡수되어 최종본에서는 사용하지 않음. 면접 대비 소재로 보관.

### 경험 (구)① — 챗봇 빌더 시스템 개발 (에임메드)

디지털 헬스케어 기업 에임메드에서, 비개발자인 현업 담당자가 직접 대화 흐름을 설계·배포할 수 있는 **챗봇 빌더 시스템**을 개발했습니다. 단순히 챗봇을 만드는 것이 아니라, **'현업이 스스로 대화 로직을 구성하는 도구'를 만드는 일**이었기에, 사용자가 어떤 흐름으로 사고하는지를 먼저 이해하고 이를 구조화된 인터페이스로 풀어내야 했습니다. 이 과정에서 대화 시나리오의 분기·상태 관리, 조건에 따른 응답 처리 등 **오늘날 Agentic Workflow·Function Calling의 기초가 되는 설계 경험**을 쌓았습니다.

### 경험 (구)② — 유전자 분석 LIMS 해외법인 독립운영 (마크로젠)

> ⚠️ **사용 주의:** 신입 시절 팀장 지시로 수행한 업무의 일부이며, '내가 확장을 주도'한 것이 아님. 자소서·면접에서 성과로 주장하지 말 것. LIMS는 "다뤄본 시스템" 수준으로만 언급.

유전자 분석 기업 마크로젠에서 실험실 정보관리 시스템(LIMS)을 개발하며, **유럽·싱가포르 법인이 실험 결과를 자체적으로 분석·산출하도록 시스템을 확장**하는 과제를 수행했습니다. 기존에는 해외 법인의 실험 결과 파일이 모두 한국 본사로 모여 분석까지 본사에서 처리했는데, 이를 각 법인이 독립적으로 처리하도록 재설계한 것입니다. 이 과정에서 **레거시 Java 백엔드까지 직접 다루며 풀스택 역량을 넓혔습니다.**
