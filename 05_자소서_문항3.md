# 자소서 문항 3 — 직무 역량 향상을 위한 노력·경험 (2개 이상)

> **문항:** 지원 직무에 필요한 역량 향상을 위해 노력·경험한 바 (2개 이상)
> **필수 포함:** 수행기간 / 지원자 역할 / 성과 / (실패 시) 실패로 느낀 점 / **객관적 증빙 가능**한 경험
> **전략:** 경험①=생성형AI 서비스 개발(챗봇 빌더) → 경험②=풀스택·레거시 Java로 해외법인 독립운영 시스템 완성. 직무의 '풀스택 AI 엔지니어' 역량을 두 축으로 증빙.
> ※ Claude Code(AI개발도구) 활용 리뉴얼은 문항 2에서 다루므로, 문항 3은 중복을 피해 챗봇빌더 + LIMS로 구성.
> **분량:** 각 500자 내외 / 폼 글자수(1000자) 제한 확인 후 조정

## ✍️ 경험 ① — 생성형 AI 인접: 챗봇 빌더 시스템 개발 (에임메드)

**[수행기간]** 【____】 **[역할]** 풀스택 개발 (【FE/BE 범위 기재】)

디지털 헬스케어 기업 에임메드에서, 비개발자인 현업 담당자가 직접 대화 흐름을 설계·배포할 수 있는 **챗봇 빌더 시스템**을 개발했습니다. 단순히 챗봇을 만드는 것이 아니라, **'현업이 스스로 대화 로직을 구성하는 도구'를 만드는 일**이었기에, 사용자가 어떤 흐름으로 사고하는지를 먼저 이해하고 이를 구조화된 인터페이스로 풀어내야 했습니다.

이 과정에서 대화 시나리오의 분기·상태 관리, 조건에 따른 응답 처리 등 **오늘날 Agentic Workflow·Function Calling의 기초가 되는 설계 경험**을 쌓았습니다. 사용자 입장에서 "왜 이 흐름이 필요한가"를 먼저 묻고 구조로 답하는 방식은, 이번 직무가 요구하는 '구조적 해결 설계' 역량과 그대로 맞닿아 있습니다.

**[성과]** 【도입 부서/사용 규모/운영 성과 등 수치·증빙 기재】

## ✍️ 경험 ② — 풀스택·도메인 확장: 유전자 분석 LIMS 해외법인 독립운영 (마크로젠)

**[수행기간]** 【____】 **[역할]** **풀스택 개발 (FE + 레거시 Java 백엔드)**

유전자 분석 기업 마크로젠에서 실험실 정보관리 시스템(LIMS)을 개발하며, **유럽·싱가포르 법인이 실험 결과를 자체적으로 분석·산출하도록 시스템을 확장**하는 과제를 수행했습니다. 기존에는 해외 법인의 실험 결과 파일이 모두 한국 본사로 모여 분석까지 본사에서 처리했는데, 이를 각 법인이 독립적으로 처리하도록 재설계한 것입니다.

이 과정에서 저는 웹 개발자를 넘어 **레거시 Java 백엔드까지 직접 다루며 풀스택 역량을 넓혔고**, 실험 프로세스라는 낯선 도메인을 현업과 협업해 이해했습니다. 단순 화면 개발이 아니라 데이터 처리·분석 로직 전반을 다룬 경험은, 이번 직무가 요구하는 **'풀스택 AI 엔지니어'** 의 폭넓은 개발 역량과 직접 맞닿아 있습니다.

**[성과]** 해외법인 독립운영 체계 완성 / 【본사 분석 부담·대기시간 감소 등 지표·증빙 기재】

## ✍️ 경험 ③ (신규, 500자 완성본) — 생성형 AI 역량 직접 증명: 학습 사이트 + RAG·MCP 에이전트 개발

> **소재:** `/learn` 학습 사이트(next-14-app.vercel.app/learn) + `project/` RA-Assistant를 종합한 STAR 완성본 (496자, 공백 포함)
> **수행기간:** 2026.07 (2주) / **역할:** 1인 풀스택(학습 사이트 구현 + 에이전트 설계·개발·평가 전 과정)
> **증빙:** GitHub 리포지토리(테스트 83건·CI·평가 스크립트) + 배포된 학습 사이트 URL
> **STAR 매핑:** S=AX를 향한 풀스택 개발자의 출발 / T=공고 필수 스택을 작동하는 시스템으로 증명 / A=학습 사이트 구현+에이전트 개발+오류 분석 루프 / R=Hit@1 1.0 등 수치 개선+재현 가능 증빙 / 실패의 교훈 포함

풀스택 개발자로 일하며 화면을 만드는 데 그치지 않고, 현업의 업무 자체를 AI로 바꾸는 AI Transformation의 주체가 되고자 생성형 AI 학습을 시작했습니다. 먼저 직무 핵심 개념 16가지(RAG·Agentic Workflow·MCP 등)를 학습·퀴즈 웹사이트로 직접 구현하며 익혔고, 이를 검증하고자 제약 규제업무를 돕는 RAG·MCP 기반 AI 에이전트를 2주간 단독 개발했습니다. 첫 평가에서 검색 정확도(Hit@1)가 0.87에 그쳤습니다. 실패 질의를 전수 분석해 어휘 불일치가 원인임을 확인했고, 하이브리드 검색과 리랭킹, 도메인 동의어 질의확장을 단계적으로 적용해 Hit@1 1.0, 유사 오답 문서 정확도 0.79→1.0으로 개선했습니다. 전 과정은 테스트 83건과 CI로 재현되며, GitHub와 배포 사이트로 증빙 가능합니다. 낮은 수치는 실패가 아니라 개선의 출발점이며, AI 품질은 감이 아닌 측정과 오류 분석의 반복으로 만들어진다는 것을 체득했습니다.
