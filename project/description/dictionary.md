# 기술 용어 사전 (dictionary) — RA-Assistant 데모와 관련 문서의 용어 해설

> 이 문서는 RA-Assistant 데모와 `description/` 문서 4종(파일구조·아키텍처·가이드·면접질문)에
> 등장하는 기술 용어를 **앞의 개념이 뒤의 개념을 이해하는 데 도움이 되는 순서**로 정리한
> 용어 사전이다. 배열 순서는 소프트웨어 기초 → 제약 도메인 → 검색(RAG) → 에이전트(MCP) →
> 신뢰성·검증 → 운영·측정이다.
>
> 각 항목은 세 부분으로 구성된다.
> ① **정의** — 설명 문장 안에 또 다른 기술 용어가 나오면 그 자리에서 함께 풀이한다.
> ② **예시** — 이 프로젝트의 실제 코드·문서 맥락에서 그 용어가 쓰이는 예시 문장이다.
> ③ **관련** — 함께 보면 이해가 깊어지는 인접 개념이다.
>
> 처음 읽는 사람은 0부부터 순서대로, 특정 용어만 찾는 사람은 맨 아래 부록(찾아보기)을
> 이용하면 된다.

---

## 0부 — 소프트웨어·데이터 기초 (모든 부의 전제가 되는 개념)

### 클라이언트 (client) / 서버 (server)
요청을 보내는 쪽이 클라이언트이고, 요청을 받아 처리해 응답하는 쪽이 서버다. 역할의
구분이지 기계의 구분이 아니어서, 한 프로그램이 어떤 관계에서는 서버이면서 다른 관계에서는
클라이언트일 수 있다. 이 데모에서 웹 브라우저(챗 UI)는 FastAPI 서버의 클라이언트이고,
그 FastAPI 서버 안의 에이전트는 다시 MCP 서버의 클라이언트가 된다.
- **예시**: "Claude Desktop 이 stdio 로 접속하면 이 데모의 MCP 서버는 외부 클라이언트를
  받는 독립 서버로 동작한다."
- **관련**: API, MCP, stdio

### API (Application Programming Interface) / 엔드포인트 (endpoint)
API는 프로그램이 다른 프로그램의 기능을 사용할 수 있도록 약속해 둔 호출 규격이다.
사람이 화면(UI)으로 시스템을 쓰듯, 프로그램은 API로 시스템을 쓴다. 웹 API에서
**엔드포인트**는 기능 하나가 노출되는 주소(URL 경로)를 뜻한다 — 이 데모의 엔드포인트는
`/chat`(질문→답변), `/health`(상태 확인), `/api/deadlines`(마감일 목록), `/`(챗 UI)의
네 개다.
- **예시**: "UI 없이도 `curl` 로 `/chat` 엔드포인트를 직접 호출해 같은 응답(JSON)을 받을 수 있다."
- **관련**: HTTP, JSON, FastAPI

### HTTP / REST
HTTP는 웹에서 클라이언트와 서버가 요청·응답을 주고받는 통신 규약이다. 요청에는
메서드(GET=조회, POST=제출 등)와 경로가 있고, 응답에는 상태 코드(200=성공, 404=없음
등)와 본문이 있다. REST는 이 HTTP 규약 위에서 "자원을 경로로 표현하고 메서드로 행위를
구분한다"는 API 설계 관례를 말한다. 이 데모의 `/chat` 은 질문을 제출하므로 POST,
`/health` 는 상태를 조회하므로 GET을 쓴다.
- **관련**: API, 엔드포인트

### JSON (JavaScript Object Notation)
프로그램끼리 데이터를 주고받을 때 가장 널리 쓰이는 텍스트 형식이다. 중괄호의
키-값 쌍(`{"deadline_days": 15}`)과 대괄호의 배열로 구조화된 데이터를 표현하며,
사람이 읽을 수 있으면서 기계가 파싱하기 쉽다. 이 데모에서는 API 응답, MCP 도구의
출력, 업무 데이터 파일(ra_tasks.json), 감사 로그가 전부 JSON 이다. 검증기의 날짜 역할
대조(4부)는 도구 출력 JSON의 키 이름(`"deadline_date"`)을 역할 라벨로 활용한다.
- **예시**: "도구 출력을 JSON 문자열로 직렬화한 것이 검증기의 신뢰 소스 텍스트가 된다."
- **관련**: 스키마, 파싱, 직렬화

### 직렬화 (serialization)
메모리 안의 객체(파이썬 딕셔너리 등)를 저장·전송 가능한 문자열이나 바이트로 바꾸는
것이다. 반대 방향(문자열→객체)은 역직렬화라 부른다. 이 데모의 `_stringify()`는 도구
출력 객체를 JSON 문자열로 직렬화해 LLM 되먹임과 검증 신뢰 소스에 쓴다.
- **관련**: JSON, 파싱

### 스키마 (schema) / Pydantic
데이터가 가져야 하는 구조(필드 이름·타입·필수 여부)의 명세가 스키마다. 스키마가 있으면
"입력이 계약을 지켰는가"를 기계가 검증할 수 있다. **Pydantic**은 파이썬에서 스키마를
클래스로 선언하고 자동 검증해 주는 라이브러리로, 이 데모의 API 입출력(`ChatRequest`/
`ChatResponse`)이 Pydantic 모델이다. FastMCP도 함수의 타입힌트에서 도구 입력 스키마를
만들어 낸다.
- **예시**: "message 필드가 없는 요청은 Pydantic 검증에서 걸려 에이전트에 도달하지 못한다."
- **관련**: 타입힌트, 에러 계약(4부)

### 프로토콜 (protocol)
둘 이상의 프로그램이 소통하기 위해 합의한 형식·절차의 총칭이다. HTTP도 프로토콜이고,
MCP(Model Context Protocol)도 '모델과 도구 사이'라는 특정 관계를 위한 프로토콜이다.
프로토콜의 가치는 표준화에 있다 — 양쪽이 규약만 지키면 서로의 내부 구현을 몰라도
결합할 수 있다.
- **관련**: MCP, HTTP, 인터페이스

### 라이브러리 (library) / 프레임워크 (framework) / 의존성 (dependency)
라이브러리는 내 코드가 필요할 때 불러 쓰는 기능 묶음이고, 프레임워크는 반대로 전체
골격을 제공하며 내 코드가 그 틀 안에 끼워지는 것이다("제어의 역전"). 프로젝트가
동작하기 위해 필요한 외부 라이브러리·프레임워크의 목록이 **의존성**이며, 이 데모는
`requirements.txt`에 선언한다(fastapi·fastmcp·uvicorn·pydantic·anthropic 수준으로
가볍다). 의존성이 적을수록 설치가 단순하고 어디서든 재현되기 쉽다 — 이 데모가 RAG
파이프라인을 numpy·torch 없이 순수 파이썬으로 구현한 이유다.
- **예시**: "LangChain 같은 프레임워크 없이 각 단계를 직접 구현해 원리 이해를 증명하되,
  실무 교체 지점은 인터페이스로 남겼다."
- **관련**: 가상환경, 확장 지점

### 환경변수 (environment variable) / .env
운영체제가 프로세스에 넘겨주는 이름-값 설정이다. 코드를 고치지 않고 실행 환경마다
동작을 바꿀 수 있고, API 키 같은 비밀 값을 소스 코드 밖에 두는 표준 방법이기도 하다.
`.env`는 환경변수들을 적어 두는 관례적 파일로, 이 데모의 `run.sh`가 자동으로 읽어
로드한다. 이 데모의 모든 하이퍼파라미터(CHUNK_SIZE, HYBRID_ALPHA …)와 모드 스위치
(ANTHROPIC_API_KEY)가 환경변수로 제어된다.
- **예시**: "`ANTHROPIC_API_KEY` 환경변수의 존재 여부 하나가 LLM 모드와 오프라인 모드를
  가르는 분기점이다."
- **관련**: 하이퍼파라미터, graceful degradation(3부)

### 가상환경 (venv)
프로젝트마다 독립된 파이썬 실행 환경(인터프리터 + 설치된 라이브러리)을 만드는 도구다.
프로젝트 A와 B가 같은 라이브러리의 다른 버전을 요구할 때 서로 간섭하지 않게 한다.
이 데모의 `run.sh`는 `.venv` 디렉터리에 가상환경을 만들고 그 안에 의존성을 설치한다.
- **관련**: 의존성

### 마크다운 (Markdown) / YAML / frontmatter
- **마크다운**은 `#`(제목), `**굵게**` 같은 간단한 기호로 서식을 표현하는 경량 문서
  형식이다. 이 데모의 규제문서·설명 문서·보고서 초안이 모두 마크다운이다.
- **YAML**은 들여쓰기 기반의 사람이 읽기 쉬운 데이터 표기 형식이다.
- **frontmatter**는 마크다운 문서 맨 위에 `---`로 감싸 넣는 YAML 메타데이터 블록이다.
  이 데모의 규제문서는 frontmatter 에 doc_id·title·version·effective_date·status·
  superseded_by 를 담으며, 이 메타데이터가 출처 표시와 버전 인지 검색(1부)의 뿌리다.
- **예시**: "frontmatter 의 `status: superseded` 한 줄이 그 문서를 기본 검색에서
  제외시키는 스위치다 — 그래서 이 필드의 누락을 preflight 가 기동 전에 잡는다."
- **관련**: 파싱, preflight(5부)

### 파싱 (parsing)
텍스트를 규칙에 따라 해석해 구조화된 데이터로 바꾸는 것이다. 이 데모의 로더는
마크다운에서 frontmatter 를 파싱하고, 리트리버는 시행일 문자열을 날짜 객체로 파싱하며,
검증기는 답변에서 수치·날짜 클레임을 파싱(추출)한다. 파싱 실패의 처리 방향이 설계
결정이 된다 — 조용히 넘기면 조용한 실패(4부)가 되고, 이 데모는 fail-closed 나 명시적
에러로 처리한다.
- **관련**: 정규식(4부), fail-closed(4부)

### 인덱스 (index) / 인덱싱 (indexing)
찾기를 빠르게 하기 위해 미리 만들어 두는 자료구조가 인덱스이고, 그것을 만드는 과정이
인덱싱이다. 책 뒤의 찾아보기와 같은 원리다. 이 데모는 서버 부팅 시 1회 규제문서를
청킹·임베딩해 벡터 인덱스와 BM25 역색인을 만들고, 요청 경로에는 가벼운 검색만 남긴다
— 무거운 작업(인덱싱 시간)과 가벼운 작업(질의 시간)을 분리하는 것이 검색 시스템의
기본 구조다.
- **예시**: "인덱싱은 부팅 시 1회, 질의 시간의 전체 검색 지연은 ~0.9ms 수준이다."
- **관련**: 역색인(2부), 벡터 스토어(2부), 캐시

### 캐시 (cache)
한 번 계산한 결과를 저장해 두고 재사용하는 것이다. 같은 계산을 반복하면 지연이
누적되므로, 바뀌지 않는 값은 미리 계산해 둔다. 이 데모의 리랭커는 청크 토큰·제목
가중치·시행일 파싱 결과를 인덱싱 시점에 캐시한다 — 질의마다 재계산하면 지연이 몇 배로
늘기 때문이다.
- **관련**: 인덱싱

### 벡터 (vector) / 차원 (dimension)
벡터는 숫자 여러 개를 순서대로 늘어놓은 것이고, 그 숫자의 개수가 차원이다. 텍스트를
벡터로 바꾸면(임베딩, 2부) "비슷하다"를 수학(두 벡터의 거리·각도)으로 잴 수 있게 된다.
차원 하나하나가 무엇을 뜻하는지는 방식에 따라 다르다 — TF-IDF 희소 벡터는 차원 하나가
어휘 하나에 대응하고, 밀집 임베딩의 차원은 학습이 만들어 낸 추상적 의미 축이다.
- **관련**: 임베딩(2부), 코사인 유사도(2부)

### 정규화 (normalization)
서로 다른 형태·범위의 값을 비교 가능한 하나의 기준으로 맞추는 작업의 총칭이다.
이 데모에는 두 종류의 정규화가 나온다. 수치 스케일 정규화(min-max, 2부)는 점수 범위가
다른 두 검색 신호를 0~1로 맞춰 결합 가능하게 하고, 표기 정규화(4부)는 "2026년 7월
25일"과 "2026-07-25"처럼 값이 같은 다른 표기를 하나의 표준형으로 접어 대조 가능하게
한다. 공통 원리는 하나다 — 비교하려면 먼저 같은 자(尺) 위에 올려야 한다.
- **관련**: min-max 정규화(2부), 표기 변형 vs 환산(4부)

### 결정론적 (deterministic) / 확률적 (probabilistic)
같은 입력에 항상 같은 출력이 나오면 결정론적이고, 실행마다 출력이 달라질 수 있으면
확률적이다. 규칙·정규식·사전 대조는 결정론적이라 결과를 재현·감사할 수 있고, LLM 의
생성은 확률적이라 같은 질문에도 답이 달라질 수 있다. 이 구분이 이 데모의 중심 설계
원칙이다 — 보고기한 계산·중대성 판정 같은 컴플라이언스 계산은 결정론적 도구가 맡고,
LLM 은 자연어 이해·도구 선택·설명만 맡는다.
- **예시**: "검증자가 확률적이면 '검증의 검증'이 다시 필요해지는 순환이 생기므로,
  사후 검증 게이트도 결정론(규칙)으로 만들었다."
- **관련**: LLM(3부), 사후 검증 게이트(4부), 감사 가능성

### 인터페이스 (interface) / 파이썬 Protocol
인터페이스는 "이 자리에 오는 것은 이런 메서드를 제공해야 한다"는 약속만 정의하고
구현은 정의하지 않는 계약이다. 구현을 계약 뒤로 숨기면 구현끼리 갈아 끼울 수 있게
된다. 파이썬의 `Protocol` 타입이 이를 표현하는 도구이며, 이 데모의
`EmbeddingProvider` Protocol 하나에 TF-IDF·해싱·Voyage 세 구현이 꽂힌다 — "임베더를
교체할 수 있다"는 주장이 말이 아니라 동작하는 코드로 존재하는 방식이다.
- **예시**: "실서비스 전환 시 교체되는 자리(임베더·벡터스토어·리랭커·마스킹)가 전부
  인터페이스로 명시되어 있다."
- **관련**: 확장 지점, 라이브러리/프레임워크

### 파이프라인 (pipeline)
데이터가 여러 처리 단계를 순서대로 통과하는 구조다. 각 단계의 출력이 다음 단계의
입력이 된다. 이 데모의 RAG 파이프라인은 로드 → 청킹 → 임베딩 → 인덱싱(인덱싱 시간)과
질의확장 → 하이브리드 검색 → 리랭킹(질의 시간)으로 구성된다. 파이프라인의 함정은
단계별 테스트가 전부 통과해도 **결합**이 틀릴 수 있다는 것이다 — 이 데모에서 폐지
필터와 시점 필터의 교집합이 공집합이 되던 결함이 그 실례다.
- **관련**: RAG(2부), 필터 조합 결함

### 로그 (log) / 구조화 로그 (structured log)
프로그램이 실행 중에 남기는 기록이다. 자유 문장 로그는 사람이 읽기엔 좋지만 기계로
집계·검색하기 어렵고, **구조화 로그**는 JSON 같은 정형 형식으로 남겨 필드 단위 검색과
자동 분석이 가능하다. 이 데모의 span 로그와 검증 감사 로그는 모두 구조화(JSON) 로그다.
- **예시**: "`span {"name": "tool.search_regulations", "ms": 4.8, "ok": true}` — 어떤
  도구가 몇 ms 에 성공했는지가 기계가 읽을 수 있는 형태로 남는다."
- **관련**: 관측성(5부), 감사 로그(5부)

### 테스트의 종류 — 단위 / 통합 / 회귀 / 불변식 / fuzz / 스모크
- **단위 테스트(unit test)**는 함수·모듈 하나를 고립시켜 검사한다.
- **통합 테스트(integration test)**는 여러 구성요소의 결합을 검사한다 — 단위가 전부
  통과해도 결합이 틀릴 수 있으므로 별도 층이 필요하다.
- **회귀 테스트(regression test)**는 한 번 발견된 결함·반례를 테스트로 고정해 재발을
  막는다. 이 프로젝트의 규칙은 "리뷰에서 나온 반례는 반드시 테스트가 된다"이다.
- **불변식 테스트(invariant test)**는 특정 입출력 값이 아니라 항상 성립해야 하는 성질
  (예: "리랭킹 결과는 후보의 부분집합이다")을 고정한다.
- **fuzz 테스트**는 무작위·경계 입력을 대량으로 넣어 예상 밖의 깨짐을 찾는다.
- **스모크 테스트(smoke test)**는 "일단 기본 동작이 사는가"를 확인하는 최소 검사로,
  이 데모의 preflight canary(5부)가 이 성격이다.
- **예시**: "이 데모의 pytest 195케이스에는 단위·통합·회귀·불변식/fuzz 층이 섞여 있고,
  스모크는 preflight 가 기동 전에 맡는다."
- **관련**: CI(5부), preflight(5부)

### 종료 코드 (exit code)
프로그램이 끝나면서 운영체제에 남기는 정수 신호다. 관례상 0은 성공, 0이 아닌 값은
실패를 뜻하며, 셸 스크립트와 CI 는 이 값으로 다음 단계 진행 여부를 결정한다. 이 데모의
preflight 가 문제 발견 시 exit 1 을 반환하는 것이 "기동 차단"의 실체다 — run.sh 는
exit 0 이 아니면 서버를 띄우지 않는다. 반대로 평가 스크립트는 측정만 하고 exit 0
이므로, 지표의 강제는 pytest 가드가 따로 맡는다(측정과 강제의 구분, 5부).
- **관련**: preflight(5부), 핀과 실측(5부)

### 하이퍼파라미터 (hyperparameter)
시스템이 스스로 학습하는 값이 아니라 설계자가 밖에서 정해 주는 조절 값이다. 청크
크기(500자), 겹침(80자), 하이브리드 가중(α=0.5), 리랭커 블렌딩(0.9), 섹션 페널티
(0.3/0.055) 등이 이 데모의 하이퍼파라미터다. 이 데모의 원칙은 "값의 근거를 감이 아니라
스윕(2부)으로 재현 가능하게 남긴다"이다.
- **예시**: "모든 하이퍼파라미터가 환경변수로 제어되고, 기본값의 근거는
  `eval/sweep.py` 로 재현된다."
- **관련**: 스윕/ablation(2부), 환경변수

## 1부 — 제약 도메인 (이 시스템이 다루는 업무)

### RA (Regulatory Affairs, 인허가/규제업무)
의약품을 시장에 내고 유지하는 데 필요한 규제 업무를 담당하는 직무다. 의약품은 아무나
팔 수 없고 규제당국의 허가를 받아야 하며, 허가 후에도 변경·갱신·표시 관리 의무가
이어진다. RA 담당자의 일상은 "이 변경은 허가 대상인가 신고 대상인가", "심사 기간은
며칠인가", "제출 서류는 무엇인가" 같은 질문에 규정 문서를 근거로 답하는 것이고, 답이
틀리면 곧 규제 리스크가 된다. 이 데모의 왼쪽 절반(규정 검색·마감·체크리스트)이 RA
업무를 겨냥한다.
- **예시**: "RA 담당자가 '변경허가 처리기한'을 물으면 시스템은 규제문서를 검색해 근거
  문단과 출처를 함께 반환한다."
- **관련**: PV, 품목허가, 규제당국

### 규제당국 / 식약처 / ICH
의약품 규제를 집행하는 정부 기관이 규제당국이다. 한국은 식품의약품안전처(식약처,
MFDS), 미국은 FDA, 유럽은 EMA 가 해당한다. **ICH**(International Council for
Harmonisation, 의약품 규제 국제조화 회의)는 주요국 규제 기준을 조화시키는 국제
협의체로, 이 데모가 참조하는 E2D(시판 후 안전성 데이터 관리) 같은 가이드라인을 만든다.
- **예시**: "데모 코퍼스의 frontmatter 에는 authority(식품의약품안전처) 필드가 있다."
- **관련**: 최소보고요건(ICH E2D), KAERS

### 품목허가 / 변경허가·변경신고 / IND / DMF
- **품목허가**는 의약품 한 품목을 판매해도 된다는 당국의 승인이다. 데모 코퍼스의
  REG-001이 이 절차를 다룬다.
- **변경허가/변경신고**는 이미 허가받은 품목의 제조방법·성분·표시 등을 바꿀 때 필요한
  절차다. 변경의 중대성에 따라 사전 승인이 필요한 '허가'와 사후 통지로 충분한 '신고'로
  갈리며, 이 구분 자체가 RA 실무의 단골 질문이다.
- **IND**(Investigational New Drug application, 임상시험계획 승인)는 사람 대상
  임상시험을 시작하기 전에 받아야 하는 승인이다.
- **DMF**(Drug Master File, 원료의약품 등록)는 완제의약품에 쓰이는 원료의약품의 품질
  자료를 당국에 등록하는 제도다.
- **예시**: "데모의 하드네거티브 문서(의료기기 허가·화장품 표시·DMF 등)는 품목허가
  질문과 어휘가 겹치도록 일부러 넣은 오답 후보다."
- **관련**: 하드네거티브(2부), 체크리스트 도구

### GMP / 데이터 완전성 (ALCOA)
**GMP**(Good Manufacturing Practice, 의약품 제조·품질관리 기준)는 의약품을 일관된
품질로 만들기 위한 제조소 운영 기준이다. 그 핵심 요건 중 하나가 **데이터 완전성**
(data integrity)이며, 기록이 조작·누락 없이 남아야 한다는 원칙을 ALCOA —
Attributable(귀속 가능)·Legible(판독 가능)·Contemporaneous(동시 기록)·Original(원본)·
Accurate(정확) — 로 요약한다. 교차오염(다른 제품 성분이 섞이는 것) 방지도 GMP 의 단골
주제다.
- **예시**: "'제품이 섞이는 것'이라는 구어 질문은 질의확장 사전이 GMP 정식 용어
  '교차오염·혼동'으로 연결해 준다."
- **관련**: 질의확장(2부), 감사

### 표시기재 / 첨부문서 (라벨링)
의약품 용기·포장에 적어야 하는 정보(명칭·성분·용법·주의사항 등)가 **표시기재**이고,
상자 안에 들어가는 상세 설명 문서가 **첨부문서**다. 현업 구어로는 묶어서 '라벨링'이라
부른다. 허가사항에 기재된 반응인지(예상 여부 판정)의 기준 문서이기도 해서 PV 와도
연결된다.
- **예시**: "'라벨'이라는 구어는 질의확장 사전이 '표시기재·첨부문서'로 확장한다."
- **관련**: 예상 여부, 질의확장(2부)

### SOP (Standard Operating Procedure, 표준작업지침서)
조직이 특정 업무를 항상 같은 절차로 수행하도록 문서화한 지침이다. 규제 산업에서는
"문서화되지 않은 절차는 없는 절차"로 취급될 만큼 SOP 가 중요하다. 이 데모의 MCP
Prompt(`pv_case_intake`)는 케이스 처리 SOP 를 서버가 배포하는 형태로 옮긴 것이다 —
어느 클라이언트가 붙어도 같은 절차로 일하게 만든다.
- **관련**: MCP Prompts(3부)

### 컴플라이언스 (compliance) / 감사 (audit)
**컴플라이언스**는 법령·규정·기준을 지키는 것 그 자체를 뜻한다. 규제 산업에서 보고
기한을 하루 넘기는 것, 폐지된 규정을 근거로 판단하는 것이 모두 컴플라이언스 위반이다.
**감사**는 그 준수 여부를 제3자가 사후에 검증하는 활동이며, 감사를 통과하려면 "왜 그
판단이 나왔는가"를 기록으로 재구성할 수 있어야 한다(감사 가능성, auditability).
이 데모가 기한 계산을 LLM 이 아니라 결정론 규칙에 맡기는 근본 이유가 여기 있다 —
확률적 판단은 같은 입력에 같은 답을 보장하지 못해 감사를 통과할 수 없다.
- **예시**: "'그 답변이 그때 검증을 통과했는가'는 사후 감사의 질문이라서, 응답마다
  판정 요약을 감사 로그로 남긴다."
- **관련**: 결정론(0부), 감사 로그(5부), 추적성(4부)

### 처리기한 / 근무일 vs 역일
규제 절차마다 당국이 처리해야 하는 법정 기한이 있고, 그 단위가 **근무일**(working day,
주말·공휴일 제외)인지 **역일**(calendar day, 달력 날짜 그대로)인지에 따라 실제 달력상
기한이 크게 달라진다. "120 근무일"을 "120일"로 옮겨 적으면 수치는 같아 보여도 실제
기한이 달라지는 오류다 — 이 데모의 검증기가 '근무일'과 '일'을 다른 단위로 취급하는
이유다.
- **예시**: "검증기의 단위 목록에서 '근무일'이 '일'보다 먼저 매칭되도록 배열해 두었다."
- **관련**: 단위 엄격성(4부), 표기 변형 vs 환산(4부)

### PV (Pharmacovigilance, 약물감시)
의약품이 시판된 뒤 부작용 정보를 수집·평가·보고하는 활동이다. 허가는 수백~수천 명
규모의 제한된 임상시험 데이터로 나므로, 수십만 명이 쓰기 시작한 뒤에야 드러나는 위험이
반드시 있다 — 그것을 잡는 체계가 PV 다. 케이스 접수 → 트리아지 → 인과성 평가 → 용어
코딩 → 당국 보고로 이어지는 이 데모의 오른쪽 절반이 PV 워크플로이며, 전 단계가 규칙
기반(결정론)으로 구현되어 있다.
- **예시**: "허가가 끝이 아니라 시판 후 안전관리 의무가 이어지므로 RA 와 PV 를 한
  데모에 묶었다."
- **관련**: 이상사례, 트리아지, ICSR

### 이상사례 (AE, Adverse Event) / 이상약물반응 (ADR)
**이상사례**는 의약품 투여 후 나타난 모든 의도하지 않은 증상·질병이며, 인과관계가
확인되지 않았어도 '사례'로 수집된다. 인과성이 평가되어 "약 때문일 합리적 가능성이
있다"고 판단된 것은 **이상약물반응(ADR)**이라 구분해 부른다. 수집 단계에서는 넓게(AE),
평가 후에 좁게(ADR) — 이 구분이 PV 데이터 흐름의 뼈대다. 현업 구어로는 둘 다
"부작용"이라 부르는 경우가 많다.
- **예시**: "사용자가 '부작용'이라 물어도 동의어 확장이 규정 용어 '이상사례'로 연결해
  검색이 성공한다."
- **관련**: 중대성, 인과성 평가

### 중대성 (Seriousness) 판정
이상사례가 '중대(serious)'한지의 판정이다. 기준이 규정에 **닫힌 목록**으로 정의되어
있다 — 사망, 생명 위협, 입원 또는 입원기간 연장, 지속적/중대한 장애, 선천적 기형,
기타 의학적으로 중요한 사건. 닫힌 목록과의 '대조'이므로 판단이 아니라 확인이고, 그래서
이 데모는 규칙(결정론)으로 확정 출력한다. 참고로 중대성(seriousness)은 심각도
(severity, 증상이 얼마나 심한가)와 다른 개념이다 — 심한 두통은 severity 가 높아도
중대 기준(입원 등)에 해당하지 않으면 non-serious 다.
- **예시**: "'입원했다'는 서술이 감지되면 중대성 기준 충족으로 판정되어 15일 신속보고
  트랙에 오른다."
- **관련**: 트리아지, 결정론(0부)

### 트리아지 (Triage)
응급실에서 환자를 긴급도 순으로 분류하듯, 접수된 이상사례 케이스를 '어떤 경로로
언제까지 보고해야 하는가'로 분류하는 PV 의 첫 단계다. 이 데모의 분류 규칙은 세 갈래다
— 사망·생명위협이면 지체 없이(D+0) 보고하고, 그 외 중대 기준 충족이면 인지일로부터
15일 이내에 신속보고하며, 비중대이면 정기보고(PSUR)에 포함한다. 판정에는 항상 "최종
확정은 PV 담당자"라는 caveat 가 붙는다.
- **예시**: "트리아지 도구(assess_adverse_event)는 중대성 판정과 보고기한 계산에 근거
  규정 문단(basis)까지 붙여 반환한다."
- **관련**: 인지일, PSUR, caveat(4부)

### 인지일 (awareness date)
회사가 그 이상사례를 **알게 된 날**이다. 신속보고 기한의 기산점(카운트 시작일)이므로
컴플라이언스의 핵심 값이다 — 인지일이 하루 틀리면 기한도 하루 틀린다. 이 데모는
인지일 입력의 형식 오류(YYYY-MM-DD 가 아님)와 값 타당성 오류(미래 날짜 — 연도·월 오타
개연성)를 구분해 둘 다 시끄러운 caveat 로 알리며, 자동 정정은 하지 않는다(자동 정정은
또 다른 조용한 오류 경로이기 때문이다).
- **예시**: "인지일을 '2026/07/01'로 적으면 시스템이 조용히 무시하지 않고 '오늘 기준으로
  계산했다'는 경고를 판정 결과에 명시한다."
- **관련**: 조용한 실패(4부), caveat(4부), 날짜 역할 대조(4부)

### 예상 여부 (Expectedness)
그 반응이 허가사항(첨부문서)에 이미 기재된 '예상된(expected)' 반응인지의 판정이다.
예상치 못한(unexpected) 중대 사례일수록 새로운 위험 신호일 가능성이 높아 보고
우선순위가 높다. 서술만으로 판별할 수 없으면 이 데모는 보수적으로 '예상치 못함'으로
취급한다 — 기한을 놓치는 쪽보다 이르게 잡는 쪽이 안전한 실패 방향이기 때문이다.
- **관련**: 트리아지, 과탐/미탐(4부)

### 신속보고 (expedited report) / PSUR (정기보고)
- **신속보고**는 중대한 개별 사례를 기한 내(지체 없이 또는 15일)에 당국에 보고하는
  것이다.
- **PSUR**(Periodic Safety Update Report, 정기적 최신 안전성 정보 보고서)는 비중대
  사례와 종합적 안전성 평가를 모아 주기적으로 제출하는 보고서다.
- 개별 사례의 긴급 채널과 전체 데이터의 정기 채널이라는 두 트랙 구조를 이해하면
  트리아지의 출력("신속보고 15일" vs "PSUR 포함")이 자연스럽게 읽힌다.
- **예시**: "비중대 판정 케이스는 '비중대 이상사례 → 정기보고(PSUR)에 포함' 경로로
  분류된다."

### ICSR / KAERS / E2B
- **ICSR**(Individual Case Safety Report, 개별사례안전성보고)은 이상사례 1건을
  표준화한 보고 단위다.
- **KAERS**(Korea Adverse Event Reporting System)는 한국의약품안전관리원이 운영하는
  이상사례 보고 시스템으로, ICSR 을 제출하는 곳이다.
- **E2B**는 ICSR 을 전자적으로 교환하기 위한 ICH 의 국제 표준 형식이다 — 이 데모의
  코딩이 PT 영문명을 함께 담는 이유가 국제 보고(E2B) 호환이다.
- **예시**: "draft_ae_report 도구는 KAERS 제출을 전제로 한 ICSR '초안'을 만들고, 최종
  확정은 PV 담당자에게 남긴다."

### 최소보고요건 (minimum reporting criteria, ICH E2D 4요소)
ICH E2D 가이드라인이 정의하는 '유효한 보고'의 최소 조건 4가지다 — ① 식별 가능한 환자
② 식별 가능한 보고자 ③ 의심 의약품 ④ 이상사례. 하나라도 없으면 보고 가능한 케이스가
아니라 '정보 보완 대상'이다. 이 데모의 보고서 초안 도구는 초안 생성 **전에** 이 4요소를
검증하고, 미충족이면 초안 대신 빠진 항목과 보완 질문을 반환한다 — 미완성 케이스가
그럴듯한 문서로 포장되는 것을 막는 설계다. ④요소의 판정 기준은 '코딩에 성공했는가'가
아니라 '구체적 증상 서술이 존재하는가'다 — 코딩 사전의 빈틈이 '보고 불가' 오판으로
연쇄되는 것을 끊는다.
- **예시**: "4요소 중 의심약이 빠지면 초안 대신 '③ 의심 의약품 보완' 질문이 반환된다."
- **관련**: PII 마스킹(존재 신호와의 양립, 4부), 용어 코딩

### 인과성 평가 (Causality Assessment) / WHO-UMC 척도 / dechallenge / rechallenge
"이 증상이 정말 그 약 때문인가"의 평가다. **WHO-UMC 척도**는 확실함(Certain) →
상당히 확실함(Probable/Likely) → 가능함(Possible) → 가능성 적음(Unlikely) →
평가곤란(Unassessable)의 등급 체계이며, 판단 요소는 네 가지다 — 시간적 선후관계
(투여 후에 증상이 났는가), **dechallenge**(중단 후 호전 — 의심약을 끊었더니 증상이
나아졌는가), **rechallenge**(재투여 후 재발 — 다시 투여했더니 같은 증상이 재발했는가:
인과성의 가장 강한 증거다), 대체 원인(병용약·기저질환이 원인일 가능성) 유무.
중대성과 달리 인과성은 임상 정보의 종합 '판단'이라 규칙으로 확정할 수 없다 — 그래서
이 데모는 등급 '제안'과 부족 정보 되묻기(follow-up 질문 생성)까지만 자동화하고, 정보가
서술에 없으면 충족으로 치지 않는 보수 규칙(등급은 아래로만)을 쓴다.
- **예시**: "'중단하니 호전됐다'가 서술에 있으면 dechallenge 신호가 감지되어 Probable
  까지 제안이 가능해지고, 재투여 정보가 없으면 그것이 follow-up 질문으로 나간다."
- **관련**: 확신 수준의 설계(중대성=확정 vs 인과성=제안), 트리아지

### MedDRA / PT / LLT / SOC
- **MedDRA**(Medical Dictionary for Regulatory Activities)는 규제 보고용 국제 표준
  의학용어집이다. 수만 개 용어의 계층 구조이며 유료 라이선스라 데모에 담을 수 없다.
- **PT**(Preferred Term)는 하나의 의학적 개념을 대표하는 표준 용어다(예: '호흡곤란/
  Dyspnoea').
- **LLT**(Lowest Level Term)는 PT 아래의 최하위 용어로, 현장의 구어·변형 표현이 여기
  매핑된다(예: '숨이 차다' → PT 호흡곤란).
- **SOC**(System Organ Class)는 기관계 대분류다(예: '호흡기, 흉곽 및 종격 장애').
- 이 데모의 코딩 3계층(검수 소사전=확정 / LLT 참조=후보 / 미코딩 감지)은 MedDRA 의
  PT/LLT 구조를 소사전으로 축소한 것이며, 실무 교체 지점은 사전 테이블뿐이다.
- **예시**: "'청력 상실'은 LLT 참조 테이블에서 PT '난청(Deafness)'의 후보로 제안되고,
  사람의 승인을 거쳐야 확정 사전으로 승격된다."
- **관련**: 용어 코딩, 시그널 탐지

### 용어 코딩 (coding) / verbatim / 시그널 탐지 (signal detection)
구어 서술을 표준 용어(PT)로 바꾸는 작업이 **코딩**이고, 코딩 전의 보고자 원문 표현을
**verbatim** 이라 부른다. 코딩이 흔들리면 같은 이상사례가 다른 이름으로 흩어져
**시그널 탐지** — 축적된 보고 데이터에서 '특정 약 - 특정 반응' 조합의 보고가 기대보다
많은지(불균형)를 통계적으로 찾는 활동 — 의 집계가 무너진다. 그래서 이 데모의 확정
코딩은 정밀도 무관용(오탐 0)이고, 검수 안 된 매핑은 '후보'로만 내며, 후보와 확정을
타입 수준에서 분리해 후보가 집계에 섞이는 실수를 구조적으로 차단한다.
- **예시**: "verbatim '숨쉬기 힘들다'가 PT '호흡곤란(Dyspnoea)'으로 코딩되고, 같은 PT 는
  1건으로 dedupe 되어 집계 왜곡을 막는다."
- **관련**: 정밀도/재현율(2부), MedDRA

### 폐지본 (superseded) / 시행일 (effective date) / 버전 인지 검색
규정은 개정된다 — 개정되면 구판은 **폐지(superseded)** 상태가 되고 후속본이 현행이
된다. **시행일**은 그 버전이 효력을 갖기 시작한 날이다. 폐지된 옛 기준을 현행처럼
인용하면 그 자체가 컴플라이언스 사고이므로, 검색이 문서의 버전 상태를 알아야 한다 —
그것이 **버전 인지 검색**이다. 이 데모의 구현은 세 모드다. 기본 검색은 폐지본을
제외하고, `as_of`(기준일) 조회는 "그 시점의 현행"을 반환하며(폐지본이라도 [시행일,
후속본 시행일) 구간에 기준일이 들어오면 당시 현행이므로 포함한다), `include_superseded`
는 개정 이력 조회용으로 폐지본까지 노출한다. 코퍼스에는 구판(REG-013, 보고기한 30일)과
현행(REG-005, 15일)이 일부러 충돌하게 심어져 있어 필터의 작동이 수치로 증명된다.
- **예시**: "as_of=2025-01-01 조회에는 지금은 폐지된 REG-013 이 '당시 현행'으로
  반환되는 것이 정답이다."
- **관련**: fail-closed(4부), 폐지 체인 단조성(5부), frontmatter(0부)

## 2부 — 검색과 RAG (질문에 근거를 찾아 붙이는 기술)

### 코퍼스 (corpus)
검색·분석의 대상이 되는 문서 집합 전체를 부르는 말이다. 이 데모의 코퍼스는 규제문서
13종이며, 구성이 의도적이다 — 핵심 문서 6종 + 어휘가 겹치는 하드네거티브 6종 + 폐지
구판 1종. 코퍼스가 작으면 평가 수치의 절대값보다 모드 간 상대 비교가 의미를 갖는다.
- **관련**: 하드네거티브, 평가셋

### 토큰화 (tokenization)
텍스트를 검색·비교의 최소 단위(토큰)로 쪼개는 것이다. 가장 단순한 방식은 공백 기준
어절 분리인데, 한국어는 조사가 붙고("허가를/허가는/허가가") 띄어쓰기 변형이 많아
어절만으로는 같은 말이 다른 토큰이 되기 쉽다. 토큰화의 품질이 어휘 기반 검색(TF-IDF·
BM25) 전체의 상한을 정한다.
- **예시**: "이 데모의 토크나이저는 어절과 CJK 문자 bi-gram 을 함께 생성한다."
- **관련**: bi-gram, 형태소 분석

### bi-gram / 형태소 분석
- **bi-gram**은 연속한 두 글자를 하나의 토큰으로 만드는 기법이다("감기약" → "감기",
  "기약"). 조사가 붙거나 띄어쓰기가 달라도 부분열이 겹치므로 어휘 변형에 강해진다.
  n 글자로 일반화하면 n-gram 이라 부른다.
- **형태소 분석**은 언어학적 최소 의미 단위(형태소)로 분해하는 정석 방법이다 —
  정확하지만 형태소 분석기라는 무거운 의존성이 생긴다. 이 데모는 의존성 0 원칙 때문에
  bi-gram 근사를 택했고, 실무 확장 지점으로 형태소 분석기(또는 밀집 임베딩)를
  명시해 두었다.
- **예시**: "'감기약'과 '감기'가 bi-gram 으로 이어져 어절 불일치를 흡수한다."

### TF-IDF
단어의 중요도를 두 값의 곱으로 매기는 고전 기법이다. **TF**(Term Frequency)는 이
문서에 이 단어가 얼마나 자주 나오는가이고, **IDF**(Inverse Document Frequency)는 전체
문서 중 이 단어가 얼마나 희귀한가다. 흔한 조사·일반어는 IDF 가 낮아 자동으로 힘이
빠지고, 특정 문서에만 자주 나오는 단어가 그 문서의 특징으로 부각된다. 문서·질의를
TF-IDF 값의 벡터로 만들면 유사도를 수치로 비교할 수 있다.
- **예시**: "데모의 기본 임베더는 TF-IDF 희소 벡터다 — 외부 모델 의존성이 없어 '키 없이
  항상 실행' 원칙을 지킨다."
- **관련**: 임베딩, BM25, IDF

### BM25 (Okapi BM25) / TF 포화 / 문서 길이 보정
키워드 검색의 사실상 표준 랭킹 함수다. TF-IDF 계열이지만 두 가지 현실 보정이 붙는다.
**TF 포화**는 같은 단어가 100번 나온 문서가 10번 나온 문서보다 조금만 더 좋게 —
빈도의 효용이 점점 줄어들게 — 점수를 접는 것이고, **문서 길이 보정**은 긴 문서가
우연히 단어를 많이 담는 이점을 평균 길이 대비로 상쇄하는 것이다. 정확한 용어·코드·
고유명사 매칭에 강해 벡터 검색의 보완재로 쓰인다.
- **예시**: "하이브리드 검색은 벡터(넓은 유사)와 BM25(정확 키워드)를 결합한다."
- **관련**: 하이브리드 검색, 역색인

### 역색인 (inverted index)
"문서 → 단어 목록"이 아니라 "단어 → 그 단어가 나오는 문서 목록" 방향으로 뒤집어 만든
인덱스다. 질의 단어가 주어지면 해당 문서들을 즉시 찾을 수 있어 키워드 검색(BM25)의
기반 자료구조가 된다. 이 데모의 BM25Index 가 청크 단위 역색인 통계(단어별 문서 빈도,
문서별 단어 빈도)를 들고 있다.
- **관련**: 인덱싱(0부), BM25

### 임베딩 (embedding) / 희소 벡터 / 밀집 벡터
텍스트를 고정 차원의 숫자 벡터로 바꾸는 것이 임베딩이다. **희소(sparse) 벡터**는 어휘
하나가 차원 하나에 대응해 대부분의 값이 0인 형태다 — TF-IDF 가 이 형태이며, 어휘가
겹쳐야만 유사해진다. **밀집(dense) 벡터**는 신경망이 학습한 저차원 실수 벡터로, 어휘가
전혀 달라도 '의미'가 가까우면 유사해진다 — 상용 임베딩 API(이 데모의 VoyageEmbedder
자리)가 이 형태다. 희소는 값싸고 투명하지만 의미 간극을 못 넘고, 밀집은 의미를 넘지만
비용·의존성이 생긴다.
- **예시**: "'계속 제출해야 하나?'와 '정기 보고(PSUR)' 사이의 의미 간극은 희소 벡터로
  메울 수 없다 — 밀집 임베딩이 밥값을 하는 지점이며, 이 데모가 남겨 둔 정직한 갭이다."
- **관련**: 벡터(0부), 코사인 유사도, EmbeddingProvider(0부 인터페이스)

### OOV (Out-Of-Vocabulary) / feature hashing
**OOV**는 인덱싱 때 본 적 없는 단어가 질의에 등장하는 상황이다. TF-IDF 는 어휘집이
고정되어 OOV 단어를 표현할 수 없다. **feature hashing**은 어휘집을 만들지 않고 해시
함수로 단어를 고정 크기 벡터의 위치에 바로 사상하는 기법이다 — 어휘집이 없으므로
OOV 개념 자체가 사라진다(충돌 가능성은 감수한다). 이 데모의 HashingEmbedder 가 이
방식이며, 임베더 교체 가능성의 두 번째 증명이기도 하다.
- **관련**: 임베딩, 인터페이스(0부)

### 코사인 유사도 (cosine similarity)
두 벡터 사이 각도의 코사인 값(0~1)으로 유사도를 재는 척도다. 벡터의 길이(문서
길이의 영향)를 지우고 방향(내용의 구성비)만 비교한다는 점이 핵심이다 — 긴 문서와 짧은
질의도 공정하게 비교된다.
- **예시**: "벡터 검색은 질의 벡터와 각 청크 벡터의 코사인 유사도 상위 k 개를 뽑는
  것이다."
- **관련**: 벡터(0부), top-k

### top-k / top-n
점수 순으로 상위 몇 개를 자르는가의 표기다. 이 데모는 1차 하이브리드 검색이 top_k=8
(넓게 회수), 2차 리랭킹이 rerank_n=3(좁게 선별)을 기본값으로 쓴다. MCP 도구의 top_n
인자는 1~5로 경계 보정(클램프)되며, 이 보정은 조용한 축소로 보이지 않도록 docstring 에
명시되어 있다.
- **관련**: 재현율/정밀도, 하이퍼파라미터(0부)

### 벡터 스토어 (vector store) / Vector DB
청크 벡터를 저장하고 유사도 검색을 제공하는 저장소다. 이 데모는 리스트에 담아 전수
비교하는 인메모리 구현이며(작은 코퍼스에는 충분하다), 실무 자리는 pgvector·Qdrant 같은
**Vector DB** — 수백만 벡터에서도 빠른 근사 최근접 탐색(ANN)을 제공하는 전용
데이터베이스 — 다.
- **관련**: 인덱싱(0부), 확장 지점

### 청킹 (chunking) / overlap
문서를 검색 단위(청크)로 자르는 것이다. 통째로 인덱싱하면 문서의 서로 다른 주제가
한 벡터에 섞이고, 너무 잘게 자르면 맥락이 끊긴다. 이 데모의 청킹은 두 겹이다 —
마크다운 헤딩(`##`) 경계로 먼저 나눠 주제가 청크 안에서 끊기지 않게 하고(구조 인식),
그래도 긴 섹션은 500자 크기의 창을 80자씩 **겹쳐(overlap)** 밀며 자른다. 겹침이 없으면
경계에 걸린 문장의 정보가 앞뒤 어느 청크에서도 온전하지 않게 된다. 각 청크는 doc_id·
버전·상태 메타데이터를 함께 들고 다녀 출처 표시와 버전 필터가 청크 수준에서 가능하다.
- **예시**: "CHUNK_SIZE=500, CHUNK_OVERLAP=80 이며 값의 근거는 eval/sweep.py 로
  재현된다."
- **관련**: Contextual Retrieval, 하이퍼파라미터(0부)

### Contextual Retrieval (경량판)
청크가 문서에서 떨어져 나오면 "이것이 어느 문서의 어느 맥락인지"를 잃는다는 문제의
해법이다. 원래 아이디어(Anthropic)는 LLM 으로 청크마다 맥락 설명을 생성해 붙이는
것인데, 이 데모는 각 청크 앞에 `[문서제목 > 섹션제목]`을 붙이는 규칙 기반 경량판을
쓴다 — 청크가 고립되어도 문서·섹션 맥락이 검색 신호에 반영된다.
- **예시**: "청크 본문에 '보고 기한'만 있어도 접두 `[시판 후 안전관리 > 신속보고]`가
  문서 정합 신호를 준다."
- **관련**: 청킹

### RAG (Retrieval-Augmented Generation, 검색 증강 생성)
LLM 이 자기 파라미터에 저장된 기억만으로 답하지 않고, 먼저 **검색으로 근거 문서를
찾아** 그 근거를 프롬프트에 넣고 답하게 하는 구조다. 세 가지가 달라진다 — 근거가
실재하므로 출처를 붙일 수 있고(추적성), 근거가 없으면 "모른다"고 답할 수 있으며
(환각 억제의 토대), 모델을 재학습하지 않고 문서만 바꿔 지식을 갱신할 수 있다. 이
데모의 RAG 는 질의확장 → 하이브리드 검색 → 리랭킹의 파이프라인이다.
- **예시**: "규제문서는 자주 개정되므로, 모델 재학습 없이 문서 교체만으로 지식이
  갱신되는 RAG 가 이 도메인에 특히 맞는다."
- **관련**: 환각(3부), abstention(4부), 파이프라인(0부)

### 어휘 불일치 (vocabulary mismatch) / 질의확장 (query expansion) / 주제 표류
사용자는 "부작용"이라 묻고 문서는 "이상사례"라 쓴다 — 같은 것을 다른 어휘로 부르면
어휘 기반 검색은 매칭에 실패한다. 이것이 **어휘 불일치**다. **질의확장**은 도메인
동의어 사전으로 질의에 정식 용어를 덧붙여 이 간극을 메우는 기법이며, 이 데모의 사전은
단방향(구어 → 정식 용어)이고 원 질의를 보존한 채 덧붙이기(append)만 한다. 확장을
보수적으로 하는 이유는 **주제 표류**(topic drift) — 확장어가 원 의도와 다른 주제의
문서를 끌어오는 부작용 — 때문이다. 실제로 "DMF" 확장이 주제 표류를 일으켜 사전에서
뺀 기록이 남아 있다.
- **예시**: "'부작용이 심각하면?' 질의는 '… 이상사례 중대한'이 덧붙어 REG-005 가
  회수된다. 사전 항목은 eval 오류 분석으로 발굴·검증했다."
- **관련**: 재현율, 동의어 사전

### 하이브리드 검색 (hybrid search) / min-max 정규화
벡터 유사도(넓은 의미 매칭)와 BM25(정확 키워드 매칭)를 가중합(α)으로 결합하는 1차
검색이다. 두 점수는 스케일이 다르다 — 코사인은 0~1, BM25 는 상한이 없다 — 그대로
더하면 한쪽이 지배하므로, **min-max 정규화**(집합의 최솟값을 0, 최댓값을 1로 선형
변환)로 스케일을 맞춘 뒤 결합한다. 함정이 하나 있다 — 정규화는 버전 필터를 통과한
후보군 '안에서' 수행해야 한다. 전체 기준으로 하면 필터로 빠진 문서가 최솟값·최댓값을
왜곡한다.
- **예시**: "HYBRID_ALPHA=0.5 로 벡터와 BM25 를 같은 비중으로 결합해 top 8 을 회수한다."
- **관련**: 정규화(0부), BM25

### 재현율 (recall) / 정밀도 (precision) / F1
- **재현율**은 찾아야 할 것 중 실제로 찾은 비율이다 — 놓침(미탐)이 적을수록 높다.
- **정밀도**는 찾은 것 중 맞는 것의 비율이다 — 오답 섞임(오탐)이 적을수록 높다.
- **F1**은 둘의 조화평균으로, 한쪽만 높은 것을 벌점하는 종합 지표다.
- 두 지표는 대개 상충하므로 "어느 쪽 실패가 더 비싼가"를 업무별로 정해야 한다. 이
  데모의 2단계 검색은 1차가 재현율(넓게 회수), 2차가 정밀도(좁게 선별)를 맡는 역할
  분리이고, PV 코딩은 확정 계층이 정밀도 무관용(오탐=집계 오염), 후보 계층이 재현율
  (놓침을 검수 큐로 올림)을 맡는다.
- **예시**: "확정 코딩 재현율 0.792와 후보 포함 재현율 0.958의 갭이 곧 '검수 대기 큐'와
  'MedDRA 본체 교체' 확장 지점의 크기다."
- **관련**: 과탐/미탐(4부)

### 하드네거티브 (hard negative)
어휘가 겹쳐서 **그럴듯하게 틀린** 오답 문서다. 의약품 허가 질문에 의료기기 허가 문서가
걸리는 식으로, 무관한 오답(easy negative)과 달리 검색기를 실제로 속인다. 쉬운
평가셋에서는 모든 검색 방식이 만점이라 최적화 효과가 드러나지 않으므로, 이 데모는
하드네거티브 문서 6종을 코퍼스에 일부러 섞어 평가의 변별력을 만들었다 — "평가 설계가
곧 실력"이라는 교훈의 실체다.
- **예시**: "HardNegHit@1 0.786(벡터만) → 1.000(전체 파이프라인)이 'RAG 최적화'의 핵심
  근거 수치다."
- **관련**: 평가셋, 섹션 타입 prior

### 리랭킹 (re-ranking) / Bi-Encoder / Cross-Encoder
1차 검색이 넓게 회수한 후보를 더 정밀한 기준으로 재정렬해 상위 N개만 남기는 2차
단계다. 실무의 표준 구도는 1차 **Bi-Encoder**(질의와 문서를 각자 따로 벡터화해 비교 —
빠르지만 상호작용을 못 본다), 2차 **Cross-Encoder**(질의-문서 쌍을 한 모델에 함께
넣어 정밀 점수 — 느리지만 깊다)이다. 이 데모의 리랭커는 Cross-Encoder 의 오프라인
근사로, 4신호(본문 커버리지 0.55·정확 구문 0.20·섹션제목 0.15·문서제목 0.10)의
가중합에 섹션 타입 prior 를 더하고, 1차 점수를 10% 섞어(rerank_weight=0.9) 안정화한다.
순수 재정렬이 쉬운 질의를 오히려 떨어뜨리는 실측이 블렌딩의 근거이며, 실무
Cross-Encoder 도 1차 점수와 결합하는 관행이 있다.
- **예시**: "실무 전환 시 교체되는 자리는 `_rerank_score` 함수 하나다 — 섹션 메타를
  입력에 포함하는 형태로 prior 지식도 이식한다."
- **관련**: 섹션 타입 prior, 하드네거티브

### 섹션 타입 prior / 질의 의도 게이트
**prior** 는 내용을 보기 전에 구조·사전 지식만으로 부여하는 사전 점수다. 규제문서에는
어휘 신호를 속이는 두 가지 구조가 반복된다 — 다른 도메인의 어휘를 통째로 인용하는
**대조 섹션**("의약품 표시기재와의 차이")과 문서 전체 주제 어휘를 요약해 담는
**서두 섹션**(목적/개요/총칙)이다. 그래서 대조 섹션에 페널티(−0.3), 서두 섹션에 감쇠
(−0.055)를 준다. 단 질의 자체가 비교("차이가 뭐야?")나 정의("목적이 뭐야?")를 물으면
그 섹션이 곧 정답이므로 **게이트**가 페널티를 끈다. 과적합 방어로 '대조 섹션이 정답인
문항' 2개를 평가셋에 추가해 두었다 — 신호가 무차별 삭제라면 그 문항들이 깨진다.
- **예시**: "마지막 잔여 실패(화장품 문서의 대조 섹션)는 어휘 재가중의 어떤 조합으로도
  안 뒤집혔고, 구조 신호(prior) 추가로 풀렸다."
- **관련**: 리랭킹, 과적합

### Hit@1 / MRR / ContextRecall
검색 품질의 평가 지표다. **Hit@1**은 정답 문서가 1위로 나온 문항의 비율이다.
**MRR**(Mean Reciprocal Rank)은 정답이 처음 등장한 순위의 역수(1위=1, 2위=0.5,
3위=0.33…)의 평균으로, 1위를 놓쳐도 상위권이면 부분 점수를 준다.
**ContextRecall**은 최종 근거 묶음이 정답을 말하는 데 필요한 '재료'를 담고 있는
비율이다 — 검색이 재료를 못 주면 생성이 아무리 좋아도 정답이 불가능하므로, 생성 품질
(faithfulness)의 상한이 된다.
- **예시**: "Hit@1 0.875 → 1.000 (32문항, 벡터만 → 전체 파이프라인)."
- **관련**: 평가셋, faithfulness(4부의 groundedness 관련)

### 평가셋 / 라벨 / held-out / 일반화 / 과적합
- **평가셋**은 시스템 품질을 재기 위해 만든 문항·정답의 모음이고, 그 정답 표기가
  **라벨**이다. 라벨은 시스템 출력의 복사가 아니라 도메인 기준으로 독립 작성해야
  한다 — 동어반복이면 모든 지표가 자동으로 1.0이 된다.
- **held-out**은 튜닝에 쓰지 않고 따로 떼어 둔 평가용 데이터다. 튜닝에 쓴 데이터로 잰
  성능은 **일반화**(새 데이터에서의 성능) 증명이 아니다 — 이 데모의 abstention 문턱이
  정확히 이 한계를 가지며, 문서가 그 한계를 명시한다.
- **과적합**(overfitting)은 특정 데이터의 우연한 특징에 맞춰져 새 데이터에서 무너지는
  상태다. 섹션 prior 의 과적합 방어(반대 방향 문항 추가, ablation 테스트)가 그 대응
  사례다.
- **예시**: "PV 라벨 22케이스에는 사전이 못 잡는 롱테일 구어를 일부러 포함해 재현율이
  정직하게 1.0 미만이 되도록 설계했다."
- **관련**: Wilson 신뢰구간, 스윕/ablation

### 스윕 (sweep) / ablation
- **스윕**은 하이퍼파라미터를 범위로 바꿔가며 지표 변화를 재는 실험이다 — "왜 그
  값인가"에 대한 재현 가능한 근거가 된다.
- **ablation**(절제 실험)은 구성요소를 하나씩 빼 보며 각각의 기여도를 재는 실험이다 —
  "그 신호가 정말 필요한가"에 대한 근거가 된다.
- 이 데모는 두 실험을 일회성이 아니라 재현 스크립트(`eval/sweep.py`)로 남겼고, 지표가
  움직이지 않는 파라미터(alpha·idf_power)는 억지 최적값을 주장하는 대신 "이 코퍼스
  규모에서는 둔감하다"고 정직하게 기록했다.
- **예시**: "서두 감쇠 0.055는 유효 밴드(0.04~0.07)의 중앙이며, 밴드 경계에서 깨지는
  문항은 pytest 회귀 가드로 고정했다."
- **관련**: 하이퍼파라미터(0부), 회귀 테스트(0부)

### Wilson 95% 신뢰구간
소표본에서 비율 지표(정확도 등)의 불확실성을 표현하는 구간 추정법이다. 표본 32개에서
관측한 1.000은 '완벽'이 아니라 '이 표본에서 실패 관측 0'일 뿐이므로, [0.893, 1.000]
같은 구간을 병기해 하한을 보수적 추정으로 읽는다. 개선을 주장할 때의 규율도 여기서
나온다 — 두 설정의 신뢰구간이 크게 겹치면 '개선'이 아니라 '구분 불가'로 읽는다.
- **예시**: "이 데모의 모든 평가 스크립트가 핵심 비율 지표에 Wilson 95% CI 를 병기한다."
- **관련**: 평가셋, 통계적 정직성

## 3부 — LLM · 에이전트 · MCP (도구를 골라 쓰는 층)

### LLM (Large Language Model) / 토큰 / 환각 (hallucination)
**LLM**은 방대한 텍스트로 학습해 다음 토큰을 확률적으로 생성하는 언어 모델이다(이
데모는 Anthropic Claude). 여기서 **토큰**은 모델이 텍스트를 처리하는 조각 단위로,
대략 단어보다 조금 작은 단위라고 이해하면 된다(2부의 검색 토큰과는 다른 층위의
개념이다). 생성이 확률적이므로 **환각** — 그럴듯하지만 사실이 아닌 내용을 만들어내는
실패 — 이 구조적으로 가능하다. 규제 도메인에서 "15일"이 "30일"로 바뀌는 환각은 문장이
아무리 유창해도 컴플라이언스 사고이며, 이 데모의 신뢰성 장치 전부(근거 강제·abstention·
사후 검증)가 이 전제에서 출발한다.
- **예시**: "LLM 답변은 매 요청 새로 생성되므로, 오프라인 평가셋이 아니라 모든 응답이
  통과하는 런타임 게이트가 생성 오류의 방어선이 된다."
- **관련**: 결정론 vs 확률(0부), RAG(2부), 사후 검증 게이트(4부)

### 시스템 프롬프트 (system prompt)
대화 시작 전에 모델에게 역할·규칙을 지정하는 지시문이다. 사용자 메시지보다 우선하는
행동 규범으로 작동한다. 이 데모의 시스템 프롬프트는 "반드시 도구로 조회한 뒤 그 근거에
기반해서만 답하고, 근거가 없으면 모른다고 답하라"를 강제한다 — 환각 억제 3중 방어의
입력 측이다.
- **관련**: 환각, grounding(4부)

### Function Calling (tool use)
LLM 이 자연어 답변 대신 "이 도구를 이 인자로 호출해 달라"는 구조화된 요청을 내놓는
능력이다(Anthropic API 의 tools 파라미터). 모델은 도구의 코드를 보지 못하고 이름·설명·
입력 스키마만 보고 판단하므로, 도구 설명의 품질이 곧 도구 선택의 품질이 된다.
- **예시**: "'언제까지 보고?'라는 질문에 모델이 assess_adverse_event 를 스스로 골라
  호출한다."
- **관련**: 에이전트 루프, docstring, MCP

### 에이전트 (agent) / Agentic Workflow / 에이전트 루프
질문을 받아 도구 호출을 **스스로 계획하고 반복**하며 답을 만들어가는 시스템이 에이전트다.
'관찰 → 생각 → 도구 호출 → 관찰'의 반복이 **에이전트 루프**이고, 복합 질문에 여러
도구를 연쇄 조합하는 것이 **Agentic Workflow**다. 자율성에는 안전장치가 따라야 한다 —
이 데모의 루프에는 스텝 상한 6회(무한루프 방지), 도구 실패의 is_error 되먹임(자가
복구), 시스템 프롬프트의 근거 강제가 걸려 있고, 모든 스텝이 span 으로 계측된다.
- **예시**: "'GMP 변경인데 뭘 준비하고 언제까지?'라는 복합 질문에 검색 → 체크리스트 →
  마감일 세 도구가 연쇄 호출된다."
- **관련**: Function Calling, 관측성(5부)

### tool_result / is_error 되먹임
에이전트 루프에서 도구 실행 결과를 모델에게 돌려주는 메시지가 tool_result 다. 도구가
실패했을 때 프로그램을 중단시키는 대신, 실패 사실을 `is_error=True` 로 표시한
tool_result 로 되돌려주면 모델이 스스로 다른 도구나 다른 인자로 재시도한다 — 크래시를
자가 복구 기회로 바꾸는 설계다. 이 데모의 에러 계약(4부)이 available 목록을 주는
이유도 이 되먹임에서 모델이 정정 재료로 쓰기 때문이다.
- **관련**: 에러 계약(4부), 에이전트 루프

### MCP (Model Context Protocol) / FastMCP
**MCP**는 모델(에이전트)과 도구 사이의 표준 프로토콜이다. 도구를 에이전트 코드 안에
함수로 박으면 그 에이전트 전용이 되지만, MCP 서버로 분리하면 규약을 아는 어떤
클라이언트든(Claude Desktop·Cursor·사내 에이전트) 같은 도구를 재사용한다 — N개 모델 ×
M개 도구의 통합이 N+M 으로 줄어드는 확장성이 핵심 가치다. **FastMCP**는 파이썬 함수의
타입힌트와 docstring 에서 MCP 도구 스키마를 자동 생성해 주는 프레임워크다. MCP 의 3대
primitive 는 다음과 같다.
- **Tools** — 에이전트가 상황에 맞게 골라 호출하는 실행 단위다(이 데모 6종).
- **Resources** — 식별자로 콘텐츠를 열람하는 창구다(`regulation://{doc_id}` — 검색이
  아니라 원문 전체 열람이라는 다른 접근 패턴을 분리한 것이다).
- **Prompts** — 서버가 배포하는 표준 절차 프롬프트다(`pv_case_intake` — 어느
  클라이언트가 붙어도 같은 SOP 로 케이스를 처리하게 한다).
- **예시**: "도구를 의도 단위로 분리했다 — '언제까지 보고?'(판정)와 '보고서 만들어줘'
  (산출물)는 다른 도구지만 같은 판정 규칙 모듈을 공유해 결과가 어긋나지 않는다."
- **관련**: 프로토콜(0부), Function Calling, SOP(1부)

### docstring / 타입힌트
- **docstring**은 파이썬 함수 정의 바로 아래 붙이는 설명 문자열이고, **타입힌트**는
  인자·반환값의 타입 표기(`query: str`)다. 보통은 사람 개발자를 위한 문서지만, FastMCP
  환경에서는 이 둘이 그대로 도구 스키마가 되어 **LLM 이 읽는 사용설명서**가 된다.
  그래서 이 데모의 docstring 은 "언제 이 도구를 쓰고, 언제 다른 도구를 써야 하는지"까지
  명시한다 — 도구 선택 품질을 올리는 가장 직접적인 수단이다.
- **예시**: "assess_adverse_event 의 docstring 에는 '규정 자체가 궁금하면
  search_regulations 를 사용하라'는 경계 안내가 들어 있다."
- **관련**: Function Calling, 스키마(0부)

### stdio / 인메모리 (MCP 전송 방식)
MCP 클라이언트와 서버가 연결되는 방식이다. **인메모리**는 같은 프로세스 안에서 직접
연결하는 방식으로 이 데모의 기본값이고(`fastmcp.Client(mcp)`), **stdio**는 표준
입출력 스트림으로 별도 프로세스와 통신하는 방식으로 외부 클라이언트(Claude Desktop·
Cursor) 연결에 쓰인다(`python -m src.mcp_server.server`). stdio 단독 사용에서는 도구
인자가 에이전트 입구를 거치지 않고 외부에서 직접 들어온다 — PII 마스킹을 에이전트
입구만이 아니라 도구 계층에서도 해야 하는 이유(심층방어, 4부)가 이 경로에서 나온다.
- **관련**: 클라이언트/서버(0부), 심층방어(4부)

### FastAPI / uvicorn / lifespan
**FastAPI**는 파이썬 웹 API 프레임워크로, Pydantic 과 결합해 입출력 스키마 검증을
자동화한다. **uvicorn**은 FastAPI 앱을 실제로 구동하는 서버 프로그램이다.
**lifespan**은 FastAPI 앱의 시작·종료 시점에 실행되는 훅으로, 이 데모는 여기서 RAG
인덱스를 부팅 시 1회 구축한다 — 요청마다 재구축하지 않는 인덱싱/질의 시간 분리의
구현 지점이다.
- **예시**: "run.sh 의 마지막 줄이 uvicorn 실행이며, 그 앞에 preflight 게이트가 있다."
- **관련**: API(0부), 인덱싱(0부)

### graceful degradation (우아한 성능 저하) / 오프라인 규칙 라우터
의존성이 빠졌을 때 시스템이 통째로 죽는 대신 낮은 단계의 기능으로 계속 동작하는
설계 원칙이다. 이 데모는 LLM API 키가 없으면 **규칙 라우터** — 키워드 규칙으로 질문
의도(케이스 서술/마감/체크리스트/검색)를 분류해 같은 MCP 도구를 호출하고, 검색 근거
발췌로 답을 조립하는 결정론 경로 — 로 자동 전환한다. 두 모드는 동일한 도구·인터페이스·
검증 게이트를 쓰므로 차이는 "누가 도구를 고르는가"뿐이며, 문서가 약속한 동작(이력
조회 등)은 양쪽 모드에서 모두 참이어야 한다는 것이 기능 대칭 계약이다.
- **예시**: "라우터는 확신이 없으면 체크리스트 카테고리를 추측하지 않는다 — 항상 유효한
  인자를 추측해 넣으면 도구의 에러 계약이 발동할 기회를 잃는다."
- **관련**: 환경변수(0부), 에러 계약(4부)

### 멀티턴 (multi-turn) / 후속질문 병합
여러 턴에 걸쳐 이어지는 대화를 멀티턴이라 한다. "그럼 그건?" 같은 짧은 후속질문은
그 자체로는 검색이 불가능하므로, 오프라인 모드는 직전 사용자 발화와 병합해 맥락을
복원한다. 검증 계층도 멀티턴을 인지한다 — 직전 턴의 전제를 이어받은 수치는 '환각'이
아니라 '전제 확인 필요'로 라벨이 조정되며, 그 완화 창은 직전 턴까지로 제한된다(대화
전체를 누적하면 오래전 무관한 수치가 이후 모든 환각을 완화 라벨링하는 부작용이 생긴다).
- **관련**: from_question(4부)

### 스텁 (stub) / 모킹 (mocking)
테스트에서 실물 대신 준비된 응답을 돌려주는 가짜 구성요소를 끼우는 기법이다. 외부
API 처럼 테스트에서 실제로 부를 수 없는 것을 대체한다. 스텁의 설계 원칙은 경계를
최소로 잡는 것이다 — 이 데모의 LLM 모드 테스트는 anthropic 클라이언트 **하나만**
스텁으로 바꾸고 MCP 서버·RAG·검증 게이트는 전부 실물을 태운다. 검증하려는 것이 모델의
지능이 아니라 그 주변의 **배선**(신뢰 소스 수집·허용 집합·에러 계약)이기 때문이다.
- **예시**: "'형식 오류 as_of 가 에러 계약으로 되먹여지는 턴'을 스텁으로 결정론
  재생해, 실패한 호출이 폐지본 허용 집합을 열지 못한다는 계약을 실행으로 고정했다."
- **관련**: 테스트의 종류(0부), 에러 계약(4부)

## 4부 — 신뢰성·검증 (이 데모의 정체성)

### 추적성 (traceability) / 출처 (citation)
답의 모든 주장이 "어느 문서의 어느 부분에서 왔는가"로 되짚어지는 성질이 추적성이고,
그 근거 표시가 출처다. 규제 도메인에서는 "답이 맞는가"만큼 "왜 이 답인가"가 중요하다 —
담당자가 확정 전에 원문과 대조해야 하기 때문이다. 이 데모의 모든 답변에는 출처(문서·
섹션·버전·시행일)가 붙고, PV 판정에는 근거 규정 문단(basis)이 부착되며, UI 는 이것을
1급 정보로 표시한다.
- **예시**: "'이 15일 기한의 출처는 REG-005 §2'까지 답할 수 있어야 규제 도메인에서
  신뢰를 얻는다."
- **관련**: 감사(1부), grounding

### grounding / groundedness
답변이 검색된 근거 '안에서만' 말하는 성질이다. 오프라인 모드는 아예 근거 발췌로 답을
조립하므로 구조적으로 grounded 하고, LLM 모드는 시스템 프롬프트 강제와 사후 검증
게이트로 지킨다. 평가 지표 AnswerGroundedness 가 이 성질을 잰다.
- **관련**: RAG(2부), abstention, 사후 검증 게이트

### abstention (회피) / 과회피 (over-abstention)
근거가 부족할 때 답을 지어내는 대신 "근거를 찾지 못했습니다"라고 답하는 것이
abstention 이다. 반대 방향의 실패가 **과회피** — 답할 수 있는 정상 질문까지 회피하는
것 — 이며, 회피 설계는 이 두 실패 사이의 균형 문제다. 이 데모는 두 신호 — 최상위
근거의 리랭커 점수(SCORE_FLOOR=0.19)와 질의 토큰의 근거 커버리지(COVERAGE_FLOOR=0.28)
— 가 **둘 다** 미달일 때만 회피한다(AND 조건). OR 로 하면 구어체 정상 질문(점수는
낮지만 커버리지가 남는)까지 회피된다. 문턱값은 감이 아니라 범위내/범위밖 질문의 점수
분포 실측으로 보정했으며, 마진은 1차원이 아니라 2차원(AND)이다 — 범위내 최소 점수
(0.156)는 사실 문턱 아래이고 커버리지 축이 구제한다. 커버리지는 확장 질의 기준으로
계산한다 — 검색은 동의어 확장으로 정답을 찾는데 회피 판정만 원 질의로 보면 '정답을
찾아 놓고 모른다고 답하는' 자기모순이 생기기 때문이다.
- **예시**: "'감기약 추천해줘'(범위 밖)는 두 신호 모두 미달이라 회피되고, 이 판정의
  정확도(AbstentionAccuracy)와 과회피율(OverAbstain)을 평가가 따로 잰다."
- **관련**: 재현율/정밀도(2부), held-out(2부 — 문턱 보정의 순환 한계)

### 클레임 (claim)
답변 속에서 사실 여부를 따질 수 있는 최소 주장 단위다. 이 데모의 검증기는 답변에서
수치+단위("15일", "120 근무일"), 고유어 수사("보름"), 날짜(완전 "2026-07-25"·한국어
"2026년 7월 25일"·부분 "7월 25일"·연도 "2025년"), 방향 한정어가 붙은 표현을 클레임으로
추출해 각각 대조한다. 추출되지 않는 클레임은 검증되지 않는 클레임이다 — "수집되지
않는 실패는 통과처럼 보인다"는 이 프로젝트의 반복 교훈이 클레임 추출기를 의심 목록의
첫 줄에 올려 둔 이유다.
- **관련**: 사후 검증 게이트, 대칭적 비검출

### 신뢰 소스 (trusted sources)
사후 검증에서 "답변 속 값이 여기 존재하면 지지된 것"으로 인정하는 텍스트의 집합이며,
그 정의가 검증 모듈의 핵심 설계다 — **검색된 근거 문단 ∪ 결정론적 도구 출력 − 입력
에코(query·as_of) − 에러 계약 응답**. 도구가 '계산해 낸' 값(인지일+15일=마감일)은 근거
원문에 없어도 신뢰해야 하고(포함하지 않으면 옳은 답변마다 오탐이 난다), 사용자 입력의
에코는 빼야 한다(빼지 않으면 전제의 승격이 생긴다). 인지일(awareness_date) 에코는
남긴다 — 도구가 형식·타당성 검사와 caveat 로 감싸 '채택한 계산 기준'이므로 단순
반사인 as_of 와 신뢰 등급이 다르다는 것이 문서화된 경계다.
- **예시**: "신뢰 소스의 뺄셈 항 두 개(입력 에코·에러 계약)는 각각 '전제의 승격'과
  '에러 문구 속 에코'라는 서로 다른 구멍을 막는다."
- **관련**: 전제의 승격, 에러 계약, 2계층 신뢰 소스

### 전제의 승격 (premise promotion) / 입력 에코 (input echo)
검색 도구는 받은 질의를 출력에 그대로 되돌려 준다 — 이것이 **입력 에코**다. 에코를
신뢰 소스에 넣으면, 사용자가 틀린 수치를 전제로 물었을 때("보고 기한이 30일 맞지?")
모델이 맞장구쳐도 그 30일이 에코 안에 있으므로 검증을 통과한다 — 사용자 입력이
'근거'로 **승격**되는 구멍이다. 막는 방법은 두 겹이다 — 에코 필드(query·as_of)는 신뢰
소스에서 제거하고, 질문에 있던 미확인 수치는 from_question 라벨로 '환각'이 아니라
'전제 확인 필요' 경고를 받게 한다(경고의 면제가 아니라 종류의 조정이다).
- **예시**: "에러 문구('as_of \'2024/06/01\' 가 형식이 아님')에도 사용자 입력이
  에코되므로, 에러 계약 응답은 통째로 신뢰 소스에서 제외한다."
- **관련**: 신뢰 소스, from_question

### 에러 계약 (error contract)
도구가 잘못된 입력(존재하지 않는 필터·미지원 카테고리·형식 틀린 기준일)에 **빈 결과가
아니라 명시적 에러와 가용 값 목록**(`{"error": …, "available": [...]}`)을 돌려주는
규약이다. 오타 난 필터에 빈 목록을 주면 에이전트는 그것을 "임박한 마감이 없습니다"
라는 **자신 있는 오답**으로 포장한다 — 사용자는 틀렸다는 신호를 받을 길이 없다.
available 목록은 단순한 친절이 아니라 LLM 에이전트의 자가 정정 재료다(is_error
되먹임으로 받아 스스로 올바른 인자로 재시도한다). 이 계약에는 후속 규율이 둘 따른다 —
계약은 라우터가 기본값 추측으로 우회하면 무력화되고(안전장치를 만든 층 아래에서
우회하지 않는지까지가 계약의 완성이다), 에러 응답은 사용자 입력이 에코되므로 신뢰
소스에서 제외해야 한다.
- **예시**: "형식 틀린 as_of 를 조용히 무시하고 현행 기준으로 답하면 사용자는 '그 시점
  규정'을 받았다고 믿게 된다 — 그래서 `{'error', 'expected': 'YYYY-MM-DD'}` 로 답한다."
- **관련**: 조용한 실패, tool_result/is_error(3부)

### 사후 검증 게이트 (post-generation verification gate)
**모든 응답**이 사용자에게 나가기 전에 통과하는 런타임 검증 계층이다. 평가셋은
오프라인 '표본'을 지키지만, LLM 답변은 매 요청 새로 생성되므로 "15일"이 "30일"로
바뀌는 생성 오류를 평가셋이 막지 못한다 — 게이트는 그 '전수'를 지킨다. 검사 축은
넷이다 — 존재 축(수치·날짜 클레임이 신뢰 소스에 실재하는가), 방향 축(한정어가
뒤집혔는가), 역할 축(날짜의 역할이 뒤바뀌었는가), 버전 축(폐지본이 무단 인용됐는가).
실패 시의 방향은 차단이 아니라 **경고 부착**이다 — 검증기 자신도 오탐 가능성이 있고,
이 도구의 원칙은 '사람의 최종 확정을 빠르게'이지 자동 차단이 아니기 때문이다. 지원된
클레임에는 근거 위치 스니펫(evidence)을 붙여 사람의 대조를 빠르게 한다.
- **예시**: "답변의 '30일'이 근거 어디에도 없으면 '⚠ 자동 검증 경고'가 답변 본문·API
  verification 필드·UI 배지 세 곳에 함께 붙는다."
- **관련**: 클레임, 신뢰 소스, GateStats(5부)

### 방향 한정어 (direction qualifier)
수치·날짜에 붙어 기한·범위의 방향을 정하는 닫힌 어휘다 — 상한(이내·안에·이하·미만·
까지)과 하한(이상·이후·초과·경과)으로 나뉜다. 수치가 근거에 있어도 "15일 **이내**"가
"15일 **이후**"로 뒤집히면 수치 존재 대조는 통과하는, 가장 교묘한 컴플라이언스
왜곡이 된다. 닫힌 어휘 집합이라 결정론 대조가 가능하며, 같은 검증이 기간 표기·날짜
표기("2026-07-25까지"↔"이후")·고유어 표기("보름 이후") **모두**에 대칭 적용된다 —
일부 표기에만 있으면 같은 등급의 왜곡이 표기에 따라 한쪽만 잡히는 축 간 비대칭이
생긴다. 판정은 보수적이다 — 근거가 그 값에 대해 반대 방향만 말할 때에만 플래그하고,
근거에 한정어가 없으면 판단하지 않는다.
- **예시**: "'마감일 이후에는 지연보고로 처리된다' 같은 정당한 '이후' 용례를 오탐하지
  않도록, 근거의 방향과 실제로 충돌할 때만 경고한다."
- **관련**: 사후 검증 게이트, 표기 변형 vs 환산

### 날짜 역할 대조 (date role check)
도구 출력에 날짜가 여러 개면(인지일·마감일) 답변이 두 날짜의 역할을 맞바꿔도("보고
기한은 <인지일>입니다") 각 날짜가 신뢰 소스에 실존하므로 존재 대조는 **정의상**
통과한다 — 그리고 이것은 기한을 앞당겨 읽게 만드는, 컴플라이언스에서 가장 위험한
형태의 오류다. 해법은 결정론적 도구가 날짜를 역할 키(deadline_date·due_date·
awareness_date)로 라벨링해 출력한다는 사실을 이용하는 것이다 — 답변에서 역할 키워드
(기한/마감·인지일)에 **직접 붙은** 날짜를 그 역할의 라벨 집합과 대조한다. 키워드와
날짜 사이에 다른 말이 끼면("기한 규정은 2025-04-01 시행") 역할 주장으로 보지 않고,
신뢰 소스에 역할 라벨이 없으면 판단하지 않는다(보수성).
- **예시**: "역할 스왑은 존재 대조를 '정의상' 통과하므로, 메타 평가에
  DateRoleSwapDetection 축을 따로 만들어 핀으로 고정했다."
- **관련**: JSON(0부 — 라벨의 출처), 방향 한정어

### 표기 변형 vs 환산 / 단위 엄격성
검증기가 "같은 클레임"으로 취급하는 것의 경계 기준이다. 판정 질문은 하나다 — **"이
대조를 통과시키면 근거에 없는 새 값이 생기는가?"** 새 값이 생기면 **환산**이며
금지된다("15일"→"약 2주", "120 근무일"→"120일" — 근무일과 역일은 실제 달력 기한이
다른 별개 단위다: 단위 엄격성). 값이 그대로면 **표기 변형**이며 정규화로 동치 처리한다
(보름=15일, "2026년 7월 25일"=2026-07-25, "7월 25일"=근거 날짜의 월-일 성분,
"2025년"=근거 날짜의 연도 성분). 표기 변형을 동치 처리하지 않으면 옳은 답변이 표기
차이만으로 경고를 받는 오탐이 쌓이고, 오탐은 alert fatigue 로 검증 계층을 죽인다 —
오탐 제거도 검증 강화인 이유다.
- **예시**: "부분 날짜는 연도를 지어내지 않도록 월-일 접미로만 대조한다 — 새 값을
  만들지 않는다는 기준을 정규화 구현 자체가 지킨다."
- **관련**: 정규화(0부), 근무일 vs 역일(1부), alert fatigue

### 2계층 신뢰 소스 / from_case / from_question
사용자가 입력한 케이스 서술은 '사실'이므로 답변이 그것을 재서술하는 것은 정당하다 —
여기에 경고를 붙이면 보고서 초안마다 오탐이 난다. 그러나 "케이스는 사실이다"에서
"케이스 수치가 규정 클레임의 근거가 된다"로 넘어가면 논리적 비약이다 — 케이스의
"30일간 복용"이 답변의 "보고 기한 30일"을 지지하며 조용히 통과하는 형태다. 기계는
'재서술'과 '우연한 지지'를 구분할 수 없으므로, 3택(차단·침묵·라벨) 중 **라벨**을
택했다 — 신뢰 소스를 strict 계층(규정 근거·도구 출력)과 user facts 계층(케이스 에코)
으로 나누고, 케이스 계층에서만 지지되는 클레임에 `from_case` 라벨(집계 이름
case_origin)을 붙여 사람이 가리게 한다. 이 계층 구조는 **모든 축에 일관**해야 한다 —
방향·역할 충돌의 판정 기준은 strict 계층이며, 케이스까지 합산해 판정하면 케이스의
"15일 이후 증상"이 규정 "15일 이내"의 방향 경고를 조용히 무력화한다.
`from_question` 은 같은 철학의 질문 전제 버전이다 — 경고를 없애지 않고 종류만
'전제 확인 필요'로 조정한다.
- **예시**: "📝 케이스 서술 유래 배지가 붙은 수치가 기한·요건 같은 규정 내용으로
  제시되어 있다면 규정 원문과 대조해야 한다."
- **관련**: 신뢰 소스, case_labeled(5부)

### PII / 마스킹 (redaction) / 심층방어 (defense in depth) / NER
**PII**(Personally Identifiable Information, 개인식별정보)는 개인을 특정할 수 있는
정보 — 이름·주민등록번호·연락처·이메일·환자번호 등 — 다. 케이스 서술에 섞여 들어온
PII 가 외부 LLM API·로그로 나가면 법·보안 위반이므로, **마스킹** — 값을 대체 라벨
(`[주민번호]`)로 지우고 유형·건수만 남기는 비식별화 — 을 한다. 원 값은 어디에도
저장하지 않되, 존재 신호(`[이름]님`, "45세 남성")는 남겨 최소보고요건의 '식별 가능한
환자' 판정과 양립시킨다. **심층방어**는 한 겹이 뚫려도 다음 겹이 막도록 방어를 여러
층으로 두는 설계 원칙이며, 이 데모의 마스킹은 에이전트 입구(메시지·이력)와 MCP 도구
계층(모든 자유 텍스트 인자 — stdio 직결 우회로 차단)의 2겹이다. **NER**(Named Entity
Recognition, 개체명 인식)은 문맥으로 이름·기관 같은 개체를 인식하는 기계학습 기법으로,
정규식이 못 잡는 자유 서술 속 이름을 잡는 실무 교체 지점(Presidio 등)이다.
- **예시**: "마스킹 정규식의 경계는 `\b`가 아니라 숫자 룩어라운드다 — '연락처는
  010-…-5678로' 같은 한글 직결 표기가 `\b` 경계를 벗어나 유출되던 사각지대를 v6에서
  봉합했다."
- **관련**: 정규식/룩어라운드, stdio(3부), 최소보고요건(1부)

### 정규식 (regular expression) / `\b` / 룩어라운드 (lookaround)
- **정규식**은 문자열 패턴을 기술하는 작은 언어다. 결정론적이라 '무엇이 잡히고 무엇이
  안 잡히는가'를 감사할 수 있어, 이 데모의 클레임 추출·PII 마스킹·역할 라벨 대조가
  전부 정규식 기반이다.
- **`\b`**(word boundary)는 단어 문자(`\w`)와 비단어 문자 사이의 경계 지점에 매칭되는
  특수 기호다. 함정이 있다 — 파이썬 정규식에서 **한글도 `\w`로 취급**되므로,
  "2026-07-25입니다", "010-…-5678로"처럼 한글이 숫자에 직결되면 그 사이에 경계가
  존재하지 않아 매칭 전체가 조용히 실패한다. 한국어에서 조사가 붙는 것은 가장 흔한
  표기이므로 이 함정의 실질 피해 면적이 크다.
- **룩어라운드**는 매칭 위치의 앞뒤 문맥을 '소비하지 않고' 조건만 확인하는 구문이다
  (`(?<!\d)` = 바로 앞이 숫자가 아니어야 한다, `(?!\d)` = 바로 뒤가 숫자가 아니어야
  한다). 한글 직결은 허용하면서 숫자 연속만 배제할 수 있어 `\b`의 함정을 피한다 —
  검증기의 날짜 추출(v4)과 PII 마스킹(v6)이 모두 이 형태로 수정되었고, 같은 교훈이 한
  계층에만 적용되어 있던 기간이 곧 사각지대였다.
- **예시**: "'주민번호는 900101-…입니다'는 `\b` 기반 정규식으로는 아예 매칭되지 않아
  그대로 외부로 나갔다 — 마스킹의 비검출은 곧 유출이다."
- **관련**: 파싱(0부), 대칭적 비검출

### 조용한 실패 (silent failure) / 시끄러운 실패 (loud failure) / 대칭적 비검출
**조용한 실패**는 틀렸는데 아무 신호가 나지 않는 실패다 — 답이 그럴듯해서 운영 중
신고되지 않는다. 이 프로젝트가 찾아 고친 조용한 실패의 목록은 그 자체가 교재다 —
잘못된 형식의 인지일을 조용히 오늘로 폴백해 기한이 틀린 기준일로 계산되던 것, 오타 난
필터에 "마감 없음"이라 답하던 것, 형식 틀린 as_of 를 조용히 무시하던 것, 라우터의
체크리스트 기본 추측, 중첩 span 합산으로 지연이 2배로 보고되던 것, 필터 조합으로 시점
조회가 0건이 되던 것. 관통 원칙은 이것들을 **시끄러운 실패** — caveat·에러 계약·경고
부착·exit 1 — 로 바꾸는 것이다. 가장 악질적인 형태가 **대칭적 비검출**이다 — 추출기가
답변과 근거 **양쪽에서 똑같이** 못 잡으면 오탐조차 나지 않아 완전한 무신호가 된다
(한국어 날짜, 고유어 수사, `\b` 경계, 부분 날짜가 전부 이 형태였다). 테스트 실패로도
드러나지 않으므로, 추출기 자체를 의심 목록에 올려야만 보인다.
- **예시**: "'검증할 클레임이 수집되지 않는' 실패는 통과처럼 보인다 — 그래서 preflight
  자가 테스트가 표기 변형별로 오류를 심어 수집의 생존을 확인한다."
- **관련**: 클레임, 자가 테스트(5부)

### fail-closed / fail-open
불확실하거나 결함이 있는 상황에서 시스템이 실패하는 방향의 설계다. **fail-closed**는
확신할 수 없으면 제외·차단하는 방향이고(이 데모의 시점 조회는 시행일을 해석할 수 없는
문서를 제외한다 — 예외를 던져 검색 전체를 죽이지도, 무필터로 오답에 섞지도 않는다),
**fail-open**은 불확실해도 통과시키는 방향이다(가용성이 우선일 때 쓴다). 안전장치는
대체로 fail-closed 가 옳지만, 제외가 조용하면 데이터 결함이 은폐되므로 결함 자체는
preflight 가 파일명과 함께 시끄럽게 보고한다 — fail-closed 와 시끄러운 실패는 한
쌍으로 설계해야 한다.
- **관련**: 조용한 실패, preflight(5부)

### 과탐 / 미탐 / alert fatigue
- **미탐**(false negative)은 잡아야 할 것을 놓치는 것이고, **과탐/오탐**(false
  positive)은 잡지 말아야 할 것을 잡는 것이다.
- 검증 계층은 **두 방향 모두로 죽는다** — 미탐이 쌓이면 잡아야 할 것을 못 잡아 죽고,
  오탐이 쌓이면 **alert fatigue** — 경고가 잦아져 사람이 경고를 무시하기 시작하는 현상
  — 로 죽는다. alert fatigue 가 더 치명적인 이유는 무시가 시작되는 순간 특정 축이
  아니라 **전 축이 한꺼번에** 무력화되기 때문이다. 그래서 이 프로젝트는 옳은 답변에
  경고가 붙는 오탐 경로를 미탐 구멍과 같은 등급의 결함으로 세고 제거한다.
- 실패 비용은 계층마다 다르다 — 마스킹은 과탐(가려질 뿐)이 미탐(유출)보다 싸고,
  트리아지는 과탐(불필요한 검토 1건)이 미탐(놓친 신속보고)보다 싸다. "어느 방향의
  실패가 더 싼가"를 계층마다 명시적으로 정하는 것이 안전장치 설계의 첫 질문이다.
- **예시**: "'2025년'·'7월 25일' 표기의 미확인 오탐 제거(v5)는 탐지 추가와 같은 등급의
  검증 강화로 취급되었다."
- **관련**: 재현율/정밀도(2부), warn_rate(5부)

### caveat (경고 문구)
판정 결과에 붙는 "이 부분은 사람이 확인해야 한다"는 명시적 단서다. 시스템이 확신할
수 없거나 입력에 문제가 의심될 때, 값을 임의로 고치는 대신(자동 정정은 또 다른 조용한
오류 경로다) 계산은 그대로 수행하고 그 사실을 알린다. 이 데모의 caveat 사례는 인지일
형식 오류("오늘 기준으로 계산했다"), 미래 인지일("입력 오류 가능성"), 예상 여부 불명
("보수 적용했다"), 그리고 모든 판정에 붙는 "최종 확정은 PV 담당자"다.
- **관련**: 조용한 실패, 트리아지(1부)

### LLM judge / self-critique 순환
**LLM judge**는 다른 (또는 같은) 모델의 출력을 LLM 이 평가하게 하는 기법이다. 수치
없는 의미 왜곡(부정문 뒤집기, 주어 바꿔치기)처럼 결정론 규칙의 범위 밖 검증에 쓸 수
있어, 이 데모는 그 자리를 '2차 검증의 확장 지점'으로 명시해 두었다. 다만 검증자가
확률적이면 '검증의 검증'이 다시 필요해지는 **self-critique 순환**이 생긴다 — 이
데모의 1차 게이트가 규칙(결정론)인 근본 이유이며, 결정론 검증기는 같은 답변에 항상
같은 판정을 내려 그 자체를 평가셋으로 측정할 수 있다.
- **관련**: 결정론(0부), 메타모픽 테스트(5부)

## 5부 — 운영·측정 (배포 전과 배포 후)

### preflight (배포 전 점검) / canary / 자가 테스트 / 설정 불변식
- **preflight** 는 서버 기동 **전에** 강제되는 결정론적 점검이다(항공기의 이륙 전
  점검표에서 온 이름이다). FDE 배치에서 가장 먼저 깨지는 것은 코드가 아니라 고객의
  데이터와 설정이고, 그런 결함은 부팅이 아니라 운영 중 오답으로 나타난다 — 그래서
  기동 전에 4그룹(설정 불변식·코퍼스 무결성·업무 데이터 스키마·스모크)을 검사하고,
  문제가 있으면 exit 1 로 기동을 차단한다. 배포 시점에는 아직 사용자가 없으므로
  시끄럽게 멈추는 비용이 0이다. 게이트가 자동으로 걸리는 경로는 run.sh 와 CI 두
  가지뿐이며, 수동 uvicorn 실행은 게이트 밖이다 — "차단한다"는 주장의 성립 범위를
  문서에 명시해 두었다(숨은 전제의 문서화).
- **설정 불변식**은 하이퍼파라미터 각각이 아니라 **조합**이 지켜야 하는 관계다 —
  overlap < chunk_size, rerank_top_n ≤ retrieve_top_k 처럼, 값 하나하나는 유효해도
  조합이 모순이면 품질이 조용히 깨진다.
- **canary**는 '어려운' 케이스가 아니라 '무조건 되어야 하는' 대표 케이스를 실제
  파이프라인에 태워 보는 스모크 검사다 — 대표 질의가 정답 문서를 1위로 회수하는가,
  대표 케이스가 인지일+15일로 트리아지되는가, 과거 시점 조회에 당시 현행이던 구판이
  반환되는가(시점 조회 의미론의 생존 확인). 이것이 깨졌다면 시스템의 전제 자체가
  무너진 것이다.
- **자가 테스트(self-test)**는 안전장치 **자신**의 생존 검사다 — 검증 게이트의 모든
  경고 축에 오류를 심어 '기대한 바로 그 축에' 걸리는지, 비경고 라벨(case_origin)이
  살아 있는지, 표기 변형(고유어 방향·부분 날짜·한글 직결 PII)의 수집이 살아 있는지,
  정상 케이스에 오탐이 없는지를 기동 전에 확인한다. 고장난 안전장치를 단 채 배포되는
  것은 안전장치가 없는 것보다 나쁘다 — 있다고 믿게 만들기 때문이다. 자가 테스트는
  **대칭적으로** 적용해야 한다 — 게이트만 검사하고 마스킹은 안 하는 비대칭, 한 축만
  심어 보고 나머지 축은 안 보는 비대칭이 각각 실제로 발견·수정된 사각지대였다.
- **예시**: "축 목록과 자가 테스트 커버리지가 어긋나면 pytest 메타 테스트가 실패한다 —
  새 축을 추가하면 자가 테스트 추가가 구조적으로 강제된다."
- **관련**: 종료 코드(0부), 테스트의 종류(0부), 조용한 실패(4부)

### 폐지 체인 (supersession chain) / 시행일 단조성
폐지본이 후속 문서를 가리키고(superseded_by), 그 후속 문서가 실존하는 현행 문서인
연결 구조가 폐지 체인이다. 체인이 끊기면(가리키는 문서가 없거나, 후속본도 폐지본인데
종점이 없으면) 이력 조회와 현행 대체본 안내가 조용히 망가진다. **시행일 단조성**은
체인을 따라갈수록 시행일이 늦어져야 한다는 데이터 전제다 — 후속본 시행일이 구판보다
빠르면 시점(as_of) 조회의 '당시 현행' 구간 [구판 시행일, 후속본 시행일) 이 빈 구간이
되어 판정이 조용히 깨진다. 시점 조회 의미론이 딛고 선 이 전제를 preflight 가 기동 전에
검사한다 — 의미론을 고칠 때 그 의미론의 데이터 전제 검사를 함께 추가한 사례다.
- **관련**: 버전 인지 검색(1부), preflight

### 관측성 (observability) / Trace / Span / wall-clock / SLA / 병목
- **관측성**은 시스템이 '지금 무엇을 하고 있고, 어디서 느리고, 무엇이 실패했는가'를
  밖에서 볼 수 있게 만드는 계측이다. 실무 도구는 OpenTelemetry·LangSmith 이며, 이
  데모는 같은 개념을 의존성 없이 구현했다.
- **Trace**는 요청 1건의 전체 기록이고, **Span**은 그 안의 작업 구간 하나(도구 호출·
  LLM 스텝)다 — 이름·소요 ms·성패가 남고, 예외가 나도 ok=False span 을 남긴 뒤
  재전파한다.
- **wall-clock**은 벽시계 기준의 실제 경과 시간이다. 총 지연은 최상위 span 의
  wall-clock 으로 계산해야 한다 — 스텝 span 은 최상위 span '안에서' 실행되므로 전부
  합산하면 같은 시간이 두 번 세어져 계기판이 실제의 약 2배를 보고한다. 관측이 틀리면
  **SLA**(Service Level Agreement, 서비스 수준 합의 — 예: "응답의 95%는 1초 이내")
  판단과 **병목**(전체 지연을 지배하는 가장 느린 구간) 진단이 함께 틀린다 — 틀린
  계기판은 계기판이 없는 것보다 나쁘다(믿게 만들기 때문이다).
- **예시**: "'어제부터 느려졌어요'라는 전화에 span 데이터로 답하는 것이 배치 후 FDE 의
  일상이다."
- **관련**: 구조화 로그(0부), GateStats

### GateStats / warn_rate / warn_rate_checked / case_labeled / 분모 혼합 착시
검증 게이트의 운영 계기판이다(`/health` 로 상시 노출). **warn_rate**는 전체 응답 중
경고가 붙은 응답의 비율이며, 추이가 오르면 '답변 품질 회귀' 또는 '검증기 오탐 증가
(alert fatigue 위험)' 어느 쪽이든 조치 대상이라는 조기 신호다. 다만 이 해석에는
트래픽 믹스가 일정하다는 숨은 전제가 있다 — 회피·무클레임 응답은 자명하게 통과하므로,
범위 밖 질문의 비중이 변하면 품질 변화 없이도 warn_rate 가 움직인다. 이것이 **분모
혼합 착시**이며, 그래서 수치·날짜 클레임이 있던 응답만 분모로 잡은
**warn_rate_checked** 를 병기한다 — 추이 해석이 목적인 지표일수록 분모의 구성이
명시되어야 한다. **case_labeled** 는 from_case 라벨이 붙은 응답 수다 — 라벨은
ok=True 라 경고율에는 절대 나타나지 않으므로, 답변이 규정 근거 대신 사용자 서술에
기대기 시작하는 이동은 이 추이로만 보인다.
- **예시**: "집계는 응답 단위다 — 한 응답에 미확인 수치가 3개여도 unsupported 축은
  +1 이다(계기판이 재는 것은 경고가 붙은 '응답'의 비율이다)."
- **관련**: alert fatigue(4부), 관측성

### 감사 로그 (audit log)
"그 답변이 그때 검증을 통과했는가"라는 사후 감사 질문에 답하기 위해 응답마다 남기는
판정 요약이다(JSON, PII 없음, case_origin 라벨 포함). 답변 원문이 아니라 판정 요약만
남기는 것도 설계다 — 로그 자체가 개인정보 유출 경로가 되면 안 되기 때문이다.
- **관련**: 감사(1부), 구조화 로그(0부)

### 메타모픽 테스트 (metamorphic testing)
정답 라벨을 만들기 어려운 시스템(여기서는 검증기 자신)을 평가하는 기법이다. 정상
입력과 그것을 **결정론적으로 변조**한 입력 사이에 성립해야 하는 관계("변조했으니
잡혀야 한다", "동치 표기이니 통과해야 한다")를 검사한다. 이 데모의 verify_eval 은
정상 답변에 7종 변조 — 교차문서 수치 치환·오프셋·방향 뒤집기·고유어 치환·날짜
시프트·날짜 역할 스왑·부분 날짜 시프트 — 를 가해 탐지율을 재고, 동시에 정상·동치
표기의 통과율(오탐 감시 4축)을 잰다. "그 검증기는 누가 검증하나"에 대한 이 프로젝트의
답이다.
- **예시**: "교차문서 치환은 '코퍼스의 다른 문서에는 실존하는 값'으로 바꾼다 — 검증이
  전역 코퍼스가 아니라 '이 질의의 근거'를 기준으로 하는지까지 함께 확인된다."
- **관련**: 핀과 실측, 자가 테스트

### 핀과 실측 / 측정과 강제 / 표본 수 하한
- **핀(pin)**은 깨지는 순간이 곧 회귀인 고정 장치로서의 지표다. 합성 변조의 탐지율
  1.0은 새로운 발견이 아니라 검증기 회귀를 고정하는 핀이며, 구성이 개입하지 않는 진짜
  **실측**은 실응답 전수 통과율(E2EPassRate) 하나다 — 둘을 구분해 읽는 것이 지표
  해석의 규율이다.
- **측정과 강제의 구분** — '측정이 존재한다'와 '측정이 지킨다'는 다른 명제다. 평가
  스크립트는 측정·출력 후 exit 0 이므로, CI 에 스크립트만 두면 탐지율이 0.5로 떨어져도
  CI 는 초록불이다(계기판은 있는데 알람이 없는 상태). pytest 가드가 탐지율·오탐률을
  CI 실패로 강제해야 핀에 알람이 달린다.
- **표본 수 하한** — 변조 하네스는 치환 실패를 표본에서 제외하므로, 치환 정규식이
  조용히 깨지면 표본이 줄어든 채 탐지율 1.0이 유지된다. 분자만 보는 지표는 분모의
  수축을 보지 못하므로, 표본 수의 하한도 함께 강제한다.
- **예시**: "'이 지표는 핀이다'라고 부르는 것과 그 핀을 CI 가 강제하는 것 사이의
  간극이 실제로 발견된 논리적 비약이었다."
- **관련**: 메타모픽 테스트, CI, 종료 코드(0부)

### 회귀 (regression) / CI / GitHub Actions
- **회귀**는 잘 되던 것이 이후의 변경 때문에 다시 깨지는 것이다. 시스템이 커질수록
  회귀는 필연이므로, 문제는 회귀를 없애는 것이 아니라 **회귀가 나면 즉시 아는 것**이다.
- **CI**(Continuous Integration)는 코드가 푸시될 때마다 자동으로 검사(이 데모는
  preflight + pytest 195케이스 + 평가 4종)를 실행하는 파이프라인이다.
- **GitHub Actions**는 GitHub 가 제공하는 CI 실행 환경이며, 워크플로 정의 파일은
  저장소 루트의 `.github/workflows/` 에 있어야 인식된다 — 이 데모의 ci.yml 이
  `project/` 밖에 있는 이유이고, 문서의 구조 지도가 이 위치를 틀리게 그렸던 것이 v6
  점검에서 발견된 문서-실재 불일치 사례다.
- **예시**: "'리뷰에서 나온 반례는 반드시 회귀 테스트가 된다'가 검토 루프를 수렴시키는
  규칙이다."
- **관련**: 테스트의 종류(0부), 핀과 실측

### 논리적 비약 / 사각지대 (이 프로젝트의 판정 기준)
검토 루프에서 무엇을 '결함'으로 셀 것인가의 명문화된 기준이다(상세: 아키텍처.md 7장,
20렌즈 재적용 기록은 7-4).
- **논리적 비약**의 판정 질문은 **"전제를 대라"**다 — 어떤 설계 명제가 숨은 전제를
  딛고 서 있는데 그 전제를 검사하는 코드가 없으면 결함이다. 네 가지 형태를 본다 —
  숨은 전제("입구 마스킹이므로 전부 안전" ← 모든 입력이 입구를 지난다는 전제,
  "preflight 가 차단한다" ← 모든 기동이 run.sh·CI 를 지난다는 전제), 성질의 승격
  ("케이스는 사실" → "규정 클레임의 근거"), 범위의 과확장(문서 단위로만 성립하는 이력
  면제를 응답 전역 bool 로 적용), 측정과 강제의 혼동(exit 0 스크립트를 '핀'이라 부름).
- **사각지대**의 판정 질문은 **"고장 나면 누가 아는가"**다 — 실패해도 아무 신호가 없는
  경로는 결함이다. 탐색 절차는 넷이다 — 축×표기 매트릭스의 빈 칸 찾기(대칭성 감사 —
  안전장치 **사이의 교훈 전파**까지 포함한다: 검증기의 `\b` 교훈이 마스킹에 전파되지
  않았던 v6 사례), 조용한 실패 인벤토리, 오탐도 결함으로 세기, 안전장치 자신을 검사
  대상에 올리기.
- 기준에는 **반대면**(알고 남긴 경계)도 명문화되어 있다 — 보수적 미판정, 트리아지의
  과탐 방향, draft_markdown 재조립의 한계, awareness_date 에코의 신뢰 같은 항목은
  결함이 아니라 문서화된 스코프 결정이다. 기준이 없으면 검토가 취향이 되고, 반대면이
  없으면 검토가 무한이 된다.
- **예시**: "비약은 '전제를 대라', 사각지대는 '고장 나면 누가 아는가' — 두 질문에
  코드로 답할 수 없으면 결함으로 세고, 답을 만든 뒤에는 그 답 자체를 다시 같은 질문의
  대상에 올린다."
- **관련**: preflight, 자가 테스트, 회귀 테스트
