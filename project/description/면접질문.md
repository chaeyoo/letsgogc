# 면접 예상 질문·답변 — FDE(Forward Deployed Engineer) 직무 기준

> 이 데모 프로젝트를 본 면접관이 **FDE 직무 관점**에서 던질 법한 질문과 답변을 정리했다.
> FDE의 평가 축 — ① 고객/도메인 밀착, ② 엔드투엔드 통합 역량, ③ 엔터프라이즈 신뢰성 감각,
> ④ 기술 깊이(RAG·Agent·MCP), ⑤ 측정·검증 문화, ⑥ 커뮤니케이션 — 을 따라 구성했다.
>
> 더 깊은 설계 배경 스토리는 `docs/면접노트.md`(28개 주제)를, 낯선 용어는 같은 폴더의
> `dictionary.md`(용어 사전)를 함께 볼 것.

## 준비 우선순위 — 어느 문항부터 준비할 것인가

이 문서의 문항 수는 실전 면접에서 소화 가능한 양을 훌쩍 넘는다. 실전 출제 확률로
3단계로 나눠 준비할 것 (v7 점검의 FDE 적정성 평가 반영):

| 등급 | 문항 | 준비 방식 |
|---|---|---|
| ★★★ 실전 출제 유력 — 구술 리허설 필수 | A 전체, B 전체, C-1~C-7, D-1~D-5, E-1~E-3, F-1~F-6, G 전체, H 전체, **I 전체(배치 현실)** | 각 답변을 30~45초 코어 + 꼬리질문 유도로 압축해 소리 내어 연습 |
| ★★ 꼬리질문 대비 — 흐름만 소화 | E-4~E-9, F-7~F-13 | 결론 한 문장 + 근거 파일 위치만 기억 |
| ★ 내가 꺼낼 카드 — 암기 금지 | E-10~E-19, 부록 표의 내부 용어 행 | 면접관이 물을 수 없는 자문자답(strict 계층·case_origin·20렌즈 등 내부 용어 의존). 시연·심화 대화에서 **내가 화제를 열었을 때만** 사용 |

**대표 스토리 90초 구술본** (하나만 준비한다면 이것 — E-3 + E-6 결합):
"이 프로젝트에서 가장 배운 게 많았던 결함 하나를 말씀드리면 — 버전 인지 검색에 '폐지본
기본 제외'와 '시점(as_of) 조회' 두 필터가 있는데, 각각은 단위 테스트를 통과했지만 겹치면
개정된 규정의 과거 시점 조회가 0건이 됩니다. 시점 조회가 가장 필요한 규정에서만 기능이
무너지는 조합 결함이었죠. '그 시점의 현행을 반환한다'로 의미론을 다시 세우고, 그 의미론이
딛고 선 데이터 전제까지 배포 전 점검으로 고정했습니다. 그리고 '검증기는 누가 검증하나'는
질문에는 메타모픽 평가로 답했습니다 — 정상 답변에 수치 치환·방향 뒤집기 같은 결정론적
변조를 가해 검증 게이트가 실제로 잡는지 7개 축 84건으로 재고, 그 탐지율이 떨어지면 CI 가
실패하게 핀을 박았습니다. 기능 테스트가 아니라 '안전장치 자신의 고장'을 측정하는 층이
있어야 규제 도메인에 AI 를 배치할 수 있다고 생각합니다."

## A. FDE 직무 이해 — "왜 이 프로젝트가 FDE인가"

### A-1. 이 프로젝트가 FDE 역량을 어떻게 증명하나? 모델 엔지니어링 프로젝트와 뭐가 다른가?
**답변:** FDE는 모델을 만드는 사람이 아니라 **기성 LLM을 고객의 현업 워크플로에 배치해
끝까지 작동시키는** 사람입니다. 이 데모는 그 정의대로 만들었습니다 — 모델 학습·파인튜닝은
의도적으로 범위 밖에 두고, 대신 (1) 실존 RA·PV 담당자의 반복업무(규정 검색·이상사례 처리·
기한 관리)를 정확히 겨냥했고, (2) 데이터→RAG→MCP→에이전트→API→UI→평가→CI 전 구간을 혼자
연결했고, (3) 출처 추적·환각 억제·PII 보호·감사 가능성 같은 엔터프라이즈 배치의 실제 관문을
아키텍처에 박았습니다. ML 이론 이해는 BM25·TF-IDF·리랭킹을 순수 파이썬으로 직접 구현하는
것으로 보였습니다.

### A-2. 실제 고객사(예: 제약사)에 배치한다면 첫 2주에 무엇을 하겠나?
**답변:** 코드보다 먼저 **업무와 데이터의 실체**를 확인합니다. (1) RA·PV 담당자의 실제 질문
로그·자주 찾는 규정·SOP를 수집해 이 데모의 QA셋 같은 평가셋부터 만들고, (2) 문서 소스의
실체(PDF/HWP/그룹웨어)와 버전 관리 체계를 파악해 로더 계층의 교체 범위를 정하고,
(3) 보안 요건(PII·망분리·LLM API 반출 정책)을 확인합니다. 이 데모에서 로더·임베더·마스킹이
전부 교체 가능한 인터페이스로 설계된 이유가 바로 이 현장 편차 때문입니다. 성공 기준도
초기에 수치로 합의합니다 — "검색 Hit@1 몇 이상, 환각 0건" 같은 측정 가능한 계약이요.

### A-3. 현업 담당자가 "AI를 못 믿겠다"고 하면 어떻게 설득하나?
**답변:** 말이 아니라 **구조**로 설득합니다. 이 데모의 방식이 그 답인데 — 모든 답에 출처가
붙고(추적성), 근거가 없으면 "모른다"고 답하고(abstention), 기한 계산 같은 컴플라이언스는
LLM이 아니라 감사 가능한 규칙 도구가 하고, 나가는 답변의 수치까지 검증 게이트가 근거와
대조합니다. 그리고 "믿어달라"가 아니라 평가 수치(환각률·회피 정확도)를 보여주고, 담당자가
직접 틀린 케이스를 찾으면 그것이 평가셋에 들어가 회귀 테스트가 되는 루프를 제안합니다.
신뢰는 데모가 아니라 **반복 가능한 측정**에서 나온다고 생각합니다.

### A-4. 이 데모에서 가장 자랑하고 싶은 설계 하나는?
**답변(예시):** LLM(확률)과 규칙(결정론)의 **경계 설계**입니다. "언제까지 보고?"라는 질문에서
자연어 이해·도구 선택은 LLM이 하지만, 15일 기한 계산은 규칙 도구가 합니다. 같은 규칙
기반이라도 중대성 판정(닫힌 목록 대조)은 '확정'으로, 인과성 평가(종합 판단)는 '제안+질문'으로
확신 수준을 다르게 출력합니다. 업무의 인식론적 성격에 따라 자동화의 확신 수준을 조절하는 것 —
이게 도메인을 이해하고 도구를 만든다는 뜻이고, FDE가 현장에서 매일 해야 하는 판단이라고
생각합니다.

### A-5. 고객사 배치 당일(Day-0), 서버를 띄우기 전에 무엇을 점검하겠나?
**답변:** 이 데모에 그 답을 코드로 넣어뒀습니다 — `python -m src.preflight`, 기동 전 차단
게이트입니다. FDE 배치에서 가장 먼저 깨지는 건 코드가 아니라 **고객의 데이터와 현장 설정**
이고, 그 결함은 부팅이 아니라 운영 중 오답으로 나타납니다. 예를 들어 status 필드가 빠진
폐지 구판 문서는 버전 필터를 그대로 통과해 현행 답변에 섞입니다 — 부팅 로그는 깨끗하죠.
그래서 4그룹을 강제합니다: 설정 불변식(값 각각이 아니라 **조합**의 모순), 코퍼스 무결성
(필수 메타·doc_id 유일성·폐지 체인), 업무 데이터 스키마, 그리고 스모크 canary — 대표 질의
1건·대표 케이스 1건과 **안전장치 자가 테스트**(검증 게이트: 심은 오류가 실제로 걸리는가,
PII 마스킹: 심은 개인정보가 실제로 지워지는가)까지요.
운영 중 검증 게이트는 '경고 부착'(차단 안 함)이지만 preflight는 exit 1로 **차단**합니다 —
배포 시점엔 아직 사용자가 없어서 시끄럽게 멈추는 비용이 0이니까요. 같은 신뢰 장치라도
시점에 따라 실패 방향을 다르게 설계해야 한다고 생각합니다.

## B. 도메인(RA·PV) 이해

### B-1. RA와 PV가 무슨 업무인지, 왜 이 둘을 묶었는지 설명해보라.
**답변:** RA(Regulatory Affairs)는 의약품의 허가·변경·표시기재 등 규제 인허가 업무이고,
PV(Pharmacovigilance)는 시판 후 이상사례를 수집·평가·보고하는 약물감시 업무입니다.
허가가 끝이 아니라 시판 후 안전관리 의무가 이어지므로 두 업무는 한 몸으로 맞물립니다.
데모 범위도 그 흐름대로 — RA의 문서 검색·기한 관리에 더해, PV의 케이스 접수→트리아지→
인과성→코딩→보고서 초안 전 워크플로를 담았습니다.

### B-2. 중대성(Seriousness) 판정을 왜 규칙으로 '확정'할 수 있다고 봤나?
**답변:** 중대성 기준이 규정에 **닫힌 목록**(사망·생명위협·입원·장애·기형·의학적 중요)으로
정의되어 있기 때문입니다. '판단'이 아니라 '대조'라서 결정론적 규칙으로 충분하고, 오히려
규칙이어야 감사 가능합니다. 반면 인과성(WHO-UMC)은 임상 정보의 종합 판단이라 규칙의 출력을
'제안 + 부족 정보 되묻기'로 낮췄습니다. 평가에서도 중대성·기한 정확도는 1.000이 '정상'이고,
깨지는 순간이 곧 규칙 회귀라서 CI가 지킵니다.

### B-3. MedDRA 코딩을 왜 3계층으로 만들었나?
**답변:** 코딩이 흔들리면 같은 이상사례가 다른 용어로 흩어져 **시그널 탐지(집계)가 무너지기**
때문입니다. 그래서 정밀도 무관용 원칙으로 — ①검수된 사전 매칭만 확정(집계 대상),
②LLT 스타일 참조 매칭은 후보(사람 승인 후 승격 — 사전 성장의 운영 루프), ③사전에 없는
증상 서술은 존재만 감지. 후보와 확정을 타입 수준에서 분리해 후보가 집계에 섞이는 실수를
구조적으로 막았습니다. 평가에서 확정 재현율 0.792와 후보 재현율 0.958의 갭이 각각
'검수 대기 큐'와 'MedDRA 본체 교체'라는 확장 지점의 크기입니다.

### B-4. PII를 마스킹하면 '식별 가능한 환자'라는 최소보고요건과 모순되지 않나?
**답변:** 요건의 본질은 '환자를 특정할 정보가 **존재**하는가'이지 원문 값이 아닙니다.
그래서 값은 지우되 존재 신호(`[이름]님`, "45세 남성")는 남기는 마스킹을 설계했고,
마스킹된 케이스도 환자 요건이 충족으로 판정됨을 테스트로 보증합니다. 외부 API 경계에서는
값을 감추고, 요건 판정은 신호로 — 두 요구를 양립시킨 지점이 이 프로젝트에서 도메인과
엔지니어링이 만나는 가장 재미있는 부분이었습니다.

## C. RAG 최적화

### C-1. 왜 벡터 검색만으로는 부족했나?
**답변:** 두 가지 실패 모드 때문입니다. (1) **어휘 불일치** — 사용자는 "부작용"이라 묻는데
문서는 "이상사례"라 씁니다. 이건 도메인 동의어 질의확장으로 풀었습니다. (2) **하드네거티브** —
어휘가 겹치는 오답 문서(의약품 허가 질문에 의료기기 허가 문서)가 벡터 유사도로는 이깁니다.
이건 리랭킹(문서제목 반증 신호 + 섹션 타입 prior)으로 풀었습니다. 수치로: Hit@1
0.875(벡터만)→1.000, 하드네거티브 0.786→1.000 (32문항).

### C-2. 하이브리드 검색에서 벡터와 BM25를 어떻게 결합했나? 함정은 없었나?
**답변:** 각 점수를 min-max 정규화한 뒤 α=0.5로 가중합해 top 8을 회수합니다. 정규화가
필수인 이유는 코사인(0~1)과 BM25(무제한)의 스케일이 달라 그대로 더하면 한쪽이 지배하기
때문입니다. 디테일 하나 — 정규화를 **버전 필터 통과 후보군 안에서** 수행합니다. 전체 기준으로
하면 필터로 빠진 문서가 스케일을 왜곡합니다. 그리고 정직한 트레이드오프: 작은 코퍼스에서는
하이브리드가 쉬운 질의의 Hit@1을 살짝 떨어뜨립니다(0.875→0.844). 대신 ContextRecall이 오르고
리랭킹이 회복시킵니다 — 좋아 보이는 숫자가 아니라 방법론을 보여주는 게 목적이었습니다.

### C-3. 리랭커를 직접 만들었는데, Cross-Encoder와의 관계는?
**답변:** 1차 하이브리드가 Bi-Encoder 감각(질의·문서 각각 벡터화, 빠르고 넓게), 2차 리랭커가
Cross-Encoder 감각(질의-청크를 함께 보고 정밀 재점수)입니다. 데모에서는 4신호(본문 커버리지·
정확 구문·섹션제목·문서제목) 가중합 + 섹션 타입 prior로 근사했고, 실무에서는 `_rerank_score`
자리를 실제 Cross-Encoder로 교체합니다. 흥미로운 발견 하나 — 순수 재정렬은 쉬운 질의를
오히려 떨어뜨려서, 1차 점수를 10% 섞어(rerank_weight=0.9) 안정화했습니다. 실무
Cross-Encoder도 first-stage 점수와 결합하는 관행이 있는데, 같은 이유입니다.

### C-4. 섹션 타입 prior가 특정 실패 문항 하나를 위한 과적합 아닌가?
**답변:** 세 가지로 방어합니다. (1) 이 신호는 특정 문서가 아니라 코퍼스에 5곳 반복되는
**구조 패턴**("~와의 차이" 대조 섹션, "목적/개요" 서두 섹션)에 작동합니다 — 실제 규제문서에서
흔한 형식이라 코퍼스가 클수록 더 자주 만납니다. (2) 반대 방향 문항(대조 섹션이 정답인 질문)을
평가셋에 **추가해 놓고** 개선했습니다 — 신호가 무차별 삭제라면 그 문항들이 깨졌을 겁니다.
질의가 "차이가 뭐야?"를 물으면 게이트가 페널티를 끕니다. (3) prior를 끄면 실패가 재현되는
ablation 테스트를 박았습니다. 그래도 32문항은 작아서, '완성'이 아니라 sweep으로 재보정
가능한 노브로 남겼습니다.

### C-5. 청킹 전략과 그 이유는?
**답변:** 두 겹입니다. 마크다운 헤딩 단위 구조 분할(주제가 청크 안에서 안 끊기게) + 긴 섹션은
500자/80자 overlap 슬라이딩 윈도우(경계에 걸린 문장 유실 방지). 각 청크 앞에
`[문서제목 > 섹션제목]`을 붙이는데, Anthropic Contextual Retrieval 아이디어의 경량판입니다 —
청크가 고립되어도 어느 맥락인지 검색 신호에 반영됩니다. 그리고 청크가 doc_id·버전·상태
메타를 들고 다니게 해서 출처 표시와 버전 필터가 청크 레벨에서 가능해집니다.

### C-6. TF-IDF는 진짜 임베딩이 아니지 않나? 실서비스에서는?
**답변:** 맞습니다. TF-IDF는 희소·어휘 기반이라 의미 임베딩의 대체가 아닙니다. 데모의
제1원칙이 "키 없이 항상 실행"이라 기본값으로 택했고, 그 갭을 질의확장·리랭킹으로 메운 것이
이 프로젝트의 최적화 스토리입니다. 대신 `EmbeddingProvider` 인터페이스에 실제 상용 임베딩
API를 호출하는 `VoyageEmbedder`를 구현해 뒀습니다 — 교체 지점이 가설이 아니라 동작하는
코드로 존재합니다. 남은 정직한 갭도 압니다: "계속 제출해야 하나?"와 "정기 보고(PSUR)" 사이의
의미 간극은 어휘 신호로 못 메웁니다 — 밀집 임베딩이 밥값을 하는 지점이고, 다음 확장
1순위입니다.

### C-7. 검색 품질을 어떻게 측정했고, 평가셋은 어떻게 만들었나?
**답변:** 32문항 QA셋으로 Hit@1·MRR·ContextRecall·HardNegHit@1을 측정합니다. 핵심은 평가셋
설계인데 — 초기 6문서 코퍼스에서는 모든 모드의 수치가 동일해서(0.875) 최적화 효과가 안
보였습니다. 그래서 어휘는 겹치지만 정답이 다른 **하드네거티브 문서**(의료기기 허가·화장품
표시·DMF 등)를 일부러 추가해 코퍼스를 어렵게 만들었습니다. 평가가 변별력이 없으면 최적화도
증명이 안 됩니다 — "평가 설계가 곧 실력"이라는 게 이 프로젝트에서 배운 것입니다. 소표본이라
Wilson 95% 신뢰구간을 병기합니다: Hit@1 1.000의 실제 의미는 [0.893, 1.000], '완벽'이 아니라
'이 표본에서 실패 관측 0'입니다.

## D. Agent · Function Calling · MCP

### D-1. Function Calling과 MCP의 관계를 설명해보라.
**답변:** Function Calling은 "모델이 도구를 호출하는 능력"(Anthropic API의 tools 파라미터)이고,
MCP는 "도구를 표준 규격으로 제공하는 프로토콜"입니다. 이 데모는 MCP 서버의 도구 스키마를
`list_tools()`로 받아 Anthropic tools 포맷으로 변환해 둘을 연결합니다. 도구를 MCP로 분리하면
N개 모델 × M개 도구의 통합이 N+M으로 줄고, 같은 도구 서버를 Claude Desktop·Cursor·사내
에이전트가 재사용합니다.

### D-2. 에이전트 루프의 안전장치는?
**답변:** 세 가지입니다. (1) 스텝 상한 6회 — 무한루프 방지. (2) 도구 실패 흡수 — 예외를
크래시 대신 `is_error=True`인 tool_result로 모델에 되먹여, 모델이 스스로 다른 도구나 인자로
재시도하게 합니다. (3) 시스템 프롬프트로 근거 강제 — "반드시 도구로 조회 후 근거 기반으로만,
근거 없으면 모른다고". 여기에 관측성이 겹칩니다 — 모든 스텝이 span으로 기록되어 어떤 도구가
몇 ms에 성공/실패했는지 추적됩니다.

### D-3. LLM 에이전트의 도구 선택 품질은 어떻게 올렸나?
**답변:** **docstring이 곧 LLM용 사용설명서**라는 원칙으로요. FastMCP는 함수의 타입힌트와
docstring으로 도구 스키마를 만드는데, 각 docstring에 "언제 이 도구를 쓰고 언제 다른 도구를
써야 하는지"를 명시했습니다(예: assess_adverse_event에 "규정 자체가 궁금하면
search_regulations 사용"). 도구도 의도 단위로 분리했습니다 — "언제까지 보고?"(판정)와
"보고서 만들어줘"(산출물)는 다른 도구지만 같은 판정 규칙 모듈을 공유해 결과 불일치가
없습니다.

### D-4. API 키가 없을 때도 동작한다는데, 어떻게?
**답변:** 오프라인 모드가 같은 MCP 도구를 규칙 라우터로 호출합니다. 케이스 서술(환자+사건
어휘의 AND 조건)이면 PV 도구, 마감 어휘면 마감일 도구, 기본은 검색 — LLM 모드의 도구 선택과
1:1 대응되는 분기입니다. 답변은 근거 발췌로 조립합니다. 정확히 말하면 두 모드가 공유하는
것은 **도구 인터페이스·출처 체계·사후 검증 게이트**까지고, **회피 판정 방식은 다릅니다** —
문턱 기반 abstention 은 오프라인 검색 경로 전용이고, LLM 모드는 시스템 프롬프트의 도구
강제 + 사후 게이트 + grounded 신호(도구 근거 확보 여부 — 도구 없이 답하면 False 라벨)가
그 역할을 나눠 맡습니다(아키텍처.md 7-3 에 경계로 명문화). 처음에는 이 차이를 "abstention 도
동일"이라고 뭉뚱그려 썼는데, v7 점검에서 범위의 과확장으로 판정해 정정한 지점입니다.
엔터프라이즈 환경에서 외부 API 장애·키 정책 변화에도 데모와 핵심 기능이 죽지 않는
graceful degradation 입니다.

### D-5. MCP Prompts는 왜 썼나? 도구만 있으면 되지 않나?
**답변:** 도구만 배포하면 '올바른 사용 순서'는 클라이언트마다 제각각이 됩니다.
`pv_case_intake` 프롬프트는 케이스 접수 SOP(트리아지→근거 확인→초안→누락 안내)를 서버가
배포합니다 — 어떤 클라이언트가 붙어도 같은 절차로 처리하게요. 조직의 업무 절차를 코드처럼
버전 관리해 배포하는 것, 이게 사내 MCP 플랫폼에서 Prompts의 실질적 의미라고 생각합니다.

## E. 신뢰성 · 환각 억제 · 검증

### E-1. 환각을 어떻게 억제하나?
**답변:** 3중 방어입니다. (1) **입력 측**: 시스템 프롬프트로 도구 조회 강제 + 모든 답에 출처
부착. (2) **판정 측**: abstention — 근거 관련도와 질의 커버리지 **둘 다** 문턱 미달일 때만
"근거 없음"으로 회피(AND 조건 — OR면 구어체 정상 질문까지 과회피합니다). 문턱은 감이 아니라
범위내/밖 점수 분포를 실측해 마진 중앙에 놓았고, 리랭커가 바뀌면 재보정합니다.
(3) **출력 측**: 사후 검증 게이트가 답변 속 수치·날짜·방향 한정어를 근거와 대조합니다.
측정: AbstentionAccuracy 1.000, OverAbstain 0.000, AnswerGroundedness 1.000.

### E-2. 답변 사후 검증 게이트는 왜 필요한가? 평가로 충분하지 않나?
**답변:** 평가는 오프라인 **표본** 측정입니다. 운영에서 LLM 답변은 매 요청 새로 생성되고,
"15일"이 "30일"로 바뀌는 생성 오류를 평가셋은 못 막습니다. 그래서 **전수**를 지키는 런타임
게이트를 따로 뒀습니다. 디테일 두 가지 — 신뢰 소스는 '검색 근거 ∪ 결정론적 도구 출력 −
질문 에코'입니다. 도구가 계산한 마감일은 근거 원문에 없으니 포함해야 하고(안 하면 오탐),
사용자 질의 에코는 빼야 합니다(안 빼면 사용자가 틀린 수치를 전제로 물었을 때 맞장구쳐도
통과하는 구멍). 실패 방향은 차단이 아니라 경고 부착입니다 — 검증기 자신도 오탐 가능성이
있고, 원칙은 '사람의 최종 확정을 빠르게'니까요.

### E-3. "그 검증기는 누가 검증하나?"
**답변:** 메타모픽 평가로요(`eval/verify_eval.py`). 정상 답변에 결정론적 변조 — 교차문서 수치
치환(의료기기 80 근무일↔의약품 120 근무일), 오프셋(15일→22일), 방향 뒤집기(이내→이후),
고유어 치환(15일→열흘), 날짜 시프트, 날짜 역할 스왑(인지일↔마감일), 부분 날짜 시프트
("7월 25일"→"7월 28일") — 를 가해 오류 케이스를
합성하고 탐지율을 잽니다. 7축 84건 전부 탐지, 동시에 정상 답변 통과율(CleanPassRate)로
오탐도 감시합니다 — 오탐이 늘면 alert fatigue로 계층 전체가 죽으니까요. 해석 규율도
중요합니다: 합성 축의 1.0은 '핀'(깨지면 검증기 회귀)이고, 진짜 실측은 실응답 전수가
게이트를 통과하는 E2EPassRate 하나라서 표에서 따로 읽습니다. 여기에 더해 배포 시점에는
preflight가 안전장치 자가 테스트(심은 오류가 실제로 걸리는가)를 수행합니다 — 고장난
안전장치를 단 채 배포되는 것은 없는 것보다 나쁘기 때문입니다. 이 자가 테스트는 검증
게이트에만 있다가 PII 마스킹으로 확장했습니다 — 같은 원칙을 한 안전장치에만 적용하는
비대칭 자체가 사각지대였습니다.

### E-4. 날짜 역할 대조 축은 왜 추가했나? 존재 대조로 충분하지 않나?
**답변:** 존재 대조에는 **정의상 못 잡는** 변조가 있기 때문입니다. 트리아지 도구의 출력에는
날짜가 여러 개 있습니다(인지일·마감일). 답변이 "보고 기한은 7월 10일입니다(인지일 7월 25일)"
처럼 두 날짜의 역할을 맞바꿔도, 각 날짜는 신뢰 소스에 실존하므로 '존재하는가' 검사는
통과합니다 — 그리고 이건 기한을 앞당겨 읽게 만드는, 컴플라이언스에서 가장 위험한 형태의
오류입니다. 해법은 방향 한정어와 같은 원리입니다: 결정론적 도구는 날짜를 역할 키
(deadline_date·awareness_date)로 라벨링해 출력하므로, 답변에서 역할 키워드(기한/마감·인지일)에
직접 붙은 날짜를 그 라벨과 대조할 수 있습니다 — 닫힌 키워드·라벨 기반이라 기계 검증이
가능한 영역이에요. 보수성도 같습니다: 라벨이 없으면(검색 근거만 있으면) 판단하지 않고,
키워드가 날짜에 직접 붙지 않으면("기한 규정은 2025-04-01 시행") 역할 주장으로 보지 않습니다.
추가하면서 메타 평가에 DateRoleSwapDetection 축을 함께 넣어 탐지를 핀으로 고정했고,
기존 DateCleanPassRate가 새 축의 오탐 감시를 겸합니다. '기계 검증 가능'의 경계는 고정된
선이 아니라 재심사할수록 넓어지는 선이라는 게 이 프로젝트에서 반복된 패턴입니다 —
방향 한정어(v2)도, 역할 대조(v3)도 처음엔 'LLM judge의 몫'으로 분류했던 것들입니다.

### E-5. 버전 인지 검색은 왜 필요하고 어떻게 했나?
**답변:** 규제는 개정이 잦고 **폐지된 옛 기준을 인용하면 그 자체가 컴플라이언스 사고**이기
때문입니다. 문서 frontmatter에 version·effective_date·status를 두고 청크로 전파해,
폐지본(superseded)은 기본 검색에서 제외하고 `as_of`로 과거 시점 조회, `include_superseded`로
개정 이력 조회를 지원합니다. 증명을 위해 구판 문서(보고기한 30일)와 현행(15일)이 충돌하게
코퍼스에 일부러 심었고, 기본 검색은 15일만 답함을 테스트로 보증합니다. 사후 검증 게이트도
이력 요청이 없는데 폐지본이 출처에 섞이면 경고를 붙입니다.

### E-6. 기능이 다 각각 테스트를 통과하는데도 시스템이 틀릴 수 있나? 실제 사례가 있나?
**답변:** 있습니다 — 이 프로젝트에서 가장 배운 게 많았던 결함입니다. 버전 인지 검색에는
두 필터가 있습니다: '폐지본 기본 제외'와 '시점(as_of) 조회 — 그 시점에 시행 전인 문서
제외'. 각각은 단위 테스트를 통과했고 각각의 명세도 맞습니다. 그런데 둘을 **그대로 겹치면**
개정된 적 있는 규정의 과거 시점 조회에서 구판은 폐지 필터에, 신판은 시행일 필터에 걸려
**아무 버전도 안 나옵니다**. "시점 조회를 지원한다"는 주장이 정작 개정 이력이 있는 —
즉 시점 조회가 필요한 유일한 — 규정에서만 무너지는 조합 결함이죠. 수정은 의미론을 바로
세우는 것이었습니다: as_of 조회는 '그 시점의 현행'을 반환한다 — 폐지본이라도
[시행일, 후속본 시행일) 구간이면 당시 현행이므로 포함합니다. 그리고 이 의미론이 딛고
선 데이터 전제(폐지 체인의 시행일 단조성)를 preflight 검사로, 의미론 자체의 생존을 시점
조회 canary 로 고정했습니다. 교훈은 두 가지입니다 — 필터의 교집합은 필터별 테스트가
못 잡는다(조합을 검사하라), 그리고 "기능 X를 지원한다"는 문서 주장마다 "X가 가장 필요한
케이스에서 실제로 동작하는가"를 물어야 한다는 것입니다.

### E-7. 검증 게이트 자체에도 사각지대가 있었나?
**답변:** 세 개를 찾아 고쳤고, 셋 다 '조용해서 위험한' 형태였습니다. (1) **한국어 날짜
표기** — 검증기는 ISO(2026-07-25)만 알았고 "2026년 7월 25일"은 날짜로 추출조차 안 됐습니다.
위험한 점은 대칭성 때문에 **아무 신호가 없었다**는 것입니다: 답변과 근거 양쪽에서 똑같이
안 잡히니 오탐도 없고, '검증할 클레임이 수집되지 않는' 실패는 통과처럼 보입니다. 게다가
날짜의 '25일' 성분이 기간 클레임("25일")으로 오추출되어 근거의 "15일 이내"류 기간과 우연히
매칭될 수도 있었죠. 표기 정규화(환산이 아니라 보름=15일과 같은 철학)로 고쳤습니다.
(2) **경계 문자 가정** — 날짜 정규식의 `\b`는 한글이 `\w`라는 사실 때문에 "2026-07-25입니다"
처럼 조사가 바로 붙는(한국어에서 가장 흔한) 표기를 아예 못 잡았습니다. 룩어라운드로
교체했습니다. (3) **에러 응답의 승격** — 형식 틀린 as_of 에 도구가 명시적 에러를 답하게
했더니, 그 에러 문구("as_of '2024/06/01' 가 형식이 아님")가 신뢰 소스에 들어가 사용자의
잘못된 입력이 '근거'로 승격되는 새 구멍이 열렸습니다. 수정이 새 구멍을 만든 사례라서,
에러 계약 응답은 신뢰 소스에서 통째로 제외하고 에러 문구에 예시 날짜도 넣지 않도록
고쳤습니다. 셋 다 회귀 테스트로 고정되어 있습니다.

### E-8. '안전장치를 끄는 스위치'는 어떻게 다뤄야 하나?
**답변:** 일반 기능보다 오탐 비용이 비싸다는 것을 인정하고, 발동 조건을 더 보수적으로
잡아야 합니다. 이 프로젝트의 사례로 — 이력 조회 의도가 감지되면 폐지본을 검색에 포함하고
동시에 '폐지본 인용 경고'를 끕니다(이력 조회에서 폐지본은 결함이 아니라 목적이니까요).
처음 감지 어휘에 "이력"과 "폐지"를 넣었는데, 리뷰에서 반례가 나왔습니다: "접수·처리 이력
관리 요건은?"(현행 질문), "이 규정 폐지됐나요?"(현행 상태 질문)도 매칭됩니다. 일반 라우팅
오탐이면 검색 결과가 조금 달라지고 말지만, 이건 **버전 경고까지 꺼진 채** 폐지 규정이
현행처럼 나가는 오탐입니다. 그래서 명시적 이력 구문("예전", "구판", "개정 이력"…)만 남기고
확신 없는 어휘는 뺐습니다 — 코딩 사전의 '확신 없는 항목은 안 넣는다'와 같은 철학인데,
스위치가 안전장치를 무력화하는 종류일수록 이 보수성이 더 중요해집니다. LLM 모드의
allow_superseded 도 같은 규율을 적용해, as_of 검색이 **성공했을 때만** 켜집니다(형식
오류로 실패한 호출이 경고만 끄는 것을 방지).

### E-9. 안전장치를 다 만든 뒤에도 사각지대는 남는다 — 어떻게 계속 찾아내나?
**답변:** 완성된 시스템의 사각지대는 기능 버그가 아니라 **논리적 비약** 속에 숨습니다.
그래서 두 가지 방법을 씁니다. 첫째, **경계 주장의 반례 경로 탐색** — "입구에서 마스킹하므로
이후 전부 안전"이라는 명제에는 '모든 입력이 입구를 지난다'는 숨은 전제가 있습니다. MCP
서버를 stdio로 단독 사용하면 도구 인자(보고자·환자 정보)가 입구를 우회합니다 — 실제로
case_description 만 마스킹하고 부가 인자는 통과시키는 우회로를 찾아 도구 계층 마스킹으로
막았습니다. 둘째, **조용한 실패 인벤토리** — "이 코드가 틀렸을 때 사용자가 신호를 받는가"를
경로마다 묻습니다. 그렇게 찾은 조용한 실패가 여섯인데 — 잘못된 형식의 인지일을 조용히
오늘로 폴백해 기한이 틀린 기준일로 계산되던 것(→ caveat 명시), 오타 난 마감일 유형 필터에
"마감 없음"이라는 자신 있는 오답을 주던 것(→ error+available 계약), 형식 틀린 as_of 를
조용히 무시하고 현행 기준으로 답하던 것(→ 같은 에러 계약의 대칭 적용), 오프라인 라우터가
미지원 체크리스트 요청에 '품목허가'를 기본 추측해 도구의 에러 계약을 우회하던 것(→ 추측
제거 + 지원 목록 안내 + 검색 폴백), 중첩 span 합산으로 지연 계기판이 실제의 ~2배를
보고하던 것(→ wall-clock 계산), 그리고 개정된 규정의 시점 조회가 0건이 되던 필터 조합
결함(→ '당시 현행' 구간 의미론, E-6)입니다. 전부 답이 그럴듯해서 운영 중엔 절대 신고되지
않는 형태고, 그래서 각각 회귀 테스트로 고정했습니다. 안전장치 자가 테스트의 **대칭성**도
같은 방법으로 찾았습니다 — 검증 게이트는 자가 테스트하면서 PII 마스킹은 안 하고
있었으니까요. 셋째, **자기 수정에 대한 적대적 리뷰 루프** — 사각지대를 고친 코드가 새
사각지대를 만들 수 있습니다(실제로 as_of 에러 계약 추가가 '에러 문구 속 사용자 입력의
신뢰 소스 승격'이라는 새 구멍을 열었습니다, E-7). 그래서 수정 뒤 '이 수정이 틀렸다면
어떻게 틀렸을까'를 별도 리뷰 패스로 다시 묻고, 거기서 나온 반례를 다시 회귀 테스트로
고정합니다(F-8).

### E-10. 사용자가 케이스로 제공한 정보는 신뢰 소스인가?
**답변:** "반쯤"이라는 게 정직한 답이고, 그 반쯤을 구조로 만들었습니다. 케이스 서술은
사용자가 제공한 '사실'이라 답변이 재서술하는 것은 정당합니다 — 여기에 경고를 붙이면
보고서 초안마다 오탐이 나서 alert fatigue로 검증 계층이 죽습니다. 그런데 "케이스는
사실이다"에서 "케이스가 답변의 모든 클레임을 지지할 수 있다"로 넘어가면 그건 논리적
비약입니다: 환자가 "30일간 복용"했다는 서술이, 답변의 "보고 기한 30일"이라는 **규정
클레임**의 근거로 승격되어 조용히 통과할 수 있으니까요. 기계는 '케이스 재서술'과
'케이스 수치가 우연히 규정 클레임을 지지'를 구분할 수 없습니다. 그래서 3택 —
차단(재서술 오탐), 침묵(승격 은폐), 라벨 — 중 라벨을 택했습니다: 신뢰 소스를
2계층(규정·도구 = strict / 케이스 에코 = user facts)으로 나누고, 케이스 계층에서만
지지되는 클레임에 `from_case` 라벨을 붙여 API·UI·감사 로그에 노출합니다. 경고 종류를
조정하는 from_question 라벨과 같은 철학 — 신뢰하지도, 조용히 넘기지도 않는다 — 의
'지지된 클레임' 버전입니다. 한계도 명시해 뒀습니다: 보고서 초안(draft_markdown)처럼
케이스 서술이 다른 필드 안에 재조립된 경우는 필드 단위 분리가 잡지 못합니다.

### E-11. '안전장치를 끄는 스위치'의 오탐을 좁혔다고 했는데(E-8), 스위치의 '면적'은?
**답변:** 발동 조건뿐 아니라 **발동 범위**도 최소화해야 한다는 것을 이번 검토에서
배웠습니다. LLM 모드의 allow_superseded 는 as_of 검색이 한 번이라도 성공하면 켜지는
**응답 전역 bool**이었습니다 — 그러면 같은 턴의 '현행' 검색 쪽에 상류 결함(버전 필터
버그, status 메타 오류)으로 폐지본이 섞여 들어도 경고가 함께 꺼집니다. 검증 게이트는
정확히 그런 상류 결함을 잡으라고 있는 심층방어인데, 이력 턴 동안 버전 축이 통째로
무장해제되는 거죠. "이력 조회에서 폐지본은 목적"이라는 명제가 성립하는 범위는 '이력
검색이 실제로 반환한 문서'까지입니다 — 그래서 스위치를 bool에서 **이력 검색이 반환한
doc_id 집합**으로 바꿔, 그 문서들의 폐지본 인용만 면제합니다. 안전장치를 끄는 스위치는
발동 조건(E-8)과 발동 면적(E-11) 두 방향 모두에서 근거가 성립하는 만큼만 열려야 합니다.

### E-12. 방향 한정어 축을 만들었는데도 남아 있던 사각지대는?
**답변:** **축 간 비대칭**입니다. 방향 검증("이내"↔"이후")이 수치+단위 표기에만 붙어
있었습니다 — 근거가 "2026-07-25**까지** 제출"인 기한을 답변이 "2026-07-25 **이후**
제출"로 뒤집으면, 날짜 자체는 근거에 실존하니 존재 대조를 통과하고, 방향 축은 날짜를
보지 않으니 그냥 지나갑니다. "15일 이내→이후"와 완전히 같은 등급의 기한 왜곡이
**표기가 기간이냐 날짜냐에 따라 한쪽만 잡히는** 상태였죠. 찾은 방법은 축의 매트릭스를
그려보는 것이었습니다 — 클레임 타입(수치·날짜) × 검증 축(존재·방향·역할)을 표로 놓으면
빈 칸이 보입니다. 수정은 같은 보수 규칙의 대칭 적용입니다: 근거가 그 날짜에 대해 반대
방향만 말할 때만 플래그하고, 근거에 한정어가 없으면(도구의 JSON 라벨 등) 판단하지
않습니다 — "마감일 이후에는 지연보고로 처리된다" 같은 정당한 '이후' 용례를 오탐하지
않기 위해서요.

### E-13. 검증 대상 표기의 공간은 어떻게 넓혔나? '환산 금지' 원칙과 충돌하지 않나?
**답변:** 다섯 번째 검토(v5)에서 표기 공간의 빈 칸 세 개를 더 찾았습니다 — 연도 없는
부분 날짜("보고 기한은 7월 25일"), 연도 단독 표기("2025년 4월 개정"), 그리고 고유어
수사에 붙은 방향 한정어("보름 이후"). 셋 다 이중 결함이었습니다: 옳은 답변에는
오탐이 붙고(부분 날짜의 '25일'이 기간으로 오추출, 연도가 미확인 수치로 판정), 틀린
표기는 아예 추출되지 않아 조용히 통과했죠. 기준은 '환산 금지' 원칙과 정확히
양립합니다 — "15일→약 2주"는 값을 **바꾸는 환산**이라 금지하지만, "2026-07-25"의
월-일 성분을 "7월 25일"로, 연도 성분을 "2025년"으로 쓰는 것은 값이 같은 **표기
변형**입니다(보름=15일과 같은 계열). 판정 질문은 하나예요: "이 대조를 통과시키면
근거에 없는 **새 값**이 생기는가?" 생기면 환산(금지), 안 생기면 표기 변형(정규화
대상)입니다. 부분 날짜는 연도를 지어내지 않도록 월-일 접미로만 대조하고, 연도
표기는 근거 날짜의 연도 성분으로만 대조합니다.

### E-14. 2계층 신뢰 소스를 만들었는데도 남아 있던 비약은?
**답변:** **계층 설계를 존재 축에만 적용한 비일관성**입니다. 케이스 서술은 존재
대조에서는 별도 계층(from_case 라벨)으로 다뤄지는데, 방향 한정어 대조는 규정 근거와
케이스를 **합산한 지도**로 판정하고 있었습니다. 그러면 케이스의 "복용 15일 이후 증상
발생"이 규정의 "15일 이내" 방향 뒤집기 경고를 **조용히 무력화**합니다 — 답변이 "보고는
15일 이후에 하면 됩니다"라고 해도 케이스 문장 덕에 통과하죠. 컴플라이언스에서 가장
위험한 방향 왜곡이 사용자 서술로 면제되는 셈입니다. 수정 원칙은 "충돌 판정의 기준은
strict 계층(규정·도구), 케이스는 판정 근거가 아니라 **모호성 라벨의 근거**"입니다 —
케이스에 같은 방향 표현이 있으면 경고를 끄는 대신 from_case 를 붙여 '케이스 재서술인지
규정 왜곡인지'를 사람이 가리게 합니다. 존재 축에서 했던 3택(차단·침묵·라벨)의 라벨
선택을 방향 축에도 똑같이 적용한 겁니다. 부수 효과도 있습니다 — 케이스에만 있는
한정어가 거꾸로 충돌을 '만들어내는' 오탐 경로도 함께 사라졌습니다(규정 근거 없는 판정
금지). 교훈: 계층·라벨 같은 구조적 결정은 **모든 축에 대칭으로** 적용됐는지 감사해야
하고, 한 축에만 적용된 구조는 그 자체가 다음 사각지대의 좌표입니다.

### E-15. 미탐만 잡으면 되지, 오탐 제거에 왜 공을 들이나?
**답변:** 검증 계층은 두 방향으로 죽기 때문입니다. 미탐이 쌓이면 잡아야 할 것을
못 잡아 죽고, **오탐이 쌓이면 경고가 무시되기 시작해서** 죽습니다 — alert fatigue 는
전 축을 한꺼번에 무력화하니까 오히려 더 치명적입니다. 그래서 이 프로젝트는 오탐
경로를 미탐 구멍과 같은 등급의 결함으로 셉니다. v5에서 제거한 오탐이 그 예입니다 —
근거가 "2025-04-01"(날짜)로만 말하는데 답변이 "2025년 개정"이라고 쓰면 '2025년'이
미확인 수치로 경고되고, "7월 25일"이라고 쓰면 '25일'이 기간 클레임으로 오추출되어
경고됐습니다. 둘 다 옳은 답변입니다. 메타 평가도 이 철학대로 양방향입니다 — 탐지율
7축(잡아야 정상)과 오탐 감시 4축(통과해야 정상)을 함께 재고, 둘 다 pytest 가드가
CI 실패로 강제합니다. preflight 자가 테스트에도 clean 케이스가 들어 있어 '심은 오류는
걸리고 정상은 통과한다'를 배포 전에 양방향으로 확인합니다.

### E-16. 검토를 반복하면서 '결함'의 판정 기준은 뭐였나? 아무거나 고치다 보면 끝이 없지 않나?
**답변:** 기준을 명문화해서 문서(description/아키텍처.md 7장)에 남겼습니다. 두 질문으로
요약됩니다 — 논리적 비약은 **"전제를 대라"**: 설계 명제마다 숨은 전제(모든 입력이
입구를 지난다, 트래픽 믹스가 일정하다, 이력 면제는 그 문서까지만 성립한다)를 대게
하고, 그 전제를 검사하는 코드가 없으면 결함입니다. 사각지대는 **"고장 나면 누가
아는가"**: 실패했을 때 아무 신호가 없는 경로 — 특히 답변·근거 양쪽에서 똑같이 추출되지
않아 오탐조차 없는 '대칭적 비검출'과, 경고가 아니라서 죽어도 소리가 없는 라벨 —
를 결함으로 셉니다. 탐색 절차도 정해져 있습니다: 축×표기 매트릭스의 빈 칸 찾기,
조용한 실패 인벤토리, 오탐도 결함으로 세기, 안전장치 자신을 검사 대상에 올리기.
그리고 끝이 없지 않도록 **반대면**도 명문화했습니다 — 보수적 미판정, 트리아지의 과탐
방향, draft_markdown 재조립 같은 '알고 남긴 경계'는 결함이 아니라 문서화된 스코프
결정입니다. 기준이 없으면 검토가 취향이 되고, 반대면이 없으면 검토가 무한이 됩니다.

### E-17. 형식 검증은 다 했는데, 값 자체가 말이 안 되는 입력은?
**답변:** 그 비대칭도 v5에서 찾았습니다. 인지일은 형식 오류(YYYY-MM-DD 아님)에는
시끄러운 caveat 를 달아 놓고, 형식은 맞지만 **미래 날짜**인 인지일(2027 오타 등)은
조용히 통과시키고 있었습니다 — 보고 기한이 1년 뒤로 밀려도 아무 신호가 없죠. 형식
검사(parse 가능한가)와 타당성 검사(값이 도메인에서 말이 되는가)는 다른 층인데, 한쪽만
있으면 나머지가 사각지대가 됩니다. 수정은 같은 원칙의 연장입니다 — 미래 인지일이면
계산은 입력값대로 하되(자동 정정은 또 다른 조용한 폴백이니까) '입력 오류 가능성'
caveat 를 명시합니다. 실무로 일반화하면: 날짜 필드마다 형식/범위/관계(인지일 ≤ 오늘,
시행일 단조성 — 이건 preflight 가 코퍼스에 대해 이미 검사합니다) 세 층을 구분해서
어느 층까지 검사하는지 명시하는 것이 입력 검증 설계라고 생각합니다.

### E-18. 검증기에서 배운 교훈이 다른 안전장치에도 적용됐는지는 누가 확인하나?
**답변:** 그 질문 자체가 v6(20렌즈 점검)에서 가장 무거운 발견을 만들었습니다. 검증기의
날짜 추출은 "정규식 `\b`는 한글이 `\w`라서 한글 직결 표기에서 경계가 성립하지 않는다"는
교훈을 배워 룩어라운드로 전환돼 있었습니다. 그런데 **같은 저장소의 PII 마스킹 정규식은
여전히 `\b`를 쓰고 있었습니다** — "연락처는 010-…-5678로 부탁드립니다", "주민번호는
…입니다"처럼 조사가 직결된, 한국어에서 가장 흔한 문장의 전화·주민번호가 통째로 마스킹을
벗어나 외부 API·로그로 나가는 경로였죠. 검증기의 비검출은 '경고 누락'이지만 마스킹의
비검출은 '유출'이라 같은 결함이라도 실패 비용이 더 큽니다. 반대 방향 결함도 함께
나왔습니다 — 이메일 정규식의 `[\w.+-]`는 한글을 삼켜 "…com입니다"의 '입니다'까지
마스킹했는데, 값은 가려지니 유출은 아니지만 마스킹 리포트가 원문 범위를 왜곡합니다
(오탐도 결함으로 세는 원칙의 마스킹 버전). 수정은 룩어라운드·ASCII 한정이고,
preflight PII 자가 테스트에 한글 직결 표기를 심어 재발을 구조로 막았습니다. 교훈을
일반화하면 — 대칭성 감사는 한 장치의 축들 사이만이 아니라 **장치들 사이(교훈의 전파)**
에도 돌려야 하고, "A에서 고친 버그의 패턴이 B에도 있는가"는 버그 수정 절차의 마지막
단계여야 한다는 것입니다.

### E-19. 트리아지 도구가 에코하는 인지일(awareness_date)은 신뢰 소스인가? as_of 는 빼면서 왜 이건 남기나?
**답변:** 남기는 것이 맞고, 그 이유를 문서화해 뒀습니다(아키텍처.md 7-3). 겉보기엔 둘 다
'사용자 입력의 에코'지만 성격이 다릅니다. as_of 는 검색 필터 파라미터의 **단순 반사**라
그 값 자체가 어떤 검증도 거치지 않고, 신뢰 소스에 넣으면 사용자가 지정한 기준일이 답변
날짜 클레임의 '근거'로 승격됩니다. 반면 인지일은 도구가 형식 검사(오류면 caveat)와
타당성 검사(미래 날짜면 caveat)를 거쳐 **채택한 계산 기준**이고, 출력의 마감일이 그 값의
함수입니다 — 기준일을 신뢰 소스에서 빼면 옳은 답변이 인지일을 재서술할 때마다 오탐이
붙습니다. 즉 판단 기준은 "에코인가"가 아니라 "**도구가 그 값을 검사하고 caveat 로 감싼
뒤 책임지고 출력했는가**"입니다. 검사 없는 자유 텍스트 에코(케이스 서술)가 from_case
라벨 계층으로 분리되는 것과 등급이 갈리는 지점이 정확히 여기입니다.

## F. 측정·검증 문화 (FDE의 신뢰 구축 방식)

### F-1. 규칙 기반 모듈까지 수치 평가를 만든 이유는?
**답변:** 단위 테스트는 개별 규칙을 지키지만, **사전 한 줄·정규식 한 글자를 고쳤을 때 전체
품질이 어디로 움직였는지**는 못 잡습니다. 그래서 라벨 22케이스로 PV 전 단계를 측정하는
하네스를 만들었습니다. 라벨은 코드 출력의 복사가 아니라 도메인 기준으로 독립 작성했고,
코드가 틀리길 기대하는 문항(사전이 못 잡는 롱테일 구어)을 일부러 넣어 재현율이 정직하게
1.0 미만이 나오게 설계했습니다 — 동어반복이면 전부 1.000이어야죠. 이 하네스의 실용 가치는
회귀 감지입니다. CI가 매 푸시마다 돌립니다.

### F-2. 하이퍼파라미터는 어떻게 정했나?
**답변:** 전부 스윕·ablation으로 정했고, 일회성 실험이 아니라 재현 스크립트
(`python -m eval.sweep`)로 남겼습니다. 정직함 하나 — alpha와 idf_power는 이 코퍼스 규모에서
지표가 안 움직입니다. 그래서 억지로 '최적값'을 주장하지 않고 중립값을 문서화해 남겼고,
코퍼스가 커지면 같은 스크립트로 재보정합니다. 반대로 서두 섹션 감쇠(0.055)는 유효 밴드가
0.04~0.07로 좁다는 것 자체가 위험 정보라서, 경계에서 깨지는 문항을 pytest 회귀 가드로
박아뒀습니다.

### F-3. 실패한 실험도 있었나?
**답변:** 있고, 기록해뒀습니다. "정답 섹션은 질의어를 반복한다"는 가설로 tf 포화 밀도 신호를
넣었는데, 하드네거티브의 대조 섹션이 정확히 그 용어를 반복하는 바람에 지표가 악화돼
제거했습니다. 질의확장에서도 "DMF" 확장이 주제 표류를 일으켜 뺐습니다. 기록하는 이유는
"왜 그 신호는 없나"에 수치로 답할 수 있고, 같은 가설을 다시 시도하는 낭비를 막기 위해서입니다.
개선 과정 전체가 오류 분석 루프였습니다: Hit@1 0.867→0.900(동의어 보강)→0.967(리랭커 필드
분리)→1.000(섹션 타입 prior).

### F-4. 지표가 전부 1.000이면 데이터셋이 쉬운 것 아닌가?
**답변:** 맞는 지적이고, 두 가지로 답합니다. (1) 소표본 지표에는 Wilson 95% 신뢰구간을
병기합니다 — 1.000의 실제 의미는 [0.893, 1.000]이고, 두 구간이 크게 겹치면 '개선'이 아니라
'구분 불가'로 읽는 규율을 지킵니다. (2) 절대 수치보다 **보정 과정**을 봐달라고 말씀드립니다.
처음엔 실패가 있었고, 원인을 분류해 고쳤고, 그 과정이 스크립트로 재현됩니다. 데이터가 커지면
같은 하네스로 다시 벌어진 갭을 측정하면 됩니다.

### F-5. 시스템 개선 시 부작용(회귀)은 어떻게 관리하나?
**답변:** 실제 사례로 답하겠습니다. 리랭커를 개선했더니 abstention 과회피가 **악화**됐습니다.
원인은 회피 판정의 커버리지 신호가 원 질의만 봐서 — 검색은 동의어 확장으로 정답을 찾는데
회피 판정은 "근거 없음"을 선택하는 자기모순이었습니다. 신호를 확장 질의 기준으로 고치고
분포를 재실측해 문턱을 다시 잡았고, '구어 범위내 질문이 회피되면 안 된다'는 회귀 테스트를
박았습니다. 교훈: 파이프라인은 부품이 아니라 **결합된 시스템**이고, 문턱값이 리랭커 점수
스케일에 결합돼 있으면 리랭커 변경 시 재보정이 선택이 아니라 필수 절차라는 것 — 그 절차를
테스트가 강제하게 만들었습니다.

### F-6. 배치 후에 이 시스템이 계속 건강한지는 뭘로 아나?
**답변:** 두 계기로요. (1) **검증 게이트 경고율 계기판** — 게이트의 통과/경고를 축별로
집계해 `/health`에 `warn_rate`로 노출합니다. 검증 계층의 운영 리스크는 오탐 하나하나가
아니라 **경고율의 추이**입니다: 경고가 잦아지면 담당자가 경고를 무시하기 시작하고(alert
fatigue) 그 순간 계층 전체가 사실상 죽는데, 이 신호를 볼 수단이 없으면 죽는 순간을 모른 채
지나갑니다. 경고율 상승은 '답변 품질 회귀' 또는 '검증기 오탐 증가' 둘 중 하나라 어느 쪽이든
조치 대상입니다. (2) **응답 단위 감사 로그** — 판정 요약(JSON, PII 없음)을 매 응답 남깁니다.
규제 도메인에서 "그 답변이 그때 검증을 통과했는가"는 사후 감사의 질문이라서요. 여기에
기존 트레이스(span)의 지연·성패가 겹쳐져 "어제부터 느려졌어요"와 "요즘 경고가 늘었어요"
둘 다에 데이터로 답할 수 있습니다.

### F-7. abstention 문턱값을 평가셋 분포로 보정해 놓고, 같은 평가셋으로 AbstentionAccuracy 1.0을 주장하면 순환 아닌가?
**답변:** 정확한 지적이고, 인정합니다 — 문턱(SCORE_FLOOR·COVERAGE_FLOOR)은 범위내/범위밖
문항의 점수 분포를 실측해 마진 중앙에 놓은 값이라, 같은 문항들로 잰 AbstentionAccuracy
1.0은 학습 데이터 정확도에 가깝지 held-out 일반화 성능이 아닙니다. 그래서 이 수치를 읽는
규율을 따로 둡니다: (1) 그 1.0이 주장하는 것은 "이 분포에서 마진이 존재하고 문턱이 그 안에
있다"까지이며, 마진의 실체가 실질 정보입니다. 여기서 정확히 말할 부분이 있는데 —
이 마진은 **1차원이 아니라 2차원(AND)**입니다. 범위내 최소 '점수'는 0.156으로 사실
SCORE_FLOOR(0.19) **아래**입니다. 그 문항이 회피되지 않는 것은 커버리지 축(0.314 >
COVERAGE_FLOOR 0.28)이 구제하기 때문이고, 반대로 범위밖 최대는 점수 0.167·커버리지
0.250으로 **두 축 모두** 문턱 아래라 회피됩니다. 즉 "범위밖 최대 0.167 vs 범위내 최소
0.201"처럼 점수 축 하나로 요약하면 그 자체가 논리적 비약이 됩니다(0.201은 '커버리지가
최저인 문항의 점수'라는 조건부 수치입니다). 마진이 좁아지면 문턱이 아니라 검색을
고쳐야 한다는 신호로 읽되, 어느 '축'의 마진이 좁아졌는지까지 봐야 합니다. (2) 새 문항은 문턱 보정에
쓰지 않고 먼저 평가로 태워 마진이 유지되는지 봅니다. (3) 실배치에서는 고객 질문 로그로
홀드아웃 셋을 따로 만들어 재검증하는 것이 첫 절차입니다 — 데모의 32+8문항은 보정용이지
일반화 증명용이 아니라는 것을 구분해서 말씀드리는 게 정직하다고 생각합니다.

### F-8. 코드를 고친 뒤 "고쳤다"는 것은 어떻게 확신하나? (검증 루프)
**답변:** 세 겹 루프를 절차로 돌립니다. **1차 — 재현 가능한 실행**: preflight + 전체
테스트 + 평가 하네스 5종을 돌려 수정이 기존 지표를 깨지 않았는지, 새 동작에 회귀
테스트가 붙었는지 확인합니다. **2차 — 자기 수정에 대한 적대적 리뷰**: 수정 diff 를
놓고 '이 수정이 틀렸다면 어떻게 틀렸을까'를 별도 관점(줄 단위 검토·삭제된 동작의 감사·
호출자 추적)으로 다시 묻습니다. 이 프로젝트에서 실제로 이 패스가 잡은 것들이 —
as_of 에러 계약이 신뢰 소스 승격 구멍을 새로 연 것, 이력 감지 어휘("이력"·"폐지")가
현행 질문을 과잉 매칭해 버전 경고를 끄던 것, 폴백 안내문이 회피 응답과 자기모순이던 것,
전제 완화 라벨이 대화 전체로 무한 누적되던 것입니다. 1차 실행에서는 전부 초록불이던
결함들입니다. **3차 — 문서·코드 일치성과 실경로 관측**: 문서가 주장하는 동작("이력을
요청하면 구판 노출")이 모든 모드에서 실제로 참인지 실행으로 확인합니다 — 여기서 오프라인
모드에 이력 경로가 아예 없다는 불일치를 찾았습니다. 반례가 나오면 고치고 그 반례를 회귀
테스트로 고정한 뒤 루프를 다시 돕니다 — '리뷰에서 나온 반례는 반드시 테스트가 된다'가
루프를 수렴시키는 규칙입니다.

### F-9. "이 지표는 회귀를 고정하는 핀이다"라고 했는데, 그 핀은 무엇이 강제하나?
**답변:** 이 질문이 실제로 이 프로젝트의 사각지대를 하나 잡았습니다. 검증기 메타 평가의
탐지율 축들은 "깨지는 순간이 곧 회귀인 핀"이라 불러 왔는데, CI는 그 평가 스크립트를
**실행만** 하고 있었습니다 — 스크립트는 측정·출력 후 exit 0이라, 탐지율이 0.5로
떨어져도 CI는 초록불입니다. '측정이 존재한다'와 '측정이 강제한다'는 다른 명제인데
문서가 둘을 섞어 쓰고 있었죠 — 계기판은 있는데 알람이 없는 상태. PV 평가에는 pytest
회귀 가드가 있으면서 검증기 평가에는 없던 비대칭이기도 합니다. 그래서
test_verify_eval.py 가 탐지율·오탐률을 CI 실패로 강제하게 했고, **표본 수 하한**도 함께
고정했습니다 — 변조 하네스는 치환 실패를 표본에서 제외하므로, 치환 정규식이 조용히
깨지면 표본이 줄어든 채 탐지율 1.0이 유지됩니다. 분자만 보는 지표는 분모의 수축을
못 봅니다.

### F-10. 경고율 계기판(warn_rate)의 추이만 보면 되나? 그 해석에 함정은 없나?
**답변:** 있습니다 — **분모 혼합**입니다. "warn_rate 상승 = 품질 회귀 또는 오탐 증가"
라는 해석에는 트래픽 믹스가 일정하다는 숨은 전제가 있습니다. 회피 응답·수치 클레임이
없는 응답은 자명하게 통과하므로, 범위 밖 질문의 비중이 늘면 품질 변화가 없어도
warn_rate 가 내려갑니다 — 계기판이 "좋아지고 있다"는 착시를 만들죠. 그래서 전체
경고율과 함께 **클레임이 있던 응답만의 경고율(warn_rate_checked)** 을 병기합니다.
후자는 '검증할 것이 있던 응답'만 분모로 잡아 믹스 변화에 흔들리지 않습니다. 지연
계기판의 이중 계산(wall-clock)과 같은 교훈의 반복입니다 — 계기판 자체도 검증 대상이고,
추이 해석이 목적인 지표일수록 분모의 구성을 명시해야 합니다.

### F-11. LLM 모드는 API 키 없이 테스트가 안 될 텐데, 그 경로의 안전 배선은 어떻게 보증하나?
**답변:** 정확히 그게 커버리지 0의 사각지대였습니다 — "두 모드는 같은 계약"이라는 문서
주장이 LLM 모드에서는 실행으로 확인된 적이 없었죠. 신뢰 소스 수집, 이력 허용 집합,
에러 계약 제외 같은 안전 배선은 LLM 모드 쪽 코드 경로가 따로 있는데도요. 해법은 스텁의
경계를 최소로 잡는 것입니다: **anthropic 클라이언트만** 준비된 응답을 돌려주는 가짜로
바꾸고, MCP 서버·RAG·PV 도구·검증 게이트는 전부 실물을 태웁니다. 이렇게 하면 "모델이
as_of 검색을 호출하고 구판을 인용하는 턴"이나 "형식 오류 as_of 가 에러 계약으로
되먹여지는 턴"을 결정론적으로 재생해, 허용 집합이 문서 단위로 작동하는지·에러 응답이
신뢰 소스로 승격되지 않는지를 실행으로 고정할 수 있습니다. 스텁은 '모델의 지능'을
검증하지 못하지만, 검증하려는 것은 지능이 아니라 **배선**입니다 — 그 구분이 이 테스트
전략의 핵심입니다.

### F-12. preflight 가 있으니 결함 배포는 불가능한가?
**답변:** 아닙니다 — 그렇게 말하면 범위의 과확장입니다. "preflight 가 기동을 차단한다"는
명제에는 '모든 기동이 그 게이트를 지난다'는 숨은 전제가 있는데, 게이트가 자동으로
걸리는 경로는 run.sh 와 CI 두 가지뿐입니다. uvicorn 을 직접 띄우는 수동 실행(가이드의
방법 B, 포트 변경 안내)은 게이트 밖입니다. v6 점검에서 이 전제를 문서가 숨기고 있다는
것을 발견해, 가이드에 우회 경로의 존재와 '수동 실행 전 preflight 별도 실행' 권고를
명시했습니다. 실무라면 다음 단계는 전제 자체를 줄이는 것입니다 — 컨테이너
ENTRYPOINT나 프로세스 매니저에 preflight 를 박아 우회 경로를 물리적으로 없애는 쪽이요.
데모에서는 '경계를 없앴다'고 주장하는 대신 '경계가 어디까지인지'를 문서화하는 것을
선택했습니다 — 성립 범위를 명시하지 않은 안전장치 주장은 그 자체가 다음 사각지대의
좌표가 되기 때문입니다.

### F-13. 코드는 테스트가 지키는데, 문서의 주장은 무엇이 지키나?
**답변:** 문서도 검증 대상이라는 것이 이 프로젝트의 후반 교훈입니다. v6 점검에서 나온
발견 두 개가 정확히 '문서-실재 불일치'였습니다 — README 의 프로젝트 구조 트리가
CI 워크플로를 `project/` 아래에 그리고 있었는데 실제로는 저장소 루트 `.github/`에
있고(GitHub Actions 는 루트 워크플로만 인식하므로 그 위치가 맞습니다 — 틀린 것은
지도였죠), 검증기 평가 표가 v5에서 추가된 부분 날짜 2개 축을 반영하지 않고 있었습니다.
그래서 문서 점검을 코드 점검과 같은 절차로 돌립니다: 문서의 모든 수치 주장은 실측으로
재현하고(테스트 개수·평가 지표— 전부 일치했습니다), 문서의 모든 경로·구조 주장은 실제
트리와 대조하고, 표·목록은 코드의 최신 축과 대조합니다. 자동화 여지도 있습니다 —
문서 속 수치를 평가 출력에서 생성하거나, 경로 주장을 CI에서 존재 검사하는 것이요.
핵심은 "코드가 맞으면 문서는 사소하다"가 아니라는 것입니다: FDE가 고객사에 남기는
산출물의 절반은 문서고, 지도가 틀리면 다음 사람은 지도를 통째로 불신하게 됩니다.

## G. 엔지니어링 일반 · 확장성

### G-1. 이 데모를 실서비스로 가져가려면 무엇부터 바꾸나?
**답변:** 우선순위대로 — (1) 임베더: TF-IDF→상용 밀집 임베딩(`EmbeddingProvider` 자리,
Voyage 경로는 이미 동작), (2) 리랭커: 4신호 근사→실제 Cross-Encoder(섹션 메타를 입력에
포함하는 형태로 prior 지식도 이식), (3) 벡터스토어: 인메모리→pgvector/Qdrant, (4) PII:
정규식→NER(Presidio), (5) 코딩 사전→MedDRA 본체, (6) 로더: md→PDF/HWP 파서, (7) 인증·권한.
전부 이미 인터페이스로 교체 지점이 잡혀 있습니다. 반대로 바꾸지 않을 것: 컴플라이언스 계산의
결정론 원칙, 출처·버전 추적, 검증 게이트 — 이건 규모와 무관한 아키텍처 원칙입니다.

### G-2. 관측성을 데모에 왜 넣었나?
**답변:** 엔터프라이즈 에이전트에서 "무슨 도구를 몇 ms에 성공/실패로 불렀나"는 디버깅·SLA·
비용 관리의 기본이기 때문입니다. Trace/Span 구조와 구조화 로그를 의존성 없이 구현했고,
응답에 trace·latency_ms를 노출해 UI 배지로 보입니다. 실무에선 OpenTelemetry/LangSmith로
교체되는 자리입니다. FDE는 배치 후 "어제부터 느려졌어요"라는 전화를 받는 사람이라,
계측 없는 배치는 안 한다는 감각을 보여주고 싶었습니다.

### G-3. 왜 프레임워크(LangChain 등) 없이 만들었나?
**답변:** 두 가지입니다. (1) 데모가 무거운 의존성 없이 어디서든 돌아가게 — 의존성은
fastapi·fastmcp·uvicorn 수준입니다. (2) 각 단계(BM25·TF-IDF·리랭킹·tool-use 루프)를
직접 구현해 원리를 이해하고 있음을 보이려고요. 실무에서는 검증된 라이브러리를 쓰는 게
맞고, 그래서 교체 지점을 인터페이스로 명시했습니다. 프레임워크를 쓰더라도 그 아래에서
무슨 일이 일어나는지 아는 것과 모르는 것은 장애 대응에서 완전히 다릅니다.

### G-4. 한국어 검색은 어떻게 처리했나?
**답변:** 한국어는 조사·띄어쓰기 변형이 많아 어절 매칭이 약합니다. 형태소 분석기 의존성을
더하는 대신 어절 토큰 + **CJK 문자 bi-gram**을 함께 생성하는 경량 근사를 썼습니다 —
"감기약"과 "감기"가 bi-gram으로 이어집니다. 도메인 특화 갭(구어↔규정 용어)은 토크나이저가
아니라 질의확장 사전이 맡습니다. 실무 확장이라면 형태소 분석기(또는 밀집 임베딩)가 이 자리를
대체합니다.

### G-5. 이 데모의 가장 큰 약점은?
**답변:** 코퍼스(13문서)와 평가셋(32문항·22케이스)이 작아 절대 수치의 의미가 제한적이라는
점입니다. 그래서 절대치보다 모드 간 상대 비교·하드네거티브 축·신뢰구간으로 방법론을 보였고,
확장 시 같은 평가 하네스로 재검증되도록 CI에 묶었습니다. 두 번째 약점은 TF-IDF의 의미 이해
한계 — ContextRecall 잔여 실패 1건이 정확히 그 지점이고, 밀집 임베딩 교체가 첫 확장입니다.
약점을 숨기는 것보다 '측정된 갭 + 교체 지점'으로 관리하는 것이 엔지니어링이라고 생각합니다.

## H. 압박 질문 (꼬리 질문 대비)

### H-1. 이거 AI 코딩 도구로 만들었잖아. 본인이 설명할 수 있나?
**답변:** 네, 그리고 그게 FDE의 미래 업무 방식이라고 생각합니다. 중요한 건 도구가 아니라
**설계 결정의 소유권**입니다. 예를 들어 리랭커 블렌딩(rerank_weight=0.9)은 순수 재정렬이
쉬운 질의를 떨어뜨리는 실측에서 나온 결정이고, abstention의 AND 조건은 과회피 실측에서 나온
결정입니다. 어떤 파일의 어떤 함수가 왜 그렇게 생겼는지, 실패한 실험까지 포함해 설명할 수
있습니다. 원하시는 모듈을 지금 짚어주시면 설명하겠습니다.

### H-2. 규칙 기반이 대부분인데, 이게 AI 프로젝트인가?
**답변:** 규칙이 많은 것은 **의도된 경계 설계**입니다. 규제 도메인에서 기한 계산·요건 판정을
LLM에 맡기면 감사를 통과할 수 없습니다. AI의 역할은 자연어 이해·도구 선택·결과 종합
(Agentic Workflow)이고, RAG 최적화·환각 억제·검증 게이트가 이 프로젝트의 AI 엔지니어링
깊이입니다. "무엇을 LLM에 맡기고 무엇을 맡기지 않을 것인가"를 판단하는 능력이야말로
엔터프라이즈 AI 배치의 핵심 역량이라고 생각합니다.

### H-3. 실제 규정과 수치가 다르면 어떡하나? 위험하지 않나?
**답변:** 데모 문서는 샘플이고 README에 명시했습니다 — 목적은 규제 자문이 아니라 아키텍처
증명입니다. 실배치라면 (1) 문서 소스를 현업 검증된 SOP·고시로 교체하고, (2) 버전 관리
파이프라인을 문서 관리 시스템과 연동하고, (3) 지금도 있는 "최종 확정은 담당자" 원칙과
출처 대조 UI를 유지합니다. 시스템이 담당자를 대체하는 게 아니라 담당자의 검토를 빠르게
하는 구조라서, 틀린 수치가 무검토로 나가는 경로 자체가 없습니다.

### H-4. 혼자 만든 프로젝트인데, 팀에서는 어떻게 일할 건가?
**답변:** 이 프로젝트의 작업 방식 자체가 팀 협업 방식입니다 — 모든 주장이 재현 스크립트와
테스트로 뒷받침되고, 하이퍼파라미터 결정 근거가 sweep으로 남아 있고, 위험한 노브에는 회귀
가드가 박혀 있어 다음 사람이 안전하게 수정할 수 있습니다. 문턱값 보정 근거를 코드 주석에
남긴 것도 미래의 동료(와 미래의 나)를 위한 것입니다. FDE는 고객사 엔지니어·현업과 함께
일하는 직무라서, "내 머릿속이 아니라 재현 가능한 아티팩트에 지식을 남기는" 습관이 가장
중요하다고 생각합니다.

## I. 배치 현실 (비용 · 보안 환경 · 운영 · 실경력 연결)

> FDE 면접에서 출제 확률이 높은데 이 데모의 기술 서사만으로는 답이 비던 축 —
> v7 점검(아키텍처.md 7-5, 발견 J)에서 신설했다.

### I-1. 이걸 사내에 배포하면 한 달에 얼마 드나?
**답변:** 감이 아니라 계측으로 답하는 구조를 먼저 말씀드리고 싶습니다. 이 데모의 관측성
계층은 LLM 스텝·도구 호출 단위로 트레이스를 남기므로, 파일럿 기간의 실사용 로그에서
"질문 1건당 평균 LLM 스텝 수 × 스텝당 토큰"을 실측해 월 질문량을 곱하면 됩니다. 구조적
절감 지점도 설계에 있습니다 — (1) 컴플라이언스 계산·트리아지는 규칙 도구라 LLM 토큰을
쓰지 않고, (2) 시스템 프롬프트·도구 스키마는 매 요청 동일하므로 프롬프트 캐싱 대상이며,
(3) 검색이 근거를 3건으로 압축해 컨텍스트가 작습니다. 그리고 오프라인 모드가 있어 "LLM
없이 어디까지 되는가"의 원가 하한이 0원입니다 — 파일럿에서 질문 유형별로 LLM 모드가
실제로 값을 더하는 비중을 재서, LLM 호출을 그 유형에만 라우팅하는 것이 비용 최적화의
첫 수순이라고 생각합니다.

### I-2. 제약사는 망분리가 기본인데, 외부 LLM API를 못 쓰면 이 시스템은 무의미하지 않나?
**답변:** 이 데모에서 외부로 나가는 지점은 정확히 두 곳뿐입니다 — Anthropic API(LLM 모드)와
Voyage API(voyage 임베더 선택 시). 나머지(검색·트리아지·코딩·검증 게이트·preflight)는 전부
로컬 결정론 코드라 망분리 환경에서도 그대로 동작하고, 오프라인 모드는 그 환경을 위한
1차 답입니다. LLM 이 필요한 수준의 배치라면 선택지는 세 층입니다 — 사내 승인 프록시/LLM
게이트웨이 경유(반출 로그·필터 부착), 프라이빗 클라우드 리전의 관리형 모델, 온프레미스
오픈웨이트 모델. 이 데모의 에이전트가 모델 호출부와 도구를 MCP 로 분리해 둔 것이 그때
효력을 냅니다 — 모델을 바꿔도 도구·검증 계층이 그대로라서요. 그리고 어느 층이든 PII
마스킹이 반출 '이전'에 걸린다는 경계 설계는 동일하게 유지됩니다. 실제로 현 직장
(KB헬스케어)에서 ISMS 인증의 프론트엔드 보안 요건 대응과 개인정보 마스킹 적용을 실무로
수행해서, 보안 심사가 "어떤 데이터가 어디로 나가는가"의 인벤토리를 요구한다는 것을
경험으로 알고 있습니다 — 이 데모의 반출 지점 인벤토리가 그 감각의 산물입니다.

### I-3. 동시 사용자 100명이 쓰면? 인증·권한은?
**답변:** 현재 데모는 단일 프로세스·무인증입니다 — 의도된 범위 경계고, 확장 지점은
잡혀 있습니다. 동시성: 인덱스·파이프라인이 부팅 시 1회 구축되는 읽기 전용 구조라 질의
경로는 상태 공유 문제가 없고, uvicorn 워커 수평 확장이 1차 답입니다(단 인메모리 계기판은
워커별로 흩어지므로 집계 저장소 분리가 선행 — 이것도 계기판 산식을 문서화해 둔 이유입니다).
인증·권한: FastAPI 미들웨어 자리에 사내 SSO 를 붙이고, 규제문서에 권한 등급이 있다면
검색 필터(지금의 버전 필터와 같은 자리)에 문서 단위 ACL 을 거는 설계입니다. KB헬스케어에서
어드민 권한 관리 체계 정비를, 에임메드에서 NextAuth 기반 인증 구현을 해봐서, 이 자리가
"나중에 붙이는 장식"이 아니라 초기에 스키마를 잡아야 하는 층이라는 것을 압니다.

### I-4. 배치 후 새벽에 LLM API가 죽으면 무슨 일이 일어나나? 런북은?
**답변:** 이 데모의 동작부터: LLM 호출 실패는 500 으로 죽지 않고 "LLM API 호출 실패 —
키·네트워크·모델명 확인" 명시 안내로 답하며(grounded=False), 오프라인 모드로 조용히
강등하지 않습니다 — 사용자가 어느 모드의 답을 받았는지 모르게 되는 것도 조용한 실패라서요
(아키텍처.md 7-3 에 경계로 명문화). 런북으로 일반화하면: ① 감지 — /health 의 모드·경고율
계기판과 스텝 실패 span 이 1차 신호, ② 판단 — 키 만료/쿼터/네트워크/모델 단종을 실패
타입으로 구분, ③ 조치 — 키 교체 또는 공지 후 오프라인 모드 전환(키 제거 = 의도된 강등
경로, 데모 전 기능 유지), ④ 사후 — 실패 구간의 감사 로그로 영향 응답 범위 확인. 운영
문서에 "누가 새벽에 이 절차를 밟는가"까지 적는 것이 FDE 산출물이라고 생각합니다.

### I-5. 공고에 LangChain/LangGraph가 있는데 왜 안 썼나? 쓰면 뭐가 달라지나?
**답변:** 안 쓴 이유는 두 가지였습니다 — 키 없이도 도는 무의존 데모, 그리고 tool-use
루프·RAG 를 프레임워크 아래에서 직접 구현해 원리를 증명하는 것. 그 대신 무엇이 어디로
가는지는 압니다: 이 데모의 에이전트 루프(모드 분기→도구 선택→검증 게이트)는 LangGraph 로
옮기면 상태 그래프의 노드가 되고, 지금 코드에 흩어져 있는 "최대 6스텝·에러 되먹임" 같은
제어가 그래프 엣지 조건으로 선언적이 됩니다 — 멀티 에이전트나 사람 승인(HITL) 노드가
필요해지는 순간이 도입 시점입니다. RAG 계층은 LangChain 의 retriever 인터페이스와 이
데모의 `EmbeddingProvider`/`HybridRetriever` 가 1:1 로 대응되어 교체가 기계적입니다.
정직하게는, 프레임워크 실전 운영 경험은 이번 입사 후 과제로 남아 있고 — 다만 프레임워크가
감싸는 원리를 바닥부터 구현해 본 상태라 학습 곡선이 짧다는 것이 이 프로젝트의 주장입니다.

### I-6. 실제로 고객·현업과 부딪히며 일한 경험은? (데모 말고)
**답변:** 두 가지가 이 데모의 설계에 직접 흘러들었습니다. 첫째, KB헬스케어에서 외주사가
납품한 건강검진 서비스 4개 사이트를 이관받아 내재화했습니다 — 남의 코드베이스를 분석해
사내 운영 체계로 만드는 일은, FDE 가 고객사 시스템에 들어가 통합하는 일과 방향만 반대인
같은 근육입니다. 그 과정에서 ISMS 인증의 보안 요건 대응과 개인정보 마스킹을 실무로
수행했고 — 이 데모의 PII 경계 설계(입구+도구 계층, 반출 지점 인벤토리)는 그 경험의
연장입니다. 둘째, 리뉴얼 프로젝트에서 기획 초기부터 개발 관점 피드백을 제시하는 협업
방식을 정착시켜 기획 변경에 따른 재개발을 없앴습니다 — 요구사항이 확정되기 전에 기술
제약을 번역해 주는 것이 FDE 커뮤니케이션의 절반이라고 생각합니다.

### I-7. 비개발자 현업이 쓰는 도구를 만들어본 적 있나?
**답변:** 에임메드에서 사내 챗봇 빌더를 전담 개발했습니다 — 비개발자인 현업 담당자가
직접 대화 시나리오를 설계·관리·배포하는 도구로, 사내 3개 건강관리 서비스에 도입됐습니다.
그 경험에서 배운 것이 "현업 도구는 실패 상태를 현업의 언어로 말해야 한다"는 것인데,
이 데모의 설계 문법 — 오타 필터에 '마감 없음' 대신 지원 목록 안내, 보고요건 미충족이면
초안 대신 보완 질문, 검증 경고를 배지로 노출 — 이 정확히 그 문법입니다. 제약사 접점도
있습니다: 글로벌 제약사 GSK 투약 관리 시스템의 초기 개발과 AWS 인프라·배포 파이프라인
구축을 맡아, 제약 도메인 시스템이 요구하는 개발 프로세스의 엄격함을 짧게나마 경험했습니다.

## 부록: 예상 꼬리질문 빠른 답변표

| 질문 | 한 줄 답 |
|---|---|
| Hit@1과 MRR 차이? | Hit@1은 정답이 1위인 비율, MRR은 정답 첫 등장 순위의 역수 평균(2위=0.5) |
| ContextRecall이 왜 중요? | 생성이 정답을 말할 '재료'를 검색이 줬는가 — faithfulness의 상한 |
| BM25가 뭔가? | TF 포화 + 문서 길이 보정 + IDF를 결합한 키워드 랭킹 함수(Okapi BM25) |
| min-max 정규화 왜? | 코사인(0~1)과 BM25(무제한)의 스케일 차이로 한쪽이 지배하는 것 방지 |
| overlap 왜 80자? | 청크 경계에 걸린 문장의 정보 유실 방지(sweep으로 검증된 기본값) |
| WHO-UMC 등급? | Certain/Probable/Possible/Unlikely/Unassessable — 시간관계·dechallenge·rechallenge·대체원인으로 판정 |
| ICH E2D 4요소? | 식별 가능한 환자·보고자·의심 의약품·이상사례 — 하나라도 없으면 유효한 보고가 아님 |
| MedDRA PT/LLT/SOC? | PT=대표 용어, LLT=최하위 용어(구어 변형), SOC=기관계 분류 — 데모의 3계층이 이 구조의 축소판 |
| stdio vs 인메모리? | stdio는 외부 MCP 클라이언트 연결용 표준 트랜스포트, 인메모리는 같은 프로세스 직결(데모 기본) |
| Wilson 신뢰구간 왜? | 소표본에서 1.000을 '완벽'으로 과장하지 않기 위해 — [0.893, 1.000]으로 읽는다 |
| preflight가 막는 건? | 부팅은 통과하지만 운영 중 오답으로 나타나는 데이터·설정 결함(폐지 체인 단절, 모순 설정 등) — 기동 전 exit 1 차단 |
| 날짜 역할 스왑이 왜 위험? | 인지일↔마감일이 뒤바뀌어도 두 날짜 모두 근거에 실존해 존재 대조를 '정의상' 통과 — 역할 라벨 대조 축이 잡는다 |
| 경고율 계기판 왜? | 오탐 증가 = alert fatigue = 검증 계층의 사망 — 죽는 순간을 보이게 만드는 조기 신호(`/health`의 warn_rate) |
| PII 마스킹이 왜 2겹? | 에이전트 입구는 메시지만 커버 — stdio 단독 사용 시 도구 인자(보고자·환자 정보)가 입구를 우회하므로 도구 계층에서도 마스킹 |
| 조용한 실패의 예? | 잘못된 인지일의 무언 폴백, 오타 필터의 "마감 없음" 오답, as_of 형식 오류의 무언 무시, 라우터의 체크리스트 기본 추측, 지연 이중 계산, 시점 조회 0건(필터 조합) — 전부 답이 그럴듯해 신고되지 않는 형태라 caveat·에러 계약·폴백 안내·wall-clock·구간 의미론으로 시끄럽게 바꿨다 |
| 지연은 왜 wall-clock? | 스텝 span은 최상위 span 안에서 실행 — 합산하면 같은 시간이 두 번 세어져 계기판이 ~2배를 보고(틀린 계기판은 없는 것보다 나쁘다) |
| as_of 가 왜 폐지본을 반환? | 시점 조회의 계약은 '그 시점의 현행' — 폐지본이라도 [시행일, 후속본 시행일) 구간이면 당시 현행이다. 폐지 필터를 그대로 겹치면 개정된 규정의 시점 조회가 0건이 된다(필터 교집합 = 공집합) |
| 에러 응답은 왜 신뢰 소스에서 빼나? | 에러 문구에 사용자 입력이 에코된다("as_of '2024/06/01' 가…") — 넣으면 잘못된 입력 값이 검증의 '근거'로 승격된다. 에러 계약은 에이전트 되먹임용이지 근거가 아니다 |
| 한국어 날짜는 어떻게 검증? | "2026년 7월 25일"을 ISO 로 표기 정규화(환산 아님 — 보름=15일과 같은 철학)해 대조. 정규화 전엔 양쪽에서 똑같이 안 잡혀 '통과처럼 보이는' 사각지대 + 날짜의 '25일'이 기간으로 오추출되는 오염 경로가 있었다 |
| 이력 감지 어휘를 왜 좁게? | 이력 스위치는 폐지본 인용 '경고를 끄는' 스위치를 겸한다 — 안전장치를 끄는 스위치의 오탐은 일반 라우팅 오탐보다 비싸므로 명시적 구문("예전"·"구판"·"개정 이력")만 넣는다 |
| 검증 루프 3겹이란? | ① 재현 실행(테스트+평가+preflight) ② 자기 수정에 대한 적대적 리뷰(수정이 새로 연 구멍 탐색) ③ 문서·코드 일치성 검토 — 반례가 나오면 회귀 테스트로 고정 후 재루프 |
| '핀'은 무엇이 강제? | eval 스크립트는 측정만 하고 exit 0 — test_verify_eval.py 가 탐지율·오탐률·**표본 수 하한**을 CI 실패로 강제한다(하한이 없으면 치환 하네스가 조용히 수축해도 1.0 유지) |
| preflight 게이트 자가 테스트 범위? | 런타임 게이트의 **모든 경고 축**(존재·방향·역할·버전·전제 라벨)에 심은 오류 + 정상 케이스 — 축 목록과 자가 테스트 커버리지가 어긋나면 테스트가 실패한다(새 축 추가 시 자가 테스트 강제) |
| 날짜에도 방향 축이? | "2026-07-25까지"→"이후" 뒤집기는 날짜가 근거에 실존해 존재 대조를 통과 — 기간 표기에만 방향 축이 있으면 같은 왜곡이 표기에 따라 한쪽만 잡힌다(클레임 타입×검증 축 매트릭스의 빈 칸) |
| allow_superseded 가 왜 집합? | 전역 bool 은 이력 턴 동안 버전 축을 통째로 끈다 — '이력 검색이 실제 반환한 문서'까지만 면제가 성립하므로 doc_id 집합으로 면적을 좁혔다 |
| case_origin 라벨이란? | 지지 근거가 사용자 케이스 서술뿐인 클레임의 라벨 — 케이스 재서술은 정당(경고 금지)하지만 케이스 수치가 규정 클레임을 지지하는 승격은 가시화해야 한다(차단도 침묵도 아닌 라벨) |
| warn_rate_checked 왜? | warn_rate 는 회피·무클레임 응답이 분모에 섞여 트래픽 믹스 변화가 품질 변화처럼 보인다 — 클레임 있던 응답만 분모로 잡은 경고율을 병기(분모 혼합 착시 제거) |
| LLM 루프는 어떻게 테스트? | anthropic 클라이언트만 스텁, MCP·RAG·게이트는 실물 — 검증 대상은 모델의 지능이 아니라 **배선**(신뢰 소스 수집·허용 집합·에러 계약)이다 |
| 표기 변형 vs 환산의 기준? | "이 대조를 통과시키면 근거에 없는 **새 값**이 생기는가" — 생기면 환산(금지: 15일→약 2주), 안 생기면 표기 변형(정규화: 보름=15일, "7월 25일"=근거 날짜의 월-일, "2025년"=근거 날짜의 연도) |
| 방향 판정은 왜 strict 계층? | 케이스 에코를 합산하면 케이스의 "15일 이후 증상"이 규정 "15일 이내"의 방향 경고를 조용히 무력화 — 판정은 strict, 케이스 일치는 from_case 라벨로 모호성만 가시화(2계층의 축 간 일관성) |
| 오탐 제거가 왜 검증 강화? | 검증 계층은 미탐으로도 오탐으로도 죽는다 — 오탐(alert fatigue)은 전 축을 한꺼번에 무력화하므로 옳은 답변에 경고가 붙는 경로("2025년"·"7월 25일" 오탐)는 미탐 구멍과 같은 등급의 결함이다 |
| 비약·사각지대의 판정 기준? | 비약 = "전제를 대라"(숨은 전제·성질의 승격·범위 과확장·측정≠강제), 사각지대 = "고장 나면 누가 아는가"(대칭적 비검출·조용한 라벨 사망·오탐·안전장치 자신) — description/아키텍처.md 7장에 반대면(알고 남긴 경계)까지 명문화 |
| 미래 인지일은 왜 caveat? | 형식 검사와 값 타당성 검사는 다른 층 — 형식에만 caveat 를 달면 2027 오타가 기한을 1년 뒤로 조용히 민다. 계산은 입력값대로 하되(자동 정정 금지) 오타 개연성을 명시한다 |
| eval 이 안 재는 축은? | 고유어 방향·케이스 간섭 축 — 코퍼스 발췌 변조로는 표본이 0~1이라 '측정의 조용한 수축'과 구분 불가. preflight 자가 테스트·단위 테스트가 대신 고정하고, 그 경계를 eval 문서에 명시한다 |
| case_labeled 계기판 왜? | 라벨은 ok=True 라 경고율에 안 보인다 — 답변이 규정 근거 대신 사용자 서술에 기대기 시작하는 이동은 라벨률 추이로만 보인다 |
| PII 경계가 왜 룩어라운드? | `\b`는 한글이 `\w`라 "…는 010-…로"·"…입니다" 같은 조사 직결 표기에서 경계가 성립하지 않는다 — 검증기 날짜 추출의 교훈이 마스킹에 전파되지 않았던 v6 사각지대(비검출=유출) |
| preflight 우회 경로? | 게이트가 자동으로 걸리는 것은 run.sh·CI 뿐 — 수동 uvicorn 실행은 게이트 밖이다. '차단한다' 주장의 성립 범위를 문서에 명시(숨은 전제의 문서화) |
