# 사용자 가이드 — RAPV-Assistant 실제 사용법

> 이 문서는 RAPV-Assistant를 **실제로 사용할 사람**(RA·PV 담당자, 데모 시연자)을 위한 가이드다.
> 설치·실행부터 질문 요령, 화면 읽는 법, 결과를 믿어도 되는 범위까지 설명한다.
> 읽다가 낯선 용어(ICSR·KAERS·PSUR·인지일·WHO-UMC 등)가 나오면 같은 폴더의
> `dictionary.md`(용어 사전 — 기초 개념부터 순서대로 풀이)를 곁에 두고 볼 것.

---

## 1. 시작하기 전에 알아둘 것

- **API 키가 없어도 모든 기능이 동작한다.** 키가 없으면 "오프라인 모드"(규칙 라우팅 + 근거 발췌 답변),
  키가 있으면 "LLM 모드"(Claude가 도구를 조합해 자연어로 답변)로 자동 전환된다.
- 이 데모의 규제 수치(처리기한 등)는 **샘플**이며 실제 최신 법령과 다를 수 있다.
  실무 규제 판단의 근거로 사용하면 안 된다.
- 모든 판정 결과에는 "최종 확정은 담당자"라는 전제가 깔려 있다. 이 시스템은
  **자동화가 아니라 사람의 판단을 빠르게 하는 보조 도구**다.

### 요구 사항
- Python 3.11 이상 (`requirements.txt` 기준 — bash 셸 전제. 윈도우는 WSL 또는
  `.venv\Scripts\` 경로로 방법 B를 수동 실행)
- (선택) Anthropic API 키 — LLM 모드용

## 2. 설치와 실행

### 방법 A — 원커맨드 (권장)
```bash
cd project
./run.sh
# venv 생성 + 의존성 설치 + 배포 전 점검(preflight) + 서버 실행까지 한 번에
# → 브라우저에서 http://127.0.0.1:8000 접속
```

> `run.sh`는 서버를 띄우기 전에 **배포 전 점검(preflight)**을 먼저 통과해야 한다 —
> 데이터(규제문서 메타·업무 데이터)와 설정에 결함이 있으면 서버가 뜨지 않고 문제 목록이
> 출력된다. 이것은 오류가 아니라 의도된 게이트다(9장 FAQ 참고).

### 방법 B — 수동
```bash
cd project
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
.venv/bin/python -m src.preflight        # 권장: 수동 실행은 게이트를 지나지 않으므로 직접 돌린다
.venv/bin/python -m uvicorn src.api.main:app --port 8000
```

> **주의**: preflight 게이트가 자동으로 걸리는 경로는 `run.sh`와 CI 두 가지뿐이다.
> 방법 B처럼 uvicorn 을 직접 띄우면(다른 포트로 띄우는 경우 포함) 게이트를 지나지
> 않으므로, 데이터·설정을 바꿨다면 위처럼 preflight 를 별도로 실행하는 것을 권장한다
> — "preflight 가 결함 배포를 차단한다"는 말은 '기동이 run.sh·CI 를 지난다'는
> 전제 위에서만 참이다.

### LLM 모드 켜기 (선택)
```bash
export ANTHROPIC_API_KEY=sk-ant-...
./run.sh
```
또는 `project/.env` 파일에 키를 넣어두면 `run.sh`가 자동으로 로드한다(`.env.example` 참고).

### 정상 기동 확인
```bash
curl http://127.0.0.1:8000/health
# → 모드(llm/offline), 인덱스된 문서/청크 수, 하이퍼파라미터,
#    verification_gate(검증 게이트 경고율 계기판)가 보이면 정상
```

`verification_gate`는 이 프로세스가 처리한 응답 중 검증 경고가 붙은 비율(`warn_rate`)과
축별 건수를 보여준다. **소재와 수명에 주의**: 이 계기판과 응답 단위 감사 로그(판정 요약
JSON)는 데모에서 **인메모리 집계 + 콘솔(구조화 JSON) 로그**다 — 프로세스를 재시작하면
계기판은 0으로 돌아가고, 감사 로그를 남기려면 콘솔 출력을 파일로 리다이렉트해야 한다
(영속 저장소 연동은 실무 교체 지점). **단일 워커 전제**도 같은 계열이다: 인메모리 집계는
프로세스마다 독립이므로 `uvicorn --workers N`(N>1)으로 스케일하면 `/health`가 요청을 받은
한 워커의 수치만 반환해 warn_rate가 워커마다 갈린다 — 다중 워커 운영은 위 영속 저장소
교체(집계 공유)가 선행돼야 한다. run.sh·방법 B는 단일 워커라 이 전제가 성립한다.
**운영 중 이 비율이 올라가면** 답변 품질이 나빠졌거나 검증기가
오탐을 내기 시작했다는 조기 신호이므로 원인을 확인해야 한다(경고가 무시되기 시작하면
검증 체계 전체가 무력화된다). 추이는 `warn_rate_checked`(수치·날짜 클레임이 있던
응답만의 경고율)를 기준으로 읽는 것이 안전하다 — 전체 `warn_rate`는 회피·무클레임
응답이 분모에 섞여 있어, 범위 밖 질문 비중이 변하면 품질 변화 없이도 오르내린다.

## 3. 무엇을 물어볼 수 있나 — 질문 유형별 사용법

### 3-1. 규정 검색 (RAG)
규제문서의 내용을 근거·출처와 함께 찾아준다.

> - "신약 품목허가 심사는 며칠 걸려?"
> - "중대한 이상사례는 며칠 안에 보고해야 해?"
> - "GMP 데이터 완전성(ALCOA) 원칙이 뭐야?"
> - "부작용이 심각하게 나타나면 어떻게 해?" ← 구어로 물어도 동의어 확장이 정식 용어("중대한 이상사례")로 연결해준다

답변에는 근거 문서·섹션·버전이 붙는다. 문서에 근거가 없는 질문(예: "감기약 추천해줘")에는
지어내지 않고 "근거를 찾지 못했습니다"라고 답한다.

> **모드별 경계**: 위 회피(abstention)는 **문턱값 판정이 있는 오프라인 모드의 동작**이다.
> LLM 모드에는 문턱 기반 회피가 없다 — 시스템 프롬프트의 근거 강제, 답변 사후 검증
> 게이트, 그리고 `grounded` 배지(도구 근거 확보 여부)가 그 역할을 나눠 맡는다. 즉 LLM
> 모드에서 출처 없는 답이 나올 수는 있으며, 그 경우 `grounded=False` 로 표시된다
> (4장 표 참고 — "출처가 없는 답은 절대 안 나간다"는 모드 무관 보장이 아니다).

### 3-2. 이상사례(AE) 케이스 트리아지 (PV)
환자 케이스를 서술하면 중대성 판정 → 보고기한 계산 → 인과성(WHO-UMC) 제안 → 용어 코딩까지 해준다.

> - "환자가 A정 복용 후 아나필락시스로 입원했어요. 언제까지 보고해야 하나요?"
> - "환자가 B캡슐 투여 후 두드러기가 났고, 중단하니 호전됐습니다. 인과성은?"

**입력 요령**:
- 인지일을 알면 함께 적는다 — 보고 마감 '날짜'까지 계산된다. 형식은 `YYYY-MM-DD`
  (예: 2026-07-01). 형식이 잘못되면 시스템이 조용히 무시하지 않고 판정 결과에
  **"오늘 기준으로 계산했다"는 경고(caveat)**를 명시한다 — 이 경고가 보이면 인지일을
  바로잡아 다시 물어볼 것(기한이 잘못된 기준일로 계산된 상태다).
- 시간 관계("복용 후"), 중단 후 경과("중단하니 호전"), 재투여 반응, 병용약·기저질환을
  서술에 포함할수록 인과성 제안의 등급 근거가 풍부해진다. 없는 정보는 시스템이
  **follow-up 질문**으로 되물어준다.
- 환자 개인정보(이름·주민번호·연락처)가 섞여 있어도 된다 — **입구에서 자동 마스킹**되어
  외부 API·로그 어디에도 남지 않는다. 화면에는 마스킹된 유형·건수만 표시된다.
  (대화 '이력' 속 개인정보도 LLM 으로 나가기 전에 마스킹되지만, 배지 건수에는
  현재 메시지의 마스킹만 집계된다 — 이력 마스킹은 배지에 표시되지 않는다.)
  보고서 초안의 보고자·환자 정보 필드, 검색 질의, API 로 직접 보내는 대화 이력
  (`history` — 문자열·블록 리스트 표기 모두)에 개인정보가 들어가도 마찬가지로 마스킹된다
  (케이스 서술만이 아니라 모든 자유 입력 필드가 대상). 표기도 가리지 않는다 —
  "연락처는 010-…-5678로", "주민번호는 …입니다"처럼 조사가 바로 붙은 자연스러운
  한국어 문장 속 번호도 잡는다(띄어쓰기·구분자 표기에 의존하지 않음).

### 3-3. ICSR/KAERS 보고서 초안
> - "이 케이스 KAERS 보고서 초안 만들어줘" (케이스 서술과 함께)

시스템이 먼저 **최소보고요건(ICH E2D 4요소: 환자·보고자·의심약·이상사례)**을 검증한다.
- 4요소가 갖춰졌으면 → 트리아지+인과성+코딩을 종합한 마크다운 초안이 나온다.
- 하나라도 빠졌으면 → **"⛔ 정보 보완 필요(최소보고요건 미충족)" 상태 헤더가 문서 최상단에
  붙은 초안**과 함께 빠진 항목·보완 질문이 병기되어 나온다. 이것은 오류가 아니라 의도된
  동작이다 — 채워진 항목까지는 초안으로 보여주되, 미완성이 완성으로 포장되지 않도록
  상태를 시끄럽게 표시한다. 보완 질문에 답을 채워 다시 요청하면 ✅ 상태의 초안이 된다.

### 3-4. 마감일·체크리스트 (RA·PV 업무 데이터)
> - "이번 주 마감 임박한 규제 업무는?"
> - "변경허가 준비 체크리스트 줘"
> - "품목허가 제출 전에 뭘 확인해야 해?"

체크리스트는 세 유형(품목허가·변경허가·안전성보고)만 준비되어 있다. 그 밖의 유형
(예: "GMP 체크리스트")을 요청하면 **엉뚱한 체크리스트를 추측해서 주지 않고**, 지원
목록을 안내한 뒤 관련 규정 검색 결과로 대신 답한다.

> 오프라인 챗 모드의 마감일 조회는 **오늘부터 30일 이내** 고정 창으로 조회한다("다음
> 분기 마감은?"도 30일 창으로 답한다 — 다른 창이 필요하면 API/MCP 로
> `get_ra_deadlines(within_days=…)` 를 직접 호출). 지난 마감(연체)은 창과 무관하게
> 항상 포함되어 🔴 로 표시된다.

### 3-5. 복합 질문 (LLM 모드에서 진가)
> - "GMP 변경인데 뭘 준비해야 하고 언제까지야?"

LLM 모드에서는 에이전트가 검색+체크리스트+마감일 **여러 도구를 스스로 조합**해 답한다.
하단 도구 트레이스에서 어떤 도구가 어떤 순서로 호출됐는지 확인할 수 있다.

### 3-6. 개정 이력·과거 시점 조회 (버전 인지 검색)
기본 검색은 **폐지된 구판을 자동 제외**하고 현행 규정만 답한다.
> - "이상사례 보고 기한, 예전 규정에서는 어땠어? 개정 이력 포함해서 알려줘"

이력을 명시적으로 요청하면 구판(superseded)도 함께 노출되고, 출처에 버전·시행일이
표시된다. 두 모드 모두 동작한다 — 오프라인 모드는 "예전", "구판", "개정 이력" 같은
**명시적 이력 구문**을 감지해 구판을 포함한다(단순히 "이력"이라는 단어가 들어간 현행
질문, 예: "접수·처리 이력 관리 요건"은 이력 조회로 처리하지 않는다 — 이때는 현행만 답한다).

**과거 시점 조회(as_of)** 도 지원한다: 특정 날짜에 시행 중이던 버전을 찾는 기능으로,
그 시점의 현행이었다면 지금은 폐지된 구판이라도 반환된다(출처의 status 로 구분 가능).
LLM 모드에서는 "2025년 1월 시점 기준으로…"처럼 물으면 에이전트가 `as_of` 를 스스로
지정하고, API·MCP 로 직접 쓸 때는 `search_regulations(query, as_of="2025-01-01")` 로
호출한다. 기준일 형식은 `YYYY-MM-DD` — 형식이 틀리면 조용히 무시하지 않고 명시적
에러(`error`+`expected`)로 답한다(오프라인 챗 모드에는 시점 조회 경로가 없다 —
날짜 파싱 없이 문장에서 시점 의도를 안전하게 확정할 수 없어서다).

### 3-7. 후속 질문 (멀티턴)
> - (1턴) "변경허가 처리기한은?" → (2턴) "그럼 변경신고는?"

짧은 후속 질문은 직전 대화와 병합되어 해석된다. 병합 조건은 두 가지 중 하나다 —
질문이 **약 12자 이내로 짧거나**, "그럼"·"그건"·"방금"·"아까" 류 **지시어로 이어질 때**.
그 외의 후속 질문(길고 지시어도 없는 경우)은 병합되지 않으므로, 맥락을 질문에 직접
포함해 묻는 것이 안전하다(예: "그럼 변경신고는?" 대신 "변경신고 처리기한은?").
웹 UI 는 서버로 보내는 이력을 **최근 3왕복(6메시지)** 으로 절단한다 — 그보다 오래된
맥락을 이어가려면 새 질문에 맥락을 직접 포함해 물을 것(API 직접 호출 시에는 `history`
길이 제한이 없다).

## 4. 화면(응답) 읽는 법

답변 아래에 붙는 정보들의 의미:

| 표시 | 의미 | 어떻게 활용하나 |
|---|---|---|
| **출처(citations)** | 답의 근거 문서·섹션·버전·시행일 | 실무 확정 전 반드시 원문과 대조. 출처의 버전이 현행인지 확인 |
| **호출한 MCP 도구** | 에이전트가 사용한 도구와 인자 | 답이 검색에서 왔는지, 계산 도구에서 왔는지 구분 |
| **지연(ms) / 트레이스** | 스텝별 소요 시간·성패 | 데모 시연 시 "어디서 시간이 쓰였나" 설명용 |
| **PII 마스킹 배지** | 마스킹된 개인정보 유형·건수 | 원문 값은 시스템 어디에도 저장되지 않았다는 표시 |
| **근거 상태(grounded)** | 답이 도구 근거(검색·계산 결과)로 뒷받침되는지 | 오프라인 모드: False = 회피 응답. LLM 모드: False = 모델이 도구를 쓰지 않고 답했거나 실패 안내(문턱 판정이 아니라 '도구 근거 확보 여부'의 신호) — False 인 답은 근거 보증이 없으므로 실무 참고 금지 |
| **검증 배지(verification)** | 답변 속 수치·날짜·방향 한정어가 근거와 일치하는지 | ⚠️ 경고가 붙으면 해당 수치를 원문과 직접 대조할 것 |

### 검증 경고가 떴을 때
검증 게이트는 답변을 차단하지 않고 경고만 붙인다. 경고의 종류:
- **미확인 수치·날짜**: 답변의 숫자·날짜가 근거·도구 출력 어디에도 없음 → 원문 확인 필요.
  날짜는 표기 무관하게 대조된다 — "2026년 7월 25일"·"2026-07-25"는 물론, 연도 없는
  "7월 25일"(근거 날짜의 월-일과 대조), "2025년"(근거 날짜의 연도와 대조), 고유어
  수사("보름"=15일)까지 같은 클레임으로 취급하므로, **표기만 다른 옳은 답변에는 경고가
  붙지 않는다**(오탐이 잦으면 경고가 무시되기 시작하므로 오탐 제거도 검증 강화다).
- **question_origin**: 그 숫자가 사용자의 질문(직전 턴 포함)에서 온 것 → 환각이 아니라
  "질문의 전제를 확인하라"는 뜻(예: "보고 기한이 30일 맞지?"라고 물었을 때). 경고가
  사라지는 게 아니라 종류가 바뀌는 것이다 — 값 확인이 필요하다는 사실은 동일하다.
- **방향 한정어 불일치**: 수치는 맞는데 "이내"가 "이후"로 뒤집힘 → 가장 교묘한 왜곡이므로
  반드시 확인. 날짜 표기("2026-07-25**까지**" → "**이후**")와 고유어 표기("**보름 이후**")
  에도 같은 검증이 적용된다. 경고문에 "케이스 서술에는 같은 방향 표현이 있어 재서술일 수
  있으나…"가 붙어 있으면, 입력한 케이스의 문장을 답변이 옮긴 것인지(정상) 규정의 방향을
  뒤집은 것인지(오류) 사람이 가려야 한다는 뜻이다 — 시스템은 이 둘을 구분할 수 없어
  경고를 끄는 대신 모호성을 알려준다.
- **케이스 서술 유래(📝 배지)**: 경고가 아니라 **라벨**이다 — 그 수치의 지지 근거가 규정·
  도구 결과가 아니라 **당신이 입력한 케이스 서술**뿐이라는 뜻. 케이스를 그대로 재서술한
  것이면 정상이지만, 그 수치가 기한·요건 같은 **규정 내용**으로 제시되어 있다면 규정
  원문과 대조할 것(케이스의 "30일간 복용"이 "보고 기한 30일"의 근거가 될 수는 없다).
- **날짜 역할 불일치**: 답변의 날짜들이 근거에는 있는데 **역할이 뒤바뀜** — 예: 인지일이
  "보고 기한"으로 제시됨. 두 날짜 모두 실존해서 단순 대조는 통과하는 형태이므로,
  경고가 뜨면 어느 날짜가 마감일인지 도구 결과와 직접 대조할 것.
- **폐지본 인용**: 이력 조회를 요청하지 않았는데 폐지된 규정이 출처에 섞임.

## 6. MCP 서버 단독 사용 (Claude Desktop / Cursor 연결)

이 데모의 도구들은 표준 MCP 서버라서 다른 MCP 클라이언트에서도 그대로 쓸 수 있다.

```bash
# stdio 모드 단독 실행
.venv/bin/python -m src.mcp_server.server
```

Claude Desktop 설정 예 (`claude_desktop_config.json`):
```json
{
  "mcpServers": {
    "RAPV-Assistant": {
      "command": "/절대경로/project/.venv/bin/python",
      "args": ["-m", "src.mcp_server.server"],
      "env": {"PYTHONPATH": "/절대경로/project"}
    }
  }
}
```

> `env.PYTHONPATH` 는 생략하면 안 된다 — `-m src.mcp_server.server` 는 '작업 디렉터리가
> `project/`'라는 숨은 전제 위에서만 성립하는데, Claude Desktop 같은 클라이언트는 서버
> 프로세스의 cwd 를 보장하지 않으므로 그 전제가 깨지면 `ModuleNotFoundError: src` 로
> 죽는다(실측 확인). `PYTHONPATH` 가 그 전제 자체를 제거한다.

연결되면 Claude Desktop에서 `search_regulations` 등 도구 6종, `regulation://{doc_id}` 리소스,
`pv_case_intake` 프롬프트(케이스 처리 SOP)를 사용할 수 있다.

**Resource·Prompt 사용법**:
- 리소스의 `doc_id` 는 먼저 `list_regulation_documents` 도구로 목록을 받아 고른다
  (예: `regulation://REG-005` → PV 규정 원문 전체). 존재하지 않는 doc_id 에는
  "문서를 찾을 수 없음" 평문이 반환된다(도구들의 `error`+`available` 계약과 달리
  가용 목록은 붙지 않으므로, 목록 도구를 먼저 쓰는 것이 정석).
- 프롬프트는 Claude Desktop 의 프롬프트 선택 메뉴(➕ → RAPV-Assistant → pv_case_intake)에서
  케이스 서술을 인자로 넣어 호출한다 — 접수→트리아지→근거 확인→초안 순서의 SOP 가
  대화에 삽입된다(케이스 서술 속 개인정보는 삽입 전에 마스킹된다).

## 7. 설정 바꾸기 (환경변수)

| 변수 | 기본값 | 의미 |
|---|---|---|
| `ANTHROPIC_API_KEY` | (없음) | 있으면 LLM 모드, 없으면 오프라인 모드 |
| `LLM_MODEL` | claude-opus-4-8 | LLM 모드에서 쓸 모델 |
| `EMBEDDER_KIND` | tfidf | `tfidf` / `hashing` / `voyage`(VOYAGE_API_KEY 필요) |
| `QUERY_EXPANSION` | 1 | 질의확장 on/off |
| `RETRIEVE_TOP_K` / `RERANK_TOP_N` | 8 / 3 | 1차 회수 / 최종 근거 개수 |
| `HYBRID_ALPHA` | 0.5 | 벡터 vs BM25 가중 |
| `CHUNK_SIZE` / `CHUNK_OVERLAP` | 500 / 80 | 청킹 파라미터 |
| `RERANK_WEIGHT` | 0.9 | 리랭커 재점수 비중(0.9 = 1차 점수를 10% 섞어 안정화) |
| `RERANK_IDF_POWER` | 0.5 | 리랭커 토큰 가중 idf^p (0=균등) |
| `RERANK_CONTRAST_PENALTY` | 0.3 | 대조 섹션("~와의 차이") 페널티 |
| `RERANK_PREAMBLE_PENALTY` | 0.055 | 서두 섹션("목적/개요") 감쇠 |

> 기본값들은 감이 아니라 `eval/sweep.py` 스윕·ablation으로 정해진 값이다. 바꾸면
> `python -m eval.evaluate`로 검색 품질이 어떻게 변하는지 직접 확인할 수 있다.
>
> `EMBEDDER_KIND=voyage` 주의: `VOYAGE_API_KEY` 가 없으면 preflight 가 기동 전에
> 잡아주지만, 게이트를 지나지 않는 수동 uvicorn 실행에서는 부팅 인덱싱 중 에러로
> 죽는다. 또 이 모드는 부팅 시 청크 수만큼 + 질의마다 외부 임베딩 API 를 호출한다
> (지연·과금·질의 텍스트의 외부 송신 — 질의는 마스킹을 거친 뒤 송신된다).

## 9. 자주 겪는 상황 (FAQ / 트러블슈팅)

**Q. "근거를 찾지 못했습니다"만 나온다.**
정상 동작일 가능성이 높다 — 코퍼스(규제문서 13종) 범위 밖의 질문에는 지어내는 대신 회피한다.
범위 안 질문인데 회피된다면 정식 용어로 바꿔 물어보라(예: "약 부작용" → "이상사례 보고").

**Q. 보고서 초안에 "⛔ 정보 보완 필요" 상태가 붙고 보완 질문이 함께 온다.**
최소보고요건 4요소 중 빠진 것이 있다는 뜻이다 — 초안은 채워진 범위까지 생성되지만
상태 헤더가 미충족임을 최상단에 표시하고(`reportable=False`), 빠진 항목과 보완 질문이
병기된다. 그 질문(환자 정보·보고자·의심약·증상)에 답을 채워 다시 요청하면 ✅ 충족
상태의 초안이 생성된다. 이 상태로 제출하면 안 된다.

**Q. 답변이 딱딱한 발췌문이다.**
오프라인 모드다. `ANTHROPIC_API_KEY`를 설정하면 자연어 종합 답변으로 바뀐다.
근거·출처(citations) 체계는 두 모드가 동일하다(단 회피 판정 방식은 다르다 — 3-1 참고).

**Q. 키를 넣었는데 "(LLM API 호출에 실패했습니다: …)"라는 답이 온다.**
잘못되었거나 만료된 API 키(401), 네트워크 차단, 존재하지 않는 `LLM_MODEL` 값이
대표 원인이다. 시스템은 예외를 500 에러로 전파하는 대신 실패 유형과 함께 명시적으로
안내한다(grounded=False). 키·네트워크·모델명을 확인해 재시도하고, LLM 없이 쓰려면
키를 비우면 오프라인 모드로 동작한다 — **키가 잘못되어도 오프라인 모드로 자동 강등되지는
않는다**(조용한 모드 전환은 사용자가 어느 모드의 답을 받았는지 모르게 만든다).

**Q. LLM 모드 답변이 중간에 끊긴 것 같다.**
LLM 모드 답변 생성은 1024 토큰 상한이 있다. 매우 긴 복합 질문의 답이 끝문장 없이
끊겼다면 이 상한일 가능성이 높다 — 질문을 나눠서 다시 물을 것.

**Q. preflight가 "마감일 N건이 모두 과거"로 실패한다.**
의도된 동작이다. `data/ra_tasks.json` 의 샘플 마감일은 시한부 데이터라, 전부 과거가
되면 "임박한 마감" 데모가 전항목 연체 목록으로 조용히 죽는다 — 그 전에 기동 시점에
시끄럽게 알리는 것이다. 파일의 `due_date` 들을 미래 날짜로 갱신하면 통과한다.

**Q. 예전 규정 기준으로 답하는 것 같다.**
출처의 버전·시행일을 확인하라. 기본 검색은 폐지본을 제외하므로, 출처에 superseded가
보인다면 검증 배지에도 경고가 떠 있을 것이다.

**Q. `run.sh`가 "배포 전 점검 실패"로 멈추고 서버가 안 뜬다.**
의도된 동작이다. preflight가 데이터·설정 결함(규제문서 메타 누락, 폐지 체인 단절,
업무 데이터 날짜 오타, 모순된 하이퍼파라미터 조합 등)을 찾으면 서버를 띄우지 않는다 —
이런 결함은 부팅은 통과하고 **운영 중에 오답으로** 나타나기 때문이다. 출력된 문제 목록
(`[✗]` 항목)을 해결한 뒤 다시 실행하라. `.venv/bin/python -m src.preflight`로 점검만
따로 돌려볼 수도 있다.

**Q. 트리아지 결과에 "오늘 기준으로 기한을 계산했다"는 경고가 붙었다.**
인지일을 `YYYY-MM-DD` 가 아닌 형식(예: "2026/07/01", "7월 1일")으로 적은 경우다.
시스템은 계산을 멈추는 대신 오늘 기준으로 폴백하되 그 사실을 경고로 남긴다 —
표시된 기한은 잘못된 기준일로 계산된 값이므로, 인지일을 바로잡아 다시 요청할 것.

**Q. 트리아지 결과에 "인지일이 미래 날짜"라는 경고가 붙었다.**
형식은 맞지만 오늘 이후의 날짜를 인지일로 적은 경우다(연도·월 오타 개연성이 높다).
기한은 입력한 값 그대로 계산되어 있으므로, 실제 인지일이 맞는지 확인하고 오타면
바로잡아 다시 요청할 것 — 시스템은 값을 임의로 고치지 않는다(자동 정정은 또 다른
조용한 오류 경로다).

**Q. 체크리스트를 요청했는데 "요청과 일치하는 체크리스트 유형을 찾지 못했습니다"가 나온다.**
의도된 동작이다. 지원 목록(품목허가·변경허가·안전성보고)에 없는 유형을 요청하면
엉뚱한 체크리스트를 추측해서 반환하는 대신, 지원 목록을 안내하고 관련 규정 검색으로
대신 답한다. 지원 목록의 정확한 유형명으로 다시 요청하면 체크리스트가 나온다.

**Q. 과거 시점 조회(as_of)에 폐지된 문서가 나온다.**
의도된 동작이다. 시점 조회는 "그 날짜에 시행 중이던 버전"을 찾는 기능이므로, 그 시점의
현행이었다면 지금은 폐지된 구판이 정답이다. 출처의 status(superseded)와 버전·시행일로
현재 기준 폐지 여부를 확인할 수 있다. 반대로 형식이 틀린 기준일(`2025/01/01` 등)은
조용히 무시되지 않고 명시적 에러로 답한다. **빈 검색어(무신호 질의)도 같은 계약**이다 —
임의 문서가 점수 0의 '검색 결과'로 나가는 대신 명시적 에러(`error`+`expected`)로 답한다.

**Q. 마감일을 유형으로 필터했더니 "해당 유형 없음" 에러가 나온다.**
의도된 동작이다. 존재하지 않는 유형(오타 등)에 "임박한 마감이 없습니다"라고 답하면
그것은 **자신 있는 오답**이 된다 — 대신 에러와 함께 사용 가능한 유형 목록(available)을
돌려주므로, 목록에서 정확한 유형을 골라 다시 조회하면 된다(LLM 모드에서는 에이전트가
이 목록을 보고 스스로 재시도한다).

**Q. 포트 8000이 이미 사용 중이다.**
`.venv/bin/python -m uvicorn src.api.main:app --port 8001`처럼 다른 포트로 실행한다.
단, 이 수동 실행 경로는 preflight 게이트를 지나지 않으므로(2장 방법 B의 주의 참고)
데이터·설정을 바꾼 상태라면 `.venv/bin/python -m src.preflight`를 먼저 돌리는 것이 안전하다.

**Q. 환자 이름이 마스킹되지 않았다.**
정규식 기반 마스킹은 호칭이 붙은 이름("김철수님")만 잡는 경량 근사다. 데모에는 실제
개인정보를 입력하지 말 것. 실무 배치 시에는 NER 기반 비식별화로 교체하는 자리다.

## 10. 사용상 한계 (반드시 알아야 할 것)

1. **규제 자문 도구가 아니다** — 문서·수치는 데모용 샘플. 실제 업무 판단은 반드시 현행 법령·SOP로.
2. **판정은 보조** — 중대성·기한·인과성·보고요건 판정은 모두 "담당자의 최종 확정"을 전제로 한다.
3. **코퍼스가 작다** — 규제문서 13종만 인덱싱되어 있다. 그 밖의 질문은 회피된다.
4. **마스킹은 근사** — 자유 서술 속 모든 개인정보를 잡지 못한다. 실데이터 입력 금지.
