## 판단 — "왜 이렇게 설계했는가"

### Q1. 검증에 실패한 답변을 왜 차단하지 않고 경고만 붙였나?

**A.** 검증기 자신도 틀릴 수 있기 때문이다(패러프레이즈를 환각으로 오판하는 등).
오탐이 있는 검증기로 자동 차단하면 옳은 답변이 막히고, 담당자는 시스템을 불신하게 된다.
그래서 이 프로젝트의 원칙은 "사람의 최종 확정을 빠르게"다 — 답변은 내보내되 경고를
본문·API 필드·UI 배지에 **모두** 노출한다(시끄러운 실패). 대신 경고율을 `/health`
계기판으로 상시 추적해서, 경고가 너무 잦아져 담당자가 무시하기 시작하는 순간
(alert fatigue)을 조기에 잡는다. → 코드: `src/agent/agent.py _finalize()`,
`src/observability.py GateStats`

### Q2. PII 마스킹을 왜 에이전트 입구에서 하나? 도구 안에서 또 하는 건 중복 아닌가?

**A.** 입구(`RaAgent.chat()`)에서 하는 이유: 마스킹 이후의 **모든** 경로(외부 LLM API·
검색·로그·트레이스)에 원문 개인정보가 흘러들지 않게 하는 가장 확실한 위치라서다.
도구(`server.py`)에서 또 하는 이유: 도구는 에이전트를 거치지 않고 **단독으로도
호출될 수 있기 때문**이다(Claude Desktop이 stdio로 직접 연결하는 경우). 경로마다
방어를 두는 것을 심층방어(defense in depth)라 하며, "중복"이 아니라 "한 겹이
뚫려도 다음 겹이 막는" 설계다. → 코드: `src/pv/redactor.py`, `agent.py:258` 부근,
`server.py` 각 도구의 `redact()` 호출

### Q3. 답변 검증을 왜 LLM(AI 재검토)이 아니라 정규식 규칙으로 하나?

**A.** 검증자가 확률적이면 "검증의 검증"이 또 필요해지는 순환이 생긴다. 규칙 기반
검증은 같은 답변에 항상 같은 판정을 내려 **감사 가능**하고(규제 도메인 필수),
검증기 자체를 평가셋으로 측정할 수 있다. 대신 규칙이 못 잡는 것(부정문, 수치 없는
의미 왜곡)은 경계를 **명시**하고 LLM 2차 검증의 자리로 남겨뒀다 — "다 잡는다"고
주장하지 않고 잡는 범위를 정확히 선언하는 것 자체가 판단이다.
→ 코드: `src/verify/verifier.py` 모듈 docstring

### Q4. 오프라인 모드(API 키 없이 동작)는 왜 만들었나? LLM 모드만 있으면 되지 않나?

**A.** 세 가지 판단이 겹친다. ① 데모·평가가 외부 API 없이 항상 재현돼야 한다
(키·네트워크·비용에 의존하면 시연이 복불복이 된다). ② 키 문제로 LLM이 죽어도
서비스가 완전히 죽지 않는 폴백이 된다. ③ 발췌형(생성 없는) 답변은 환각이 원천
불가능해서, 검증 게이트가 "포매터가 근거 밖 수치를 만들지 않는다"는 회귀 가드
역할을 겸한다. 두 모드가 같은 MCP 도구·같은 검증 게이트를 지나므로 코드 갈래
유지비도 낮다. → 코드: `config.py LLM_AVAILABLE`, `agent.py _chat_offline()`

## 검증 — "AI가 만든 것이 맞는지 어떻게 아는가"

### Q5. AI가 짠 코드가 "돌아간다"와 "맞다"를 어떻게 구분하나?

**A.** "돌아간다"는 실행해 보면 알지만 "맞다"는 **정답을 아는 입력 집합**(평가셋)으로
재야 한다. 이 프로젝트는 질문-정답문서 쌍 32개로 검색 정확도(Hit@1)를 재고,
범위 밖 질문 세트로 회피(abstention) 정확도를 재고, 검증기 자체도 verify_eval로
잰다. AI에게 구현을 맡길수록 이 측정 장치가 유일한 안전망이다 — "그럴듯하게
돌아가는 틀린 코드"는 AI의 주특기이기 때문이다. → 코드: `eval/` 스크립트 5종,
`tests/` 292케이스

### Q6. 답변에 나온 "처리기한 15일"이 맞는 값인지 시스템은 어떻게 확인하나?

**A.** 답변에서 숫자+단위(15일)·날짜를 정규식으로 추출해, **신뢰 소스**(그 턴에
도구가 실제로 반환한 텍스트: 검색 근거 문단 + 규칙 도구의 계산값) 안에 존재하는지
대조한다. 핵심 설계는 신뢰 소스에서 **사용자 입력의 메아리(에코)를 빼는 것** —
검색 도구는 받은 질의를 출력에 되돌려주는데(query 에코), 이걸 신뢰하면 사용자가
"30일 맞죠?"라고 틀리게 물었을 때 그 30일이 '근거'로 승격되어 오답이 검증을
통과한다. → 코드: `verifier.py verify_answer()`, `agent.py _split_user_facts()`

### Q7. 검색이 "좋아졌다"는 어떻게 측정하고, 회피 문턱값(0.19/0.28)은 어떻게 정했나?

**A.** 좋아짐 = 평가셋 지표의 개선(Hit@1: 정답 문서가 1등으로 나온 비율)이지 체감이
아니다. 문턱값은 감이 아니라 **분포 실측**으로 정했다: 범위 안 질문들의 점수 분포와
범위 밖 질문들의 점수 분포를 그려서, 두 분포가 갈라지는 틈의 중앙에 문턱을 놓았다
(범위밖 최대 0.167 < 문턱 0.19 < 범위내 최소 0.201). 더 중요한 건 **재보정 규율** —
리랭커 공식이 바뀌면 점수 스케일이 바뀌므로 문턱을 반드시 다시 재야 한다는 것을
주석에 절차로 남겼다. 숫자보다 "이 숫자가 언제 무효가 되는가"를 아는 게 검증 능력이다.
→ 코드: `agent.py SCORE_FLOOR/COVERAGE_FLOOR` 주석

## 장애 대응 — "깨졌을 때 어디를 보는가"

### Q8. 현업이 "답변이 이상해요"라고 하면 어디부터 보나?

**A.** 흐름 로그를 뒤에서 앞으로 거슬러 올라간다. ① 검증 게이트(흐름 14)의
`ok=`/`summary=` — 경고가 붙었는데 사용자가 못 봤나? ② 검색 완료(흐름 09)의
`top=` — 엉뚱한 문서가 1등으로 뽑혔나(검색 문제)? ③ 도구 요청(흐름 06)의
`args=` — Claude가 질의를 이상하게 만들었나(모델 문제)? ④ 마스킹(흐름 02) —
과잉 마스킹으로 질문이 뭉개졌나? 이 순서인 이유: 답변에 가까운 층부터 보면
"어느 층 밑으로는 정상"이라는 이등분 탐색이 되기 때문이다.

### Q9. `/health`에서 grounded=False 비율이 갑자기 늘었다. 원인 후보는?

**A.** grounded=False가 되는 경로는 셋뿐이다. ① 모델이 도구를 안 부르고 답함
(시스템 프롬프트 회귀나 모델 변경 의심 — 흐름 로그에 도구 요청이 없음),
② 도구는 성공했지만 결과 0건(코퍼스 변경·검색 회귀 의심 — `results=0`),
③ LLM API 실패(키 만료·장애 — `error=` 로그). 각각 로그 서명이 다르므로
흐름 로그 몇 건만 보면 셋 중 무엇인지 갈린다. 원인 후보를 **미리 열거해 두는 것**
자체가 장애 대응 준비다 — 장애 중에 처음 생각하면 늦다.

### Q10. 샘플 규제문서를 실제 사내 문서로 교체하면 어디가 깨질까?

**A.** 예상 파열 지점 네 곳. ① 청킹: 실제 문서는 마크다운 헤딩이 없거나 표·이미지가
많아 섹션 분할 가정이 깨진다(로더를 PDF/HWP 파서로 교체할 자리). ② 버전 메타:
시행일(effective_date)이 없는 문서는 fail-closed 정책 때문에 검색에서 통째로
제외된다 — preflight가 기동 전에 잡아주지만, 메타 정비 작업이 선행돼야 한다.
③ 문턱값: 코퍼스가 바뀌면 점수 분포가 바뀌므로 0.19/0.28은 재보정 전까지 무효다.
④ 평가셋: 질문-정답 쌍을 새 코퍼스로 다시 만들어야 Hit@1이 의미를 가진다.
"교체하면 끝"이 아니라 "교체 체크리스트"를 말할 수 있는 것이 FDE의 대답이다.

### Q11. HybridRetriever 는 표준 라이브러리인가, 직접 만든 것인가? (개념과 구현의 구분)

**A.** 구분해서 답해야 정확하다.

- **개념("하이브리드 검색")은 업계 표준이다.** 벡터 검색(의미) + 키워드 검색(BM25)을
  섞는 것은 RAG 실무의 사실상 기본기이고, "hybrid search"라는 이름 그대로 통용된다.
  상용 도구들에 이미 내장돼 있다 — Elasticsearch/OpenSearch 의 하이브리드 쿼리,
  Qdrant·Weaviate 같은 벡터 DB 의 hybrid search 옵션, LangChain 의
  `EnsembleRetriever`, LlamaIndex 의 hybrid 모드 등. "1차로 넓게 회수 → 리랭커로
  좁게 재정렬"이라는 2단계 구조도 표준 패턴이다.
- **이 클래스 자체는 이 프로젝트가 직접 짠 자작이다.** pip install 로 받은 것이
  아니라 `src/rag/retriever.py` 에 순수 파이썬으로 구현했고, 이름도 개념을 따라
  지었을 뿐이다. 일부러 그렇게 한 이유: ① 외부 의존성 없이 데모가 어디서든
  돌아가게 하고, ② "이 자리에 실무에선 Cross-Encoder 리랭커/벡터 DB를 끼운다"는
  교체 지점을 명시하기 위해서다(임베더의 `EmbeddingProvider` 인터페이스 분리와
  같은 설계).

**면접 답변 스크립트** — "하이브리드 검색 아세요?"라는 질문에:

> "업계 표준 패턴이고, 제 프로젝트에서는 의존성 없이 직접 구현했습니다 —
> 벡터는 TF-IDF 코사인, 키워드는 BM25, 각각 정규화 후 5:5 결합이고,
> 그 위에 커버리지·구문·섹션·제목 4신호 리랭커로 상위 3건을 좁힙니다.
> 실무라면 이 자리에 Elasticsearch 하이브리드나 상용 임베딩+벡터 DB를 꽂는
> 구조로 인터페이스를 분리해 뒀습니다."

개념 이해(표준 패턴) + 구현 경험(직접 짬) + 실무 감각(교체 지점)을 한 번에
보여주는 답이 된다. 이 "개념은 표준, 구현은 자작, 교체 지점은 명시"라는 구분은
HybridRetriever 뿐 아니라 이 프로젝트의 임베더(TF-IDF ↔ Voyage API)·벡터 스토어
(인메모리 ↔ pgvector/Qdrant)·마스킹(정규식 ↔ NER/Presidio)에도 똑같이 적용된다.

## 용어 사전

답변에 나온 용어를 이 프로젝트의 실제 로그·코드 기준으로 설명한다.
(※ 상당수는 일반 용어가 아니라 **이 코드베이스가 붙인 이름**이다 — 몰랐다면 정상.)

### 검색(RAG) 계열

- **RAG (Retrieval-Augmented Generation)** — "찾아서(검색) 근거를 쥐여주고 답하게
  하기". LLM이 아는 척 지어내는 대신, 관련 문서를 먼저 검색해 그 근거로만 답하게
  하는 구조. 이 프로젝트의 `search_regulations` 도구가 R(검색), Claude 의 답변
  생성이 G(생성)다.
- **top / top_n / top_k** — "상위 몇 개". 검색 결과는 관련도 점수순으로 줄 세우는데,
  `top_k=8`은 1차에서 8개를 넓게 뽑는다는 뜻, `top_n=3`(=rerank_n)은 최종적으로
  3개만 남긴다는 뜻. 로그의 `top=REG-002 6. 처리기한 (0.603)`은 "1등 문서가
  REG-002의 '처리기한' 섹션이고 점수 0.603"이라는 뜻이다.
- **score (점수)** — 질문과 문서 조각이 얼마나 관련 있는지를 수치화한 것. 절대적
  의미는 없고 **같은 공식 안에서의 상대 비교**만 의미가 있다 — 그래서 공식이
  바뀌면 문턱값도 다시 재야 한다(Q7).
- **청킹(chunking)** — 긴 문서를 검색 단위 조각(청크)으로 자르는 것. 이 프로젝트는
  마크다운 헤딩(##) 경계로 자르고, 긴 섹션은 500자/80자 겹침으로 다시 자른다.
  어떻게 자르느냐가 검색 품질을 좌우한다.
- **임베딩 / 벡터 검색** — 텍스트를 숫자 벡터로 바꿔(임베딩) 벡터끼리의 각도
  (코사인 유사도)로 "의미가 비슷한가"를 재는 검색. 단어가 달라도(부작용↔이상사례)
  의미가 가까우면 잡는 게 강점.
- **BM25 / 키워드 검색** — 단어가 실제로 등장했는지를 세는 고전 검색(도서관 검색
  같은 것). 고유명사·코드(REG-005)처럼 정확히 일치해야 하는 질의에 강하다.
- **하이브리드 검색** — 벡터(의미)와 BM25(키워드)를 절반씩 섞은 것(`HYBRID_ALPHA=0.5`).
  서로의 약점을 보완한다.
- **리랭킹(reranking)** — 1차로 넓게 뽑은 후보(8개)를 더 정밀한 기준으로 다시
  줄 세워 상위만(3개) 남기는 2단계. "넓게 회수 → 좁게 정밀"이 검색의 기본 전략.
- **질의 확장(query expansion)** — 검색 전에 동의어를 붙여주는 것
  ("부작용" → "부작용 이상사례"). 사용자 어휘와 문서 어휘가 다를 때를 메운다.

### 품질·검증 계열

- **환각(hallucination)** — LLM이 근거 없이 그럴듯한 거짓을 지어내는 것.
  규제 도메인에서 "15일"이 "30일"로 바뀌면 컴플라이언스 사고라, 이 프로젝트의
  방어 장치 대부분이 이걸 겨냥한다.
- **grounded** — 이 프로젝트가 정의한 응답 필드로, "**이 답변이 도구가 실제로
  반환한 근거 위에 서 있는가**"라는 뜻. 도구를 한 번도 안 부르고 답했거나 검색이
  0건이면 False — UI에서 "근거 보증 없음" 배지가 된다. (일반 업계 용어로도
  grounding = 답변을 실제 근거에 붙들어 매기.)
- **신뢰 소스(trusted_texts)** — 답변 속 수치를 대조할 "믿을 수 있는 텍스트 모음".
  이 프로젝트의 정의: 그 턴에 도구가 반환한 것(검색 문단 + 규칙 도구의 계산값)
  **에서 사용자 입력의 메아리를 뺀 것**.
- **에코(echo)** — 도구가 받은 입력을 출력에 그대로 되돌려주는 것. 검색 결과에
  `query: "..."`가 다시 들어있는 게 에코다. 이걸 신뢰 소스에 넣으면 사용자의
  틀린 전제가 '근거'로 승격되는 구멍이 된다(Q6).
- **문턱값(threshold)** — "이 값 아래면 다르게 행동한다"는 경계선 숫자.
  `SCORE_FLOOR=0.19`(1등 근거의 점수가 이보다 낮고) + `COVERAGE_FLOOR=0.28`
  (질문 단어가 근거에 이만큼도 안 나오면) → 답을 지어내는 대신 "근거를 찾지
  못했습니다"로 회피한다.
- **커버리지(coverage)** — 질문의 단어들 중 검색된 근거 텍스트에 실제로 등장한
  비율(0~1). 낮으면 "검색이 답할 재료를 못 찾았다"는 신호.
- **회피(abstention)** — 모르면 모른다고 하는 것. 환각의 반대 전략. 과하면
  아는 것도 회피하는 과회피(over-abstention)가 되므로 문턱 보정이 필요하다.
- **Hit@1** — 평가셋 질문들에 대해 "정답 문서가 검색 1등으로 나온 비율".
  1.0이면 32문항 전부 1등 적중. 검색 개선의 기준 지표.
- **홀드아웃(holdout)** — 튜닝에 쓰지 않고 숨겨둔 별도 평가셋. 튜닝에 쓴 문제로
  점수를 재면 "시험 문제를 미리 본 것"이라, 일반화 확인용으로 따로 둔다.
- **오탐/미탐 (false positive/negative)** — 오탐 = 문제없는 것을 문제라고 경고
  (양치기 소년), 미탐 = 문제를 놓침. 마스킹은 미탐이 유출이라 과탐을 감수하고,
  검증 경고는 오탐이 잦으면 무시당하므로(alert fatigue) 둘의 비용을 저울질한다.
- **alert fatigue** — 경고가 너무 잦아 사람이 경고를 무시하게 되는 현상.
  경고 시스템의 실질적 사망 원인이라, 경고율 추이를 `/health`로 상시 감시한다.

### 아키텍처·운영 계열

- **MCP (Model Context Protocol)** — LLM과 도구(사내 시스템)를 연결하는 표준
  규격. "모델은 요청만, 실행은 도구 서버가"의 경계를 만든다. 이 프로젝트는
  같은 프로세스 안에서 인메모리로 연결하지만 메시지는 MCP 규격 그대로다.
- **tool / tool_use / stop_reason** — tool = 모델에게 주는 "함수 사용설명서"
  (이름·설명·입력 스키마). Claude가 답변 대신 "이 도구를 이 인자로 실행해 달라"고
  요청하면 응답의 stop_reason(멈춘 이유)이 `tool_use`가 된다. `end_turn`이면
  "답변을 확정했다"는 뜻.
- **에러 계약(error contract)** — 도구가 실패를 조용히 숨기지 않고
  `{"error": "...", "available": [...]}` 처럼 **정해진 형식으로** 알려주기로 한
  약속. 에이전트가 이걸 보고 스스로 재시도(자가 정정)할 수 있다.
- **fail-closed** — 판단 불가능한 것은 통과시키지 않고 막는 방향. 예: 시행일을
  해석할 수 없는 문서는 검색에서 제외 — "언제부터 유효한지 모르는 규정"을
  현행이라고 내놓는 것보다 안 내놓는 게 안전하다는 판단. (반대는 fail-open.)
- **조용한 실패 vs 시끄러운 실패** — 조용한 실패 = 틀렸는데 아무 신호 없이 지나감
  (최악). 시끄러운 실패 = 틀릴 수 있음을 경고로 드러냄. 이 프로젝트의 일관된
  원칙이 "조용한 실패 금지"다(빈 결과 에러 계약, 검증 경고 부착, preflight 기동 차단).
- **폐지본(superseded) / as_of** — 개정으로 대체된 구버전 규정이 폐지본.
  기본 검색에서는 제외하지만, `as_of=2025-01-01`(그 시점 기준 조회)이면 "당시
  현행이던" 폐지본을 보여준다. 규제 도메인 특유의 버전 인지 검색.
- **preflight** — 서버 기동 **전에** 데이터·설정·핵심 동작을 자가 점검하고,
  하나라도 실패하면 서버를 아예 안 띄우는 배포 게이트. 항공기 이륙 전 점검표에서
  온 이름.
- **canary** — 정답을 아는 대표 케이스를 미리 심어두고 그것부터 확인하는 것
  (탄광의 카나리아). preflight가 "이 질문의 1등은 REG-005여야 한다"를 매 부팅마다
  확인하는 게 canary다.
- **span / trace** — 관측성 용어. span = 한 작업 구간의 기록(무슨 도구를 몇 ms에,
  성공/실패로), trace = 요청 하나가 만든 span 모음. 로그의
  `span {"name": "llm.step0", "ms": 3851...}` 이 그것.
- **계기판(/health)** — 운영 상태를 상시 노출하는 엔드포인트. 이 프로젝트는
  실행 모드·인덱스 상태·검증 경고율을 담아, "배치 다음 날 아침에 확인하는 화면"
  역할을 한다.
