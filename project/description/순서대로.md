# 순서대로 — LLM 모드에서 프롬프트 하나가 답변이 되기까지

> RAPV-Assistant(RA·PV 도구)가 **LLM 모드**(`ANTHROPIC_API_KEY` 설정됨)로 실행 중일 때,
> 사용자가 채팅창에 질문 하나를 입력한 순간부터 답변이 화면에 그려지기까지의
> 전 과정을 **파일 → 함수** 단위로 추적한 문서다.
>
> 예제 질문(문서 전체에서 이 질문 하나로 끝까지 따라간다):
>
> ```
> 환자 김민수님(010-1234-5678)이 약 복용 후 아나필락시스로 입원했습니다.
> 언제까지 보고해야 하나요? 인지일은 2026-07-10입니다.
> ```

---

## 한 장 요약 (전체 호출 체인)

```
[브라우저]  web/index.html            fetch POST /chat {message, history}
    │
[API]      src/api/main.py           chat(req) → agent.chat(...)
    │
[에이전트]  src/agent/agent.py        RaAgent.chat()
    │                                  ├─ redact(message)          ← src/pv/redactor.py (PII 마스킹)
    │                                  ├─ _redact_history(history)
    │                                  └─ _chat_llm()              ← LLM_AVAILABLE=True 분기
    │
[LLM 루프]  src/agent/agent.py        for step in range(6):
    │                                  ├─ client.messages.create() ← Anthropic API (Claude)
    │                                  ├─ stop_reason == "tool_use" ?
    │                                  │    ├─ _safe_tool_call() → mcp_client.call_tool()
    │                                  │    │      └─ src/mcp_server/server.py 의 @mcp.tool 함수 실행
    │                                  │    │            └─ (검색이면) src/rag/pipeline.py → retriever.py
    │                                  │    └─ 도구 결과를 messages 에 되먹임 → 다음 step
    │                                  └─ stop_reason != "tool_use" → 최종 텍스트 확정
    │
[검증]      src/agent/agent.py        _finalize()
    │       src/verify/verifier.py     └─ verify_answer()          ← 수치·날짜·방향·버전 사후 검증
    │       src/observability.py       └─ record_verification()    ← 계기판 집계 + 감사 로그
    │
[응답]      src/api/main.py           ChatResponse(JSON) 반환
    │
[브라우저]  web/index.html            답변·출처·도구호출·검증 배지 렌더링
```

## 0단계. 서버 부팅 (요청 이전에 1회 일어나는 일)

프롬프트 처리를 이해하려면 부팅 시 만들어지는 **RAG 인덱스**를 먼저 알아야 한다.
질의 시점에는 이 인덱스를 "조회"만 한다.

| 순서 | 파일 | 함수 | 하는 일 |
|---|---|---|---|
| 0-1 | `run.sh` / uvicorn | — | `src.api.main:app` 기동 |
| 0-2 | `src/api/main.py` | `lifespan()` | 부팅 훅에서 `_get_pipeline()` 호출 |
| 0-3 | `src/mcp_server/server.py` | `_get_pipeline()` | `RagPipeline().build()` 를 **모듈 전역에 1회 캐시** (API 서버와 MCP 서버가 같은 인덱스 공유) |
| 0-4 | `src/rag/pipeline.py` | `RagPipeline.build()` | 아래 0-5 ~ 0-7 오케스트레이션 |
| 0-5 | `src/rag/loader.py` | `load_documents(REG_DIR)` | `data/regulations/*.md` 를 읽어 YAML frontmatter(doc_id·version·effective_date·status 등)와 본문 분리 → `Document` 리스트 |
| 0-6 | `src/rag/chunker.py` | `chunk_documents(docs, 500, 80)` | 마크다운 헤딩(`##`) 경계로 섹션 분할(`_split_by_heading`) → 500자/80자 겹침 슬라이딩 분할(`_window_split`) → 각 청크 본문 앞에 `[문서제목 > 섹션]` prefix 부착(경량 Contextual Retrieval). 버전 메타(version·effective_date·status·superseded_by)도 청크에 보존 |
| 0-7 | `src/rag/retriever.py` | `HybridRetriever.index(chunks)` | ① `vectorstore.py InMemoryVectorStore.index()` — `embedder.py` TF-IDF 임베더를 코퍼스로 `fit()` 후 전 청크 벡터화 ② `BM25Index.index()` — 키워드 인덱스 구축 ③ 리랭커용 캐시 사전 계산(청크 토큰·섹션 토큰·제목 가중치·시행일 파싱 `_parse_date`) |

부팅이 끝나면 콘솔에 `[LLM 모드] Enterprise LLM API 사용 · model=...` 배너(`src/config.py mode_banner()`)와
`[RAG] 문서 N건 · 청크 M개 인덱싱 완료`가 찍힌다.

> 모드 결정은 `src/config.py` 가 한다: `LLM_AVAILABLE = bool(ANTHROPIC_API_KEY)`.
> 이 문서는 True(LLM 모드) 경로를 따라간다.

## 1단계. 브라우저 → FastAPI (`web/index.html` → `src/api/main.py`)

1. 사용자가 채팅창에 예제 질문을 입력하고 전송하면 `web/index.html` 의 스크립트가
   `fetch('/chat', {method: 'POST', body: JSON.stringify({message, history})})` 를 보낸다.
   `history` 에는 이전 턴들의 `{role, content}` 가 담긴다(멀티턴).
2. `src/api/main.py` 의 `@app.post("/chat")` → **`chat(req: ChatRequest)`** 가 받는다.
   Pydantic 모델 `ChatRequest` 가 `message: str`, `history: list[dict]` 를 검증한다.
3. `chat()` 은 모듈 전역 싱글턴 `agent = RaAgent()` 의 **`agent.chat(req.message, req.history)`** 를 await 한다.

---

## 2단계. 에이전트 입구 (`src/agent/agent.py` → `RaAgent.chat()`)

`RaAgent.chat(message, history)` 는 모드와 무관한 **공통 전처리**를 한다:

1. **`Trace()` 생성** (`src/observability.py`) — 이후 모든 단계(LLM 스텝·도구 호출)의
   지연·성패를 span 으로 기록할 그릇.
2. **PII 마스킹**: `redact(message)` (`src/pv/redactor.py`).
   - `unicodedata.normalize("NFKC", ...)` 로 전각→반각 폴딩 후, 정규식 패턴 목록
     `_PATTERNS`(주민번호·전화번호·이메일·환자/차트번호)과 호칭 붙은 이름
     `_NAME_RE`(+오탐 방지 `_NAME_STOP`)를 순서대로 치환.
   - 예제 질문은 여기서 이렇게 바뀐다:
     ```
     환자 [이름]님([전화번호])이 약 복용 후 아나필락시스로 입원했습니다.
     언제까지 보고해야 하나요? 인지일은 2026-07-10입니다.
     ```
   - 반환된 `RedactionReport` 는 **원문을 보존하지 않고** 유형·건수만 남긴다
     (`{"이름(호칭)": 1, "전화번호": 1}`). 이 요약이 최종 응답의 `redactions` 필드가 된다.
   - 마스킹이 **에이전트 입구**인 이유: 이후의 모든 경로(외부 LLM API·검색·로그·트레이스)에
     원문 개인정보가 흘러들지 않게 하기 위해서다.
3. **이력 마스킹**: `_redact_history(history)` — 클라이언트가 보낸 이전 턴에도 PII 가
   있을 수 있으므로 문자열 콘텐츠와 anthropic 블록 리스트(`text`·`tool_result` 의 `content`)
   양쪽 표기를 모두 마스킹.
4. **모드 분기**: `config.LLM_AVAILABLE` 이 True 이므로 **`_chat_llm(message, history, trace)`** 로 진입.
   (`timed(trace, "chat", "agent")` 컨텍스트로 감싸 총 wall-clock 을 기록.)
5. 분기 결과가 돌아오면 `result.trace = trace.to_list()`, `result.latency_ms = trace.total_ms`,
   `result.redactions = red.summary()` 를 채워 API 계층으로 반환.

## 3단계. LLM tool-use 루프 (`RaAgent._chat_llm()`)

여기가 LLM 모드의 심장부다. "관찰→생각→도구호출→관찰"을 Claude 가 스스로 반복한다.

### 3-1. 준비

| 순서 | 코드 | 하는 일 |
|---|---|---|
| ① | `anthropic.AsyncAnthropic(api_key=...)` | Anthropic API 클라이언트 생성 |
| ② | `async with Client(mcp)` (`fastmcp.Client`) | `src/mcp_server/server.py` 의 `mcp` 서버 객체에 **인메모리 연결** — 별도 프로세스 없이, 그러나 도구는 반드시 MCP 규격을 통해서만 호출 |
| ③ | `await mcp_client.list_tools()` → `_to_anthropic_tools()` | MCP 도구 6종(search_regulations / get_ra_deadlines / get_submission_checklist / assess_adverse_event / draft_ae_report / list_regulation_documents)의 이름·docstring·입력 스키마를 **Anthropic tools 포맷**으로 변환. 도구 docstring 이 곧 Claude 에게 주는 사용설명서다 |
| ④ | `messages = [*history, {"role":"user", "content": message}]` | 대화 컨텍스트 조립 |

동시에 검증용 상태 변수들을 초기화한다:
`trusted_texts`(사후 검증의 신뢰 소스), `raw_search_results`(출처 수집용),
`history_doc_ids`(이력 검색이 실제 반환한 문서 — 폐지본 인용 경고 면제 범위),
`grounded_evidence`(내용 있는 도구 출력 수신 여부), `user_facts`(케이스 서술 에코 — 2계층 신뢰 소스).

### 3-2. 루프 (최대 6스텝 — 무한루프 방지)

각 step 에서:

1. **`client.messages.create(model=config.LLM_MODEL, max_tokens=1024, system=SYSTEM_PROMPT, tools=tools, messages=messages)`**
   - `SYSTEM_PROMPT`(agent.py 상단): "반드시 MCP 도구로 조회한 근거에 기반해서만 답하고,
     근거 없으면 모른다고 하라. 출처(문서명·섹션)를 함께 제시하라."
   - API 호출은 `timed(trace, f"llm.step{step}", "llm")` 로 감싸 지연을 기록.
   - 예외(잘못된 키·네트워크 오류 등)가 나면 크래시 대신 **명시적 안내문**으로 답하고
     `grounded=False` 로 `_finalize()` 에 넘긴다(예외 메시지는 답변에 싣지 않는다 —
     에러 문구에 요청 정보가 에코될 수 있어서).

2. **분기: `resp.stop_reason`**
   - `"tool_use"` 가 **아니면** → Claude 가 답변을 확정한 것. 텍스트 블록을 이어 붙여
     `AgentResult(answer=text, mode="llm", citations=..., grounded=grounded_evidence)` 를 만들고
     **`_finalize()`(5단계)로 직행** — 루프 종료.
   - `"tool_use"` **이면** → 3-3 으로.

3. **(3-3) 도구 호출 실행** — `resp.content` 의 각 `tool_use` 블록마다:
   1. **`_safe_tool_call(mcp_client, block.name, dict(block.input), trace)`**
      → `mcp_client.call_tool(name, args)` 로 MCP 도구 실행(4단계 참고).
      실패해도 크래시하지 않고 `(에러 문자열, is_error=True)` 를 반환.
   2. **신뢰 소스 축적** (성공 & 에러 계약 아님일 때만):
      - `_is_contract_error(data)` — 도구가 `{"error": ...}` 계약으로 답했으면
        신뢰 소스에서 **통째로 제외**(에러 문구에 사용자 입력이 에코되므로).
      - `_split_user_facts(data)` — 도구 출력에서 입력 에코 `query`·`as_of` 를 제거하고
        (`_ECHO_KEYS` — 사용자 전제가 검증의 '근거'로 승격되는 구멍 차단),
        케이스 에코 `case` 는 **별도 계층 `user_facts`** 로 분리(검증 시 from_case 라벨용).
      - 남은 출력을 `_stringify()`(JSON 직렬화)해 `trusted_texts` 에 추가.
      - `_has_evidence(stripped)` 가 True(빈 결과가 아님)면 `grounded_evidence = True`.
   3. **출처 수집**: `search_regulations` 성공 결과는 `raw_search_results` 에,
      `assess_adverse_event`/`draft_ae_report` 의 `basis`(판정 근거 규정)도 출처로 합류.
      `include_superseded`/`as_of` 검색이면 반환된 `doc_id` 들을 `history_doc_ids` 에 추가.
   4. `tool_calls.append(ToolCall(name, args, _summarize(data)))` — UI 표시용 요약.
   5. **되먹임**: `{"type":"tool_result", "tool_use_id":..., "content": _stringify(data), "is_error": is_error}`
      를 모아 `messages.append({"role":"user", "content": tool_results_content})`.
      → 다음 step 의 `messages.create()` 에서 Claude 가 이 결과를 "관찰"하고
      추가 도구 호출 또는 최종 답변을 결정한다. **도구가 실패해도 에러를 모델에
      되먹여 스스로 복구(다른 인자로 재시도)하게 하는 것**이 이 구조의 핵심.

4. 6스텝을 다 써도 답이 확정되지 않으면 "질문을 좁혀 달라"는 안내문 + `grounded=False` 로 `_finalize()`.

### 예제에서의 루프 전개 (전형적 시나리오)

```
step 0: Claude 판단 — "구체적 AE 케이스 + 기한 질문" → assess_adverse_event 호출
        input: {case_description: "환자 [이름]님([전화번호])이 약 복용 후 아나필락시스로
                입원했습니다...", awareness_date: "2026-07-10"}
        (도구 docstring 이 "구체적 케이스면 이 도구, 규정 자체가 궁금하면
         search_regulations" 라고 안내하므로 Claude 가 이를 근거로 선택)

step 1: 도구 결과(중대성=Serious, 기한=2026-07-25, 근거 REG-005, 인과성 제안,
        MedDRA 코딩)를 관찰 → 근거가 충분하다고 판단 → stop_reason="end_turn"
        → 최종 답변 텍스트 생성:
        "중대한 이상사례(입원)에 해당하여 신속보고 대상입니다. 인지일 2026-07-10
         기준 15일 이내인 2026-07-25까지 보고해야 합니다. (근거: 의약품 이상사례
         보고 규정 REG-005 > 신속보고 기한)"
```

(질문이 규정 조회형이었다면 step 0 에서 `search_regulations` 를, 초안 요청이었다면
`draft_ae_report` 를 선택했을 것이고, 근거가 부족하면 2~3개 도구를 연쇄 호출한다.)

## 4단계. MCP 도구 내부 (`src/mcp_server/server.py`)

`mcp_client.call_tool()` 이 실제로 실행하는 함수들. 예제에서 호출된
`assess_adverse_event` 와, 가장 대표적인 `search_regulations` 두 개를 파고든다.

### 4-A. `assess_adverse_event(case_description, awareness_date)` — PV 트리아지

| 순서 | 파일 | 함수 | 하는 일 |
|---|---|---|---|
| ① | `src/pv/redactor.py` | `redact(case_description)` | **심층방어 재마스킹** — 에이전트 경유면 이미 마스킹돼 있지만, stdio 단독 사용(Claude Desktop 직결) 대비 도구 계층에서도 마스킹 |
| ② | `src/pv/triage.py` | `assess_case(text, awareness_date)` | **규칙 기반(결정론)** 중대성 판정: 중대성 기준(사망·입원·생명위협…) 감지 → Serious 여부, 보고 경로(신속보고/PSUR), **기한 계산**(인지일 + 15일 등). 컴플라이언스 계산이라 LLM 추론이 아닌 감사 가능한 규칙으로 수행 |
| ③ | `src/pv/causality.py` | `assess_causality(text)` | WHO-UMC 인과성 등급 제안 + 부족 정보(missing_info) 생성 |
| ④ | `src/pv/coding.py` | `code_terms()` / `suggest_candidates()` / `flag_uncoded_expressions()` | 증상 표현 → MedDRA 방식 PT/SOC 코딩(확정/후보/미코딩 3계층) |
| ⑤ | (같은 파일) | `search_regulations("중대한 이상사례 보고 기한 신속보고", top_n=2)` | 판정 근거 규정 문단(REG-005)을 RAG 로 회수해 `basis` 로 부착 — **추적성** |

반환 dict 에는 `case`(마스킹된 서술 에코), `is_serious`, `deadline_date`, `rationale`,
`causality`, `coded_terms`, `basis` 등이 담긴다. 이 중 `deadline_date=2026-07-25` 같은
**도구가 계산한 값**이 5단계 검증의 신뢰 소스가 된다.

### 4-B. `search_regulations(query, top_n, as_of, include_superseded)` — RAG 검색

| 순서 | 파일 | 함수 | 하는 일 |
|---|---|---|---|

## 5단계. 사후 검증 게이트 (`_finalize()` → `src/verify/verifier.py`)

**모든 응답**(정상 답변·에러 안내·루프 한계 안내)이 이 관문을 지난다.
LLM 답변은 매번 새로 생성되므로, 답변 속 "15일"이 "30일"로 바뀌는 오류를
평가셋이 아닌 **런타임 결정론 검증**으로 잡는 것이 이 계층의 존재 이유다.

1. `src/agent/agent.py` **`_finalize(result, trusted_texts, question=..., allowed_superseded_ids=..., user_facts=...)`**
   가 `verify_answer()` 를 부른다.
2. `src/verify/verifier.py` **`verify_answer(answer, trusted_texts, citations, ...)`** 의 4개 검증 축:
   - **① 수치·날짜 존재 대조**: 답변에서 `숫자+단위`(15일, 120 근무일, 90%…),
     고유어 수량(보름, 이틀…), 날짜(ISO·한국어 표기·부분 표기)를 정규식으로 추출
     (`_NUM_UNIT_RE`, `_DATE_RE` — 한글이 `\w` 라서 `\b` 대신 룩어라운드 경계 사용)
     → 각각이 **신뢰 소스**(검색 근거 ∪ 결정론 도구 출력 − 질문 에코) 안에 존재하는지 대조.
     미지원이면 `unsupported`, 질문에 있던 값이면 `from_question`(환각이 아니라 '전제 확인 필요'),
     케이스 서술에만 지지되면 `from_case` 라벨.
   - **② 방향 한정어**: "15일 **이내**" ↔ "15일 **이후**" 방향 뒤집기를 닫힌 어휘 집합으로 검증
     (판정 기준은 strict 계층 — 케이스 서술이 방향 경고를 무력화하지 못하게).
   - **③ 날짜 역할**: 도구 출력의 역할 키(deadline_date·awareness_date)와 대조해
     "보고 기한은 (인지일)입니다" 같은 역할 맞바꿈을 잡는다.
   - **④ 인용 버전**: citations 의 `status="superseded"` 문서 인용을 경고
     (이력 검색이 실제 반환한 `allowed_superseded_ids` 의 문서만 면제).
3. **실패해도 차단하지 않는다** — `v.ok` 가 False 면 `warning_text(v)` 를 답변 본문 끝에
   부착(시끄러운 실패, 최종 확정은 사람). 판정 요약은 `result.verification` 필드로 노출.
4. `src/observability.py` **`record_verification(summary)`** — `gate_stats` 계기판에
   축별 경고율 집계(`/health` 로 상시 노출, alert fatigue 조기 신호) + JSON 감사 로그.

예제라면: 답변 속 "15일", "2026-07-25", "2026-07-10" 은 모두 `assess_adverse_event`
출력(신뢰 소스)에 존재 → 통과. "이내" 방향도 근거와 일치 → 통과. `ok=true`.

## 6단계. 응답 반환과 렌더링

1. `AgentResult` 가 `RaAgent.chat()` 으로 돌아오고, 2단계에서 만든 `Trace` 가
   `trace`(스텝별 ms·성패)·`latency_ms` 로, 마스킹 요약이 `redactions` 로 채워진다.
2. `src/api/main.py chat()` 이 `ChatResponse` 로 직렬화:
   ```json
   {
     "answer": "중대한 이상사례(입원)에 해당하여 ... 2026-07-25까지 보고 ...",
     "mode": "llm",
     "tool_calls": [{"name": "assess_adverse_event", "args": {...}, "summary": "중대 → 신속보고"}],
     "citations": [{"doc_id": "REG-005", "title": "...", "section": "...", "score": 0.83, ...}],
     "grounded": true,
     "trace": [{"name": "llm.step0", "ms": 1830.2, ...}, {"name": "tool.assess_adverse_event", ...}],
     "latency_ms": 4211.5,
     "redactions": [{"type": "이름(호칭)", "count": 1}, {"type": "전화번호", "count": 1}],
     "verification": {"ok": true, "checked": 3, ...}
   }
   ```
3. `web/index.html` 이 answer 를 말풍선으로, citations 를 출처 카드로, tool_calls 와
   trace 를 접이식 상세로, grounded·verification·redactions 를 배지로 렌더링한다.

## 부록 A. 오프라인 모드는 어디서 갈라지나

같은 `RaAgent.chat()` 입구에서 `config.LLM_AVAILABLE=False` 면 `_chat_offline()` 로 간다.
Claude 의 자율 도구 선택 대신 **규칙 라우터**가 대신한다:

- `_resolve_followup()` — 짧은 후속질문을 직전 발화와 병합(멀티턴 근사)
- `_route_intent()` — 키워드 규칙으로 ae_report / ae_triage / deadlines / checklist / search 분기
- 검색 경로는 `SCORE_FLOOR`(0.19)·`COVERAGE_FLOOR`(0.28) AND 조건으로
  근거 부족 시 **abstention**(환각 대신 회피)
- 답변은 생성이 아니라 `_format_search_answer()` 등의 **추출형 포매팅**

두 모드 모두 도구는 MCP 를 통해서만 호출하고, 같은 `_finalize()` 검증 게이트를 지난다.

턴 1 — 기준선 만들기

의약품 변경허가 처리기한은 며칠인가요?

볼 것: [흐름 01] ... history_turns=0 — 첫 질문이라 빈 배열. 이 답변이 이제 브라우저 history에 저장됩니다. 답변에 나온 기한 숫자를 기억해 두세요(턴 3에서 씁니다).

턴 2 — 후속질문 병합 (history의 첫 사용)

그럼 변경신고는?

볼 것: history_turns=2로 바뀐 것. 그리고 모드별로:
- 오프라인: _chat_offline 로그의 resolved= 값이 내가 친 문장이 아니라 "의약품 변경허가 처리기한은 며칠인가요? 그럼 변경신고는?" 으로 병합돼 있습니다 — _resolve_followup()이 "그럼"(마커) + 짧은 문장을 보고 직전 발화를 이어 붙인 것입니다.
- LLM: LLM 호출(step 0) ... messages=3 — 이전 턴 2개 + 이번 질문 1개. 병합 없이 대화록 통째로 Claude에 갑니다.

턴 3 — 틀린 전제 심기 (검증 게이트의 from_question)

턴 1에서 확인한 실제 기한과 다른 숫자로 물어보세요. 예를 들어 실제가 15일이었다면:

중대한 이상사례 보고 기한이 30일 맞지?

볼 것: _finalize() 로그의 summary= 안에서 question_origin 항목. LLM 모드에서 Claude가 "30일이 아니라 15일입니다"라고 정정하면 답변에 30일이 재서술되는데, 이 30일은 근거(도구 출력)에 없는 값입니다. 검증기가 이를 '환각'이 아니라 **'질문에 있던 값 = 전제 확인 필요'**로 구분해 라벨링하는 게 이 프로젝트 검증 설계의 백미입니다. (오프라인 모드는 발췌형이라 질문 속 숫자를 답변에 옮기지 않아 이 라벨이 잘 안 뜹니다 — LLM 모드에서 확인하세요.)

턴 4 — 전혀 다른 주제로 점프

품목허가 준비 체크리스트 알려줘

볼 것: history_turns=6 — 3턴(6개)이 찼습니다. 이 턴부터는 새 턴이 추가될 때마다 가장 오래된 턴이 shift()로 밀려납니다. 이 질문의 목적은 다음 턴을 위한 "직전 발화 갈아끼우기"입니다.

턴 5 — 후속질문 병합의 한계 확인

그럼 그건 언제까지?

볼 것: 오프라인 resolved= 로그. 병합 상대가 **턴 4(체크리스트)**입니다 — 턴 2~3에서 얘기하던 기한이 아니라요. _resolve_followup()은 직전 사용자 발화 하나만 이어 붙이기 때문입니다(전체 대화를 이해하는 게 아님). 대명사 "그건"이 뭘 가리키는지 사람은 애매하다고 느끼는데, 규칙 기반 병합은 기계적으로 직전 턴을 선택한다는 것 — 오프라인 근사의 경계가 여기서 보입니다. LLM 모드는 대화록 전체를 보므로 더 그럴듯하게 해석합니다(messages=7).

턴 6 — 기억의 끝 확인

맨 처음에 내가 물어본 질문이 뭐였지?

볼 것: history_turns=6(상한 고정). 이 시점 history에는 턴 3~5만 남고 턴 1은 이미 밀려났습니다. LLM 모드라면 Claude가 턴 1을 모르거나 턴 3부터를 "처음"으로 답합니다 — 서버가 기억을 안 하고, 브라우저도 최근 3턴만 유지한다는 것이 사용자 눈에 드러나는 순간입니다.

---
정리하면 이 세션 하나로 확인되는 원리: **① history는 턴마다 2개씩 자라고(턴1→2), ② 오프라인은 직전 발화 병합이라는 근사를 쓰며(턴2, 5), ③ 검증 게이트는 질문에서 온 수치를 환각과 구분하고(턴3), ④ 기억은 브라우저의 최근 3턴이 전부(턴4, 6)**입니다. 각 턴 사이에 흐름 로그의 history_turns=, resolved=, messages=, summary= 네 값만 추적하면 됩니다.
