# 파일 구조 안내 — 처음 이 프로젝트를 분석하는 사람을 위한 읽기 지도

> 이 문서는 RA-Assistant 데모(제약 RA·PV 업무용 RAG + MCP 에이전트)를 **처음 접하는 사람**이
> 어떤 순서로, 어떤 파일을 읽어야 전체를 가장 빠르게 이해할 수 있는지 안내한다.
> 각 파일이 "무엇을 담고 있고, 왜 존재하는지"를 함께 설명한다.

---

## 0. 30초 요약 — 이 프로젝트는 무엇인가

제약회사의 **RA(인허가/규제업무)·PV(약물감시)** 담당자를 위한 AI 어시스턴트 데모다.

- **RAG**: 규제문서를 근거·출처와 함께 검색한다 (하이브리드 검색 + 리랭킹 + 질의확장).
- **MCP**: 마감일 조회, 이상사례 트리아지, 보고서 초안 같은 업무 도구를 FastMCP 서버로 표준화했다.
- **Agent**: LLM(또는 오프라인 규칙 라우터)이 질문 의도에 맞는 MCP 도구를 골라 호출한다.
- **신뢰성 장치**: PII 마스킹(에이전트 입구 + 도구 인자 2겹), 환각 회피(abstention), 버전 인지
  검색, 답변 사후 검증 게이트(수치·날짜·방향 한정어·날짜 역할 대조), 배포 전 점검(preflight —
  검증 게이트·PII 마스킹 자가 테스트 포함), 검증 경고율 계기판. 관통 원칙은 **조용한 실패
  금지** — 틀린 기준일의 조용한 폴백, 오타 난 필터의 "마감 없음" 오답, 부풀려진 지연 계기판
  같은 '조용히 틀리는' 경로를 명시적 신호(caveat·에러·wall-clock)로 바꿨다.
- **평가/테스트**: 평가 스크립트 5종 + pytest 136케이스 + CI로 모든 주장을 수치로 재현한다.

## 1. 추천 읽기 순서 (전체 로드맵)

```
[1단계: 큰 그림]        README.md → architecture.md → docs/프로젝트_소개서.md
[2단계: 데이터]         data/regulations/*.md → data/ra_tasks.json
[3단계: 요청의 흐름]     src/api/main.py → src/agent/agent.py → src/mcp_server/server.py
[4단계: RAG 심화]       src/rag/ (loader → chunker → embedder → vectorstore
                                  → synonyms → retriever → pipeline 순서)
[5단계: PV 도메인]      src/pv/ (redactor → triage → causality → coding → report 순서)
[6단계: 신뢰성 장치]     src/verify/verifier.py → src/preflight.py → src/observability.py → src/config.py
[7단계: 증명]           eval/ → tests/ → .github/workflows/ci.yml
[8단계: 설계 근거]       docs/ARCHITECTURE.md → docs/면접노트.md
```

**왜 이 순서인가**: 코드는 "요청이 흘러가는 길"을 따라 읽는 것이 가장 빠르다.
사용자의 질문은 `API → 에이전트 → MCP 도구 → (RAG 또는 PV) → 검증 → 응답` 순서로 흐르므로,
큰 그림(1)과 재료(2)를 본 뒤 그 길(3)을 따라가고, 각 갈래(4·5)를 깊이 판 다음,
품질을 지키는 장치(6·7)와 설계 이유(8)를 마지막에 본다.

## 4. 3단계 — 요청의 흐름 (엔트리포인트부터)

### `src/api/main.py` (109줄) — 입구
- FastAPI 앱. `lifespan` 훅에서 부팅 시 RAG 인덱스를 1회 구축한다.
- 엔드포인트: `/chat`(질문→답변), `/health`(모드·인덱스 상태 + **검증 게이트 경고율 계기판**
  `verification_gate`), `/api/deadlines`, `/`(챗 UI 서빙).
- `ChatRequest`/`ChatResponse` Pydantic 스키마. 응답에 `citations`(출처), `tool_calls`(도구 트레이스),
  `trace`(스텝별 지연), `grounded`, `redactions`(PII 마스킹 건수), `verification`(사후 검증 결과)이 실린다.
- **읽는 이유**: 시스템의 계약(입출력)이 여기 다 정의돼 있다.

### `src/agent/agent.py` (606줄) — 두뇌
이 프로젝트에서 가장 중요한 파일. 순서대로:
1. **입구 PII 마스킹**: 메시지·이력 모두 `redact()` — 이후 모든 경로(LLM API·로그)에 개인정보가 안 나간다.
2. **모드 분기**: `ANTHROPIC_API_KEY`가 있으면 **LLM 모드**(Claude tool-use 루프, 최대 6스텝,
   도구 에러는 `is_error`로 되먹여 자가복구), 없으면 **오프라인 모드**(규칙 라우터).
3. **오프라인 라우팅** `_route_intent()`: 케이스 서술→PV 도구, 마감 어휘→마감일 도구,
   기본→규제문서 검색.
4. **Abstention**: `SCORE_FLOOR=0.19`(근거 관련도)와 `COVERAGE_FLOOR=0.28`(질의 커버리지)
   **둘 다** 미달일 때만 "근거 없음"으로 회피(AND 조건 — 과회피 방지). 문턱값 보정 근거가 주석에 있다.
5. **멀티턴**: `_resolve_followup()` — "그럼 그건?" 같은 후속질문을 직전 발화와 병합.
6. `_finalize()`: 출처 수집·dedupe, 사후 검증 게이트 호출, 검증 결과의 **계기판 집계·감사
   로그**(`record_verification`), 응답 조립.

### `src/mcp_server/server.py` (355줄) — 도구 상자
- FastMCP 서버. MCP 3대 primitive를 모두 구현:
  - **Tools 6종**: `search_regulations`(RAG 검색), `get_ra_deadlines`, `get_submission_checklist`,
    `assess_adverse_event`(PV 트리아지+인과성+코딩), `draft_ae_report`(ICSR 초안),
    `list_regulation_documents`
  - **Resource**: `regulation://{doc_id}` (문서 원문 열람)
  - **Prompt**: `pv_case_intake` (케이스 처리 SOP를 클라이언트 무관하게 배포)
- 각 도구의 **docstring이 곧 LLM용 사용설명서**다 — "언제 이 도구를 쓰고 언제 다른 도구를 쓸지"가 명시돼 있다.
- **도구의 실패 방향**: 존재하지 않는 필터·카테고리에는 빈 결과("마감 없음" — 자신 있는 오답)가
  아니라 `{"error", "available": [...]}` 를 반환한다 — 에이전트가 available 을 보고 스스로
  정정 재시도하는 계약(조용한 빈 결과 금지).
- **도구 계층 PII 심층방어**: PV 도구의 자유 텍스트 인자는 `case_description` 뿐 아니라
  `reporter`·`patient_info` 까지 **전부** 마스킹한다 — stdio 단독 사용 시 인자는 에이전트
  입구 마스킹을 거치지 않고 직접 들어오기 때문(우회로 차단).
- 인메모리(`fastmcp.Client(mcp)`, 데모 기본)와 stdio(`python -m src.mcp_server.server`,
  Claude Desktop/Cursor 연결) 두 방식을 지원한다.

## 5. 4단계 — RAG 파이프라인 (`src/rag/`)

데이터가 흐르는 순서대로 읽는다:

| 순서 | 파일 | 내용 |
|---|---|---|
| ① | `loader.py` (66줄) | 마크다운 + YAML frontmatter 파싱 → `Document`. 실무의 PDF/HWP 파서 자리를 대신하는 데모 입구. 버전 메타를 여기서 싣는다. |
| ② | `chunker.py` (98줄) | 두 겹 청킹: 헤딩(`##`) 구조 분할 + 긴 섹션은 500자/80자 overlap 슬라이딩 윈도우. 각 청크 앞에 `[문서제목 > 섹션제목]`을 붙인다(경량 Contextual Retrieval). 청크가 doc_id·섹션·버전 메타를 들고 다닌다. |
| ③ | `textutil.py` (39줄) | 한국어 토크나이저: 어절 + **CJK 문자 bi-gram**(조사·띄어쓰기 변형 대응, 형태소 분석기 없는 경량 근사). |
| ④ | `embedder.py` (193줄) | `EmbeddingProvider` Protocol(인터페이스) 하나에 3구현 — `TfidfEmbedder`(기본, 의존성 0), `HashingEmbedder`(feature hashing, OOV에 강함), `VoyageEmbedder`(실제 상용 임베딩 API). **임베더 교체 가능성**의 증명. 희소 벡터 + 코사인. |
| ⑤ | `vectorstore.py` (41줄) | 인메모리 벡터 스토어(코사인 top-k). 실무 Vector DB(pgvector/Qdrant)의 자리. |
| ⑥ | `synonyms.py` (66줄) | 질의확장: 도메인 동의어 사전("부작용"→"이상사례", "심각"→"중대한"). **단방향(구어→정식 용어), 원 질의에 append**. 사전 항목은 eval 오류 분석으로 발굴/검증. |
| ⑦ | `retriever.py` (359줄) | **RAG의 심장.** `BM25Index`(키워드 검색), `HybridRetriever`(1차: 벡터+BM25 min-max 정규화 후 α결합 → top 8 / 2차: 리랭킹 v3 — coverage·phrase·section·title 4신호 + 섹션 타입 prior(대조 섹션 페널티·서두 감쇠, 질의 의도 게이트) → top 3). 버전 필터 `_is_active`(폐지본 제외, `as_of` 시점 조회)도 여기에. |
| ⑧ | `pipeline.py` (97줄) | 전 단계를 묶는 `RagPipeline`: `build()`(부팅 1회 인덱싱), `retrieve()`(질의 시 검색) → `RetrievedContext`(근거 블록+citations). |

## 6. 5단계 — PV 워크플로 (`src/pv/`)

PV 담당자의 실제 케이스 처리 순서 그대로 모듈이 나뉜다. **전부 규칙 기반(결정론)** —
보고기한 계산·요건 판정은 컴플라이언스라 LLM의 확률적 추론에 맡기지 않는다는 것이 대원칙.

| 순서 | 파일 | 내용 |
|---|---|---|
| ① | `redactor.py` (93줄) | **PII 비식별화.** 정규식으로 주민번호·전화·이메일·환자번호·호칭 붙은 이름(`홍길동님`→`[이름]님`) 마스킹. 값은 지우되 존재 신호는 남긴다(최소보고요건 '식별 가능한 환자' 판정과 양립). 이름 오탐은 스톱리스트로 차단. 여러 입력 필드의 마스킹 리포트를 합산하는 `merged_summary` 포함(도구 인자별 마스킹의 단일 감사 리포트). |
| ② | `triage.py` (142줄) | **중대성 판정 + 보고기한 계산.** 중대성은 규정의 닫힌 목록(사망·생명위협·입원·장애·기형·의학적 중요) 대조라 규칙으로 '확정'. 기한: 사망·생명위협→지체 없이, 그 외 중대→인지일+15일, 비중대→PSUR. 예상 여부 불명이면 보수 적용. 인지일 형식 오류는 조용히 오늘로 폴백하지 않고 **caveat로 명시**(기한이 틀린 기준일로 계산됐다는 신호 — 시끄러운 실패). |
| ③ | `causality.py` (107줄) | **인과성 평가(WHO-UMC).** 4신호(시간관계·dechallenge·rechallenge·대체원인) 감지 → 등급을 '확정'이 아니라 **'제안'** + 부족 정보는 follow-up 질문 생성. 트리아지와 확신 수준을 다르게 설계한 것이 포인트. |
| ④ | `coding.py` (182줄) | **용어 표준화 3계층(MedDRA 방식).** ①검수된 소사전 매칭=확정(`code_terms`) ②LLT 스타일 참조=후보(`suggest_candidates`, needs_confirmation) ③사전에 없는 증상 서술=미코딩 감지(`flag_uncoded_expressions`). 후보/확정을 타입 수준에서 분리해 집계 오염을 구조적으로 차단. |
| ⑤ | `report.py` (209줄) | **ICSR/KAERS 보고서 초안.** 초안 생성 전에 최소보고요건(ICH E2D 4요소: 환자·보고자·의심약·이상사례)을 검증 — 미충족이면 `reportable=False` + 보완 질문. 의심약 감지는 노출 맥락("OO정을 복용")이 있을 때만 매칭(오탐 방지). |

## 7. 6단계 — 신뢰성·설정 계층

| 파일 | 내용 |
|---|---|
| `src/verify/verifier.py` (428줄) | **답변 사후 검증 게이트.** 모든 응답의 수치·단위(근무일≠일)·고유어 수사(보름=15일)·날짜·**방향 한정어(이내↔이후)**·**날짜 역할(기한↔인지일 — 두 날짜가 모두 근거에 있어도 역할이 뒤바뀌면 잡는 축)**을 추출해 신뢰 소스(검색 근거 ∪ 결정론적 도구 출력 − 질문 에코)와 대조. 폐지본 인용 감지. 실패 시 차단이 아니라 경고 부착. |
| `src/preflight.py` (236줄) | **배포 전 점검(FDE Day-0 게이트).** 기동 전에 4그룹을 검사: 설정 불변식(모순 조합), 코퍼스 무결성(frontmatter 필수 필드·doc_id 유일성·**폐지 체인**), 업무 데이터 스키마, 스모크 canary(RAG·PV) + **안전장치 자가 테스트**(검증 게이트: 심은 오류가 걸리는가 / PII 마스킹: 심은 개인정보가 지워지는가). 실패 시 exit 1 — `run.sh`·CI가 이 게이트 뒤에 있다. |
| `src/observability.py` (159줄) | `Trace`/`Span` 구조 + `timed()` 컨텍스트 매니저. 도구 호출·LLM 스텝별 지연·성패 기록, 구조화(JSON) 로그. 총 지연(`total_ms`)은 최상위 agent span의 **wall-clock** — 중첩 span 합산은 같은 시간을 두 번 세어 계기판이 ~2배 지연을 보고한다(관측이 틀리면 SLA 판단이 함께 틀린다). 여기에 **검증 게이트 운영 계기판**(`GateStats` — 경고율·축별 집계, `/health` 노출)과 응답 단위 **감사 로그**(`record_verification`)가 있다. OpenTelemetry 개념의 무의존성 구현. |
| `src/config.py` (48줄) | 모든 하이퍼파라미터가 환경변수로 제어: `CHUNK_SIZE=500`, `CHUNK_OVERLAP=80`, `RETRIEVE_TOP_K=8`, `RERANK_TOP_N=3`, `HYBRID_ALPHA=0.5`, `RERANK_WEIGHT=0.9`, 섹션 prior 페널티(0.3/0.055), `EMBEDDER_KIND`, `QUERY_EXPANSION`. 값의 근거는 `eval/sweep.py`로 재현. |
| `web/index.html` | 단일 페이지 챗 UI. 출처·도구 트레이스·지연·PII 마스킹·검증 배지 표시, 멀티턴 지원. |

## 8. 7단계 — 평가와 테스트 ("측정 없이 신뢰 없음")

### `eval/` — 평가 스크립트 5종 + 데이터셋

| 파일 | 무엇을 측정하나 |
|---|---|
| `evaluate.py` | **검색 품질** — 4개 모드(벡터/하이브리드/+리랭킹/+질의확장) 비교. Hit@1, MRR, ContextRecall, HardNegHit@1, 지연. |
| `sweep.py` | **하이퍼파라미터 스윕·ablation** — rerank_weight·alpha·idf_power·섹션 prior 값의 결정 근거를 표로 재현. |
| `faithfulness.py` | **답변 신뢰성** — AnswerGroundedness, CitationRate, AbstentionAccuracy, OverAbstain. |
| `pv_eval.py` | **PV 워크플로** — 라벨 22케이스로 중대성·기한·인과성·코딩(P/R/F1)·보고요건 정확도. |
| `verify_eval.py` | **검증기 자체 평가**(메타모픽) — 정상 답변에 결정론적 변조(수치 치환·오프셋·방향 뒤집기·고유어 치환·날짜 시프트·**날짜 역할 스왑**)를 가해 탐지율(6축)/오탐률 측정. |
| `stats.py` | Wilson 95% 신뢰구간 — 소표본에서 1.000을 과장하지 않기 위한 장치. |
| `qa_dataset.json` | 검색 평가 32문항(하드네거티브 14 + 섹션 prior 게이트 검증 2 포함). |
| `abstention_dataset.json` | 범위 밖 질문(회피가 정답인 문항). |
| `pv_dataset.json` | PV 라벨 22케이스(롱테일 구어를 일부러 포함해 재현율이 정직하게 1.0 미만). |
