# 파일 구조 안내 — 처음 이 프로젝트를 분석하는 사람을 위한 읽기 지도

> 이 문서는 RAPV-Assistant 데모(제약 RA·PV 업무용 RAG + MCP 에이전트)를 **처음 접하는 사람**이
> 어떤 순서로, 어떤 파일을 읽어야 전체를 가장 빠르게 이해할 수 있는지 안내한다.
> 각 파일이 "무엇을 담고 있고, 왜 존재하는지"를 함께 설명한다.
> 읽다가 낯선 기술 용어가 나오면 같은 폴더의 `dictionary.md`(용어 사전 — 선행
> 개념부터 순서대로 풀이)를 곁에 두고 볼 것.

## 0. 30초 요약 — 이 프로젝트는 무엇인가

제약회사의 **RA(인허가/규제업무)·PV(약물감시)** 담당자를 위한 AI 어시스턴트 데모다.
두 도메인을 함께 다룬다 — **RA**는 규제문서 검색·마감일·제출 체크리스트,
**PV**는 이상사례 케이스 접수부터 보고서 초안까지의 처리 워크플로.

- **RAG**: 규제문서를 근거·출처와 함께 검색한다 (하이브리드 검색 + 리랭킹 + 질의확장).
- **MCP**: 마감일 조회, 이상사례 트리아지, 보고서 초안 같은 RA·PV 업무 도구를 FastMCP 서버로 표준화했다.
- **Agent**: LLM(또는 오프라인 규칙 라우터)이 질문 의도에 맞는 MCP 도구를 골라 호출한다.
- **신뢰성 장치**: "AI가 조용히 틀리는 것"을 막는 5겹의 안전장치.
  상세 구현은 뒤의 각 장(①은 4장 에이전트·6장 redactor, ②는 4장, ③은 5장 retriever,
  ④·⑤는 7장 신뢰성 계층)에서 다루므로, 여기서는 각 장치가 **무엇을 막는지**만 파악하면 된다.
  1. **PII 마스킹** — 케이스 서술 속 개인정보(주민번호·전화·이름 등)를 외부 LLM API와
     로그로 나가기 전에 지운다. 에이전트 입구와 도구 인자, 두 겹에서 마스킹한다.
  2. **환각 회피(abstention)** — 근거가 부족한 질문에는 지어내는 대신 "근거 없음"으로 답한다.
  3. **버전 인지 검색** — 폐지된 규정은 기본 제외하고, "2024년 6월 기준으로는?"처럼
     시점(as_of)을 지정하면 **그 시점에 시행 중이던 버전**을 찾아준다.
  4. **답변 사후 검증 게이트** — 모든 답변에 대해 수치·날짜·방향 표현("이내"↔"이후")·
     날짜 역할(인지일↔마감일)이 검색 근거·도구 결과와 일치하는지 대조하고, 어긋나면
     경고를 붙인다.
  5. **배포 전 점검(preflight) + 운영 계기판** — 서버 기동 전에 데이터·설정은 물론
     안전장치 자체가 정상 작동하는지 검사해 실패 시 기동을 차단하고, 운영 중에는
     검증 경고율을 `/health` 계기판으로 집계한다.

  다섯 장치를 관통하는 원칙은 **조용한 실패 금지** — 틀린 입력을 조용히 무시하거나
  빈 결과로 얼버무리지 않고, 명시적인 에러·경고·caveat 로 시끄럽게 알린다.
- **평가/테스트**: 평가 스크립트 5종 + pytest 273케이스 + CI로 모든 주장을 수치로 재현하고,
  핵심 지표의 회귀는 pytest 가드가 CI 실패로 강제한다.
- **판정 기준의 명문화**: 이 프로젝트가 검증 관점에서 무엇을 '논리적 비약'과 '사각지대'로
  보고 제거해 왔는지의 기준은 `description/아키텍처.md` 7장에, 그 기준을 시스템 전체에
  재적용한 점검 기록은 같은 문서 7-4(v6: 20렌즈)·7-5(v7: 10렌즈)·7-6(v8)·7-7(v9)에 정리되어 있다.

## 1. 추천 읽기 순서 (전체 로드맵)

```
[1단계: 큰 그림]        README.md → description/아키텍처.md → docs/프로젝트_소개서.md
[2단계: 데이터]         data/regulations/*.md → data/ra_tasks.json
[3단계: 요청의 흐름]     src/api/main.py → src/agent/agent.py → src/mcp_server/server.py
[4단계: RA 도메인]      규제문서 검색 src/rag/ (loader → chunker → embedder → vectorstore
                                  → synonyms → retriever → pipeline 순서)
                        + RA 업무 도구 — 마감일·체크리스트 (src/ra/tasks.py 순수 로직 +
                                  server.py의 get_ra_deadlines·get_submission_checklist 래퍼
                                  + data/ra_tasks.json)
[5단계: PV 도메인]      src/pv/ (redactor → triage → causality → coding → report 순서)
[6단계: 신뢰성 장치]     src/verify/verifier.py → src/preflight.py → src/observability.py → src/config.py
[7단계: 증명]           eval/ → tests/ → .github/workflows/ci.yml
[8단계: 설계 근거]       description/아키텍처.md 6~9장 → docs/면접노트.md
```

**왜 이 순서인가**: 코드는 "요청이 흘러가는 길"을 따라 읽는 것이 가장 빠르다.
사용자의 질문은 `API → 에이전트 → MCP 도구 → (RA: 검색·마감일 또는 PV: 케이스 처리) → 검증 → 응답`
순서로 흐르므로, 큰 그림(1)과 재료(2)를 본 뒤 그 길(3)을 따라가고, 두 도메인 갈래
(4: RA, 5: PV)를 깊이 판 다음, 품질을 지키는 장치(6·7)와 설계 이유(8)를 마지막에 본다.

## 4. 3단계 — 요청의 흐름 (엔트리포인트부터)

### `src/api/main.py` (109줄) — 입구
- FastAPI 앱. `lifespan` 훅에서 부팅 시 RAG 인덱스를 1회 구축한다.
- 엔드포인트: `/chat`(질문→답변), `/health`(모드·인덱스 상태 + **검증 게이트 경고율 계기판**
  `verification_gate`), `/api/deadlines`, `/`(챗 UI 서빙).
- `ChatRequest`/`ChatResponse` Pydantic 스키마. 응답에 `citations`(출처), `tool_calls`(도구 트레이스),
  `trace`(스텝별 지연), `grounded`, `redactions`(PII 마스킹 건수), `verification`(사후 검증 결과)이 실린다.
- **읽는 이유**: 시스템의 계약(입출력)이 여기 다 정의돼 있다.

### `src/agent/agent.py` (857줄) — 두뇌
이 프로젝트에서 가장 중요한 파일. 순서대로:
1. **입구 PII 마스킹**: 메시지·이력 모두 `redact()` — 이후 모든 경로(LLM API·로그)에 개인정보가
   안 나간다. 이력은 문자열 content 만이 아니라 **anthropic 블록 리스트 표기**
   (`content=[{"type":"text",…}]`)까지 마스킹한다(문자열 표기만 막던 표기 우회로의 봉합 — v7).
2. **모드 분기**: `ANTHROPIC_API_KEY`가 있으면 **LLM 모드**(Claude tool-use 루프, 최대 6스텝,
   도구 에러는 `is_error`로 되먹여 자가복구), 없으면 **오프라인 모드**(규칙 라우터).
3. **오프라인 라우팅** `_route_intent()`: 케이스 서술→PV 도구, 마감 어휘→마감일 도구,
   기본→규제문서 검색. **추측하지 않는 라우팅**: 체크리스트 카테고리는 확신 있는 신호가
   있을 때만 매칭하고(`_guess_category` — 기본 추측 없음), 미매칭이면 지원 목록 안내 +
   검색 폴백(도구의 error+available 계약을 라우터가 우회하지 않는다). 이력 조회 의도는
   명시적 구문(`_HISTORY_MARKERS`: 예전/구판/개정 이력…)만 좁게 감지해 폐지본 포함
   검색으로 라우팅 — 단독 "이력"·"폐지"는 현행 질문과 겹쳐 **버전 경고를 끄는 오탐**이
   되므로 넣지 않는다.
4. **Abstention**: `SCORE_FLOOR=0.19`(근거 관련도)와 `COVERAGE_FLOOR=0.28`(질의 커버리지)
   **둘 다** 미달일 때만 "근거 없음"으로 회피(AND 조건 — 과회피 방지). 문턱값 보정 근거가 주석에 있다.
5. **멀티턴**: `_resolve_followup()` — "그럼 그건?" 같은 후속질문을 직전 발화와 병합.
   검증 라벨링도 멀티턴을 안다: `_question_context()`가 직전 사용자 발화까지를 '질문 전제'
   창으로 잡아, 이전 턴 전제를 이어받은 수치가 '환각'이 아니라 '전제 확인 필요'로
   라벨링된다(직전 턴까지만 — 전체 이력 누적은 완화 라벨의 면적을 무한정 키운다).
6. **신뢰 소스의 규율**: 도구 출력은 검증의 신뢰 소스가 되지만 두 가지는 제외 —
   사용자 입력 에코 필드(`_strip_query_echo`: query·as_of)와 **에러 계약 응답**
   (`_is_contract_error`: 에러 문구에도 사용자 입력이 에코된다). 사용자 전제가 '근거'로
   승격되는 구멍을 필드 단위·응답 단위 양쪽에서 막는다. 케이스 서술 에코(case)는
   **2계층으로 분리**(`_split_user_facts`) — 지지 근거로는 인정하되(재서술은 정당),
   그 계층에서만 지지되는 클레임은 from_case(case_origin) 라벨로 노출한다("케이스는
   사실"이 "규정 클레임의 근거"로 비약하는 지점의 가시화). 이력(as_of) 검색의 폐지본
   인용 허용은 전역 스위치가 아니라 **그 검색이 실제 반환한 doc_id 집합**으로 좁힌다
   (이력 턴 동안 버전 축이 통째로 꺼지는 것을 방지).
7. `_finalize()`: 출처 수집·dedupe, 사후 검증 게이트 호출, 검증 결과의 **계기판 집계·감사
   로그**(`record_verification`), 응답 조립.

### `src/mcp_server/server.py` (389줄) — 도구 상자
- FastMCP 서버. MCP 3대 primitive를 모두 구현. **얇은 래퍼 계층**이다 — 마감일·체크리스트의
  조회 로직은 `src/ra/tasks.py`(5장 참고)에, 검색은 `src/rag/`, 케이스 처리는 `src/pv/` 에
  있고, 이 파일은 인자 마스킹과 MCP 노출(스키마·docstring·에러 계약)을 맡는다:
  - **Tools 6종**: `search_regulations`(RAG 검색), `get_ra_deadlines`, `get_submission_checklist`,
    `assess_adverse_event`(PV 트리아지+인과성+코딩), `draft_ae_report`(ICSR 초안),
    `list_regulation_documents`
  - **Resource**: `regulation://{doc_id}` (문서 원문 열람)
  - **Prompt**: `pv_case_intake` (케이스 처리 SOP를 클라이언트 무관하게 배포)
- 각 도구의 **docstring이 곧 LLM용 사용설명서**다 — "언제 이 도구를 쓰고 언제 다른 도구를 쓸지"가 명시돼 있다.
- **도구의 실패 방향**: 존재하지 않는 필터·카테고리에는 빈 결과("마감 없음" — 자신 있는 오답)가
  아니라 `{"error", "available": [...]}` 를 반환한다 — 에이전트가 available 을 보고 스스로
  정정 재시도하는 계약(조용한 빈 결과 금지). 형식이 틀린 `as_of` 도 같은 계약이다
  (`{"error", "expected"}`) — 조용히 무시하고 현행 기준으로 답하면 사용자는 '그 시점 규정'을
  받았다고 믿게 된다. 에러 문구에는 예시 '날짜'를 넣지 않는다(에러 응답이 텍스트로 소비될
  때 예시 날짜가 실제 날짜 클레임처럼 읽히는 것을 차단).
- **도구 계층 PII 심층방어**: 자유 텍스트 인자는 **전부** 마스킹한다 — PV 도구의
  `case_description`·`reporter`·`patient_info` 는 물론, `search_regulations` 의 `query`
  (voyage 임베더 구성 시 외부 API 로 송신되는 경로), 필터 인자(`task_type`·`category` —
  미매칭 에러 문구에 에코됨), Prompt `pv_case_intake` 의 케이스 서술까지. stdio 단독 사용
  시 인자는 에이전트 입구 마스킹을 거치지 않고 직접 들어오기 때문(우회로 차단 — PV 도구
  2종만 마스킹하던 범위를 v7 에서 전 도구·Prompt 로 확장).
- 인메모리(`fastmcp.Client(mcp)`, 데모 기본)와 stdio(`python -m src.mcp_server.server`,
  Claude Desktop/Cursor 연결) 두 방식을 지원한다.

## 5. 4단계 — RA 도메인: RAG 파이프라인 (`src/rag/`) + RA 업무 도구

RA 담당자의 업무 축은 두 개다: **① 규제문서 검색**(이 장의 RAG 파이프라인)과
**② 마감일·제출 체크리스트 관리** — ②의 구현은 **`src/ra/tasks.py`(90줄, v9 신설)**:

## 6. 5단계 — PV 도메인: PV 워크플로 (`src/pv/`)

OpenTelemetry 개념의 무의존성 구현. |
| `src/config.py` (48줄) | 모든 하이퍼파라미터가 환경변수로 제어: `CHUNK_SIZE=500`, `CHUNK_OVERLAP=80`, `RETRIEVE_TOP_K=8`, `RERANK_TOP_N=3`, `HYBRID_ALPHA=0.5`, `RERANK_WEIGHT=0.9`, 섹션 prior 페널티(0.3/0.055), `EMBEDDER_KIND`, `QUERY_EXPANSION`. 값의 근거는 `eval/sweep.py`로 재현. |
| `web/index.html` | 단일 페이지 챗 UI. 출처·도구 트레이스·지연·PII 마스킹·검증 배지 표시, 멀티턴 지원. |

## 8. 7단계 — 평가와 테스트 ("측정 없이 신뢰 없음")

### `eval/` — 평가 스크립트 5종 + 데이터셋

| 파일 | 무엇을 측정하나 |
|---|---|
| `evaluate.py` | **검색 품질** — 4개 모드(벡터/하이브리드/+리랭킹/+질의확장) 비교. Hit@1, MRR, ContextRecall, HardNegHit@1, 지연. |
| `sweep.py` | **하이퍼파라미터 스윕·ablation** — rerank_weight·alpha·idf_power·섹션 prior 값의 결정 근거를 표로 재현. |
| `faithfulness.py` | **답변 신뢰성** — AnswerGroundedness, CitationRate, AbstentionAccuracy, OverAbstain. |
| `pv_eval.py` | **PV 워크플로** — 라벨 22케이스로 중대성·기한·인과성·코딩(P/R/F1)·보고요건 정확도. |
| `verify_eval.py` | **검증기 자체 평가**(메타모픽) — 정상 답변에 결정론적 변조(수치 치환·오프셋·방향 뒤집기·고유어 치환·날짜 시프트·**날짜 역할 스왑**·**부분 날짜 시프트**·**부분 날짜 역할 스왑**·**하이픈 범위**)를 가해 탐지율(**9축 93건**)/오탐률(4축) 측정. 신뢰 소스 뷰는 운영(agent)과 동일한 2계층(케이스 에코 분리). **측정 경계도 명시**: 고유어 방향·케이스 간섭 축은 eval 이 아니라 preflight 자가 테스트·단위 테스트가 고정한다(표본 0~1의 축을 eval 지표로 두면 '측정의 조용한 수축'과 구분되지 않는다). 이 수치들의 '핀'은 `tests/test_verify_eval.py` 가 **CI 실패로 강제**한다(스크립트 자체는 exit 0 — 측정과 강제는 다른 명제). |
| `stats.py` | Wilson 95% 신뢰구간 — 소표본에서 1.000을 과장하지 않기 위한 장치. |
| `qa_dataset.json` | 검색 평가 32문항(하드네거티브 14 + 섹션 prior 게이트 검증 2 포함) — 오류 분석 루프에 쓴 **개발셋**. |
| `holdout_dataset.json` | **홀드아웃 6문항** — 튜닝에 쓰지 않는 일반화 근사용. v9 에서 복수 정답 문항의 라벨 모호성만 `accept_doc_ids` 로 교정(검색기 튜닝 없음 — 홀드아웃 규율 유지). |
| `abstention_dataset.json` | 범위 밖 질문(회피가 정답인 문항). |
| `pv_dataset.json` | PV 라벨 22케이스(롱테일 구어를 일부러 포함해 재현율이 정직하게 1.0 미만). |

### `.github/workflows/ci.yml`
위치 주의: 이 파일은 `project/` 안이 아니라 **저장소 루트의 `.github/workflows/`** 에 있다

(`test_pv_eval.py`·`test_verify_eval.py`)가 CI 실패로 강제한다 — 실행과 강제의 구분.
