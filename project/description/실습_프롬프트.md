# 실습 프롬프트 8선 — 흐름 로그로 시스템 파악하기

> RAPV-Assistant(RA·PV 도구)의 장치들을 하나씩 발동시키는 실습 세트.
> 단순한 경로부터 시작해 점점 많은 장치가 발동하는 순서라, 앞 로그와 비교하며
> "이번엔 뭐가 달라졌나"를 보기 좋다.
>
> **사용법:** 서버를 띄우고(`./run.sh`) 터미널(흐름 로그)과 브라우저를 나란히 놓는다.
> 프롬프트를 하나 보낼 때마다 "로그에서 볼 것"을 확인하고 다음으로 넘어간다.
> 각 로그 줄 끝의 `▶ 다음:` 이 "왜 다음 호출이 일어나는가"의 인과 설명이다.

---

## 1. 규정 검색 — 가장 기본 경로

```
품목허가 심사 처리기간은 얼마나 걸리나요?
```

- **경로:** `search_regulations` 하나만 실행 → RAG 검색(동의어 확장 → 하이브리드 → 리랭킹) → 답변
- **로그에서 볼 것:** `top=REG-XXX ... (점수)` — 어떤 문서가 1등으로 뽑혔는지.
  LLM 모드라면 step 0(도구 요청) → step 1(답변 확정)의 2왕복 구조.
- **확인 포인트:** 이후 모든 프롬프트의 기준선. `grounded=True`, citations 개수를 기억해 둘 것.

## 2. 마감일 조회 — 검색이 아닌 결정론 도구

```
이번 주 마감인 규제 업무 일정 알려줘
```

- **경로:** `get_ra_deadlines` — RAG 를 거치지 않는다
- **로그에서 볼 것:** `within_days=30` 파라미터. 그리고 pipeline/retriever 로그가
  **안 나타나는 것** — 검색 도구와 조회 도구의 차이가 로그에서 바로 보인다.

## 3. 체크리스트 — 정상 경로와 에러 계약 비교 (2회 전송)

```
변경허가 체크리스트 알려줘
```

이어서:

```
GMP 체크리스트 줘
```

- **경로:** `get_submission_checklist` — 첫 번째는 정상, 두 번째는 미지원 카테고리
- **로그에서 볼 것:** 두 번째 요청에서 `error+available` 계약 발동.
  오프라인 모드는 지원 목록 안내 + 검색 폴백, LLM 모드는 Claude 가 에러를
  되먹임받아 **스스로 다른 도구/인자로 재시도**하는 것을 비교.
- **확인 포인트:** "조용한 빈 결과 금지" 원칙이 실제로 어떻게 동작하는가.

## 4. PV 트리아지 + PII 마스킹 — 중첩 도구 호출

```
환자 김민수님(010-1234-5678)이 약 복용 후 아나필락시스로 입원했습니다. 언제까지 보고해야 하나요? 인지일은 2026-07-10입니다.
```

- **경로:** 마스킹 → `assess_adverse_event` → 그 **안에서** `search_regulations` 중첩 호출(근거 부착)
- **로그에서 볼 것:** 흐름 02의 `masked=[전화번호 1, 이름 1]` — 이후 모든 로그의
  메시지가 `[이름]님`, `[전화번호]` 로 바뀌어 있는지 추적. 도구가 도구를 부르는
  중첩 구조는 이 프롬프트에서만 보인다.
- **확인 포인트:** `deadline_date` 가 인지일+15일(중대 기준)로 계산되어 나오는가.
  이 계산값이 검증의 신뢰 소스가 된다.

## 5. ICSR 보고서 초안 — 가장 긴 도구 체인

```
45세 남성 환자가 A정 복용 후 아나필락시스로 입원했습니다. 보고자는 담당 의사입니다. KAERS 보고서 초안 만들어줘. 인지일은 2026-07-10입니다.
```

- **경로:** `draft_ae_report` — 최소보고요건(ICH E2D 4요소) 검증 → 트리아지 → 인과성 → 코딩 → 초안 조립
- **로그에서 볼 것:** `reportable=True/False, missing=[...]`.
- **변형 실험:** 환자 정보나 보고자를 빼고 다시 보내면 `missing` 에 누락 요소가
  나타나고 답변에 보완 안내(followups)가 붙는다 — 최소요건 검증의 동작 확인.

## 6. 범위 밖 질문 — 환각 방지(abstention)

```
오늘 점심 메뉴 추천해줘
```

- **경로:** 검색 시도 → 근거 부족 판정 → 회피
- **로그에서 볼 것:** 오프라인 모드는 점수·커버리지가 문턱(SCORE_FLOOR 0.19 ·
  COVERAGE_FLOOR 0.28) 아래라 "근거를 찾지 못했습니다" + `grounded=False`.
  LLM 모드는 Claude 가 도구 없이(또는 빈 검색 후) 답해 `grounded=False`.
- **확인 포인트:** 1번 프롬프트와 마지막 로그의 `grounded=` 값 비교 — 근거 배지의
  의미가 여기서 체감된다.

## 7. 이력 조회 — 버전 인지 검색 + 경고 면제

```
이상사례 보고 규정의 개정 전 버전은 어떤 내용이었나요?
```

- **경로:** `search_regulations(include_superseded=True)` — 폐지본(REG-013) 포함 검색
- **로그에서 볼 것:** 검색 로그의 `include_superseded=True` 파라미터, 결과에 구판 포함.
- **확인 포인트:** 원래라면 검증 게이트에서 "폐지본 인용" 경고가 붙어야 하는데,
  이력 조회가 실제로 반환한 문서라서 면제되어 `ok=True` — 안전장치를 끄는 범위를
  문서 단위로 좁힌 설계가 동작하는 순간.

## 8. 멀티턴 후속질문 — 대화 맥락 병합 (7번에 이어서, 새로고침 금지)

```
그럼 현행은?
```

- **경로:** 브라우저가 보낸 history 와 병합 처리
- **로그에서 볼 것:** 오프라인 모드는 `_chat_offline` 로그의 `resolved=` 가 직전
  질문과 병합된 문장(`_resolve_followup`). LLM 모드는 `messages=3` 처럼 대화록
  길이가 커진 상태로 step 0 시작.
- **주의:** history 는 **브라우저 메모리**에 있다(최근 3턴). 새로고침하면 사라져서
  병합이 일어나지 않는다 — 서버는 무상태(stateless).

## 부록 — 멀티턴 원리 심화 세션 (한 세션에서 연속 6턴, 새로고침 금지)

history·검증 라벨·기억의 한계까지 보려면 아래를 순서대로 한 세션에서 보낸다.

| 턴 | 프롬프트 | 로그에서 볼 것 |
|---|---|---|
| 1 | `의약품 변경허가 처리기한은 며칠인가요?` | `history_turns=0` (기준선). 답변의 기한 숫자를 기억해 둘 것 |
| 2 | `그럼 변경신고는?` | `history_turns=2`. 오프라인: `resolved=` 병합 / LLM: `messages=3` |
| 3 | `중대한 이상사례 보고 기한이 30일 맞지?` (실제 기한과 **다른** 숫자로) | `_finalize` `summary=` 의 `question_origin` — 틀린 전제의 재서술을 '환각'이 아니라 '전제 확인 필요'로 구분(LLM 모드에서 뚜렷) |
| 4 | `품목허가 준비 체크리스트 알려줘` | `history_turns=6` — 3턴(6개) 상한 도달. 다음 턴부터 오래된 턴이 밀려남 |
| 5 | `그럼 그건 언제까지?` | 병합 상대가 턴 4(체크리스트)다 — `_resolve_followup` 은 **직전 발화 하나만** 잇는다(오프라인 근사의 경계) |
| 6 | `맨 처음에 내가 물어본 질문이 뭐였지?` | 턴 1은 이미 history 에서 밀려남 — 기억은 브라우저의 최근 3턴이 전부임이 드러난다 |

이 세션으로 확인되는 원리: ① history 는 턴마다 2개씩 자란다(user+assistant),
② 오프라인은 직전 발화 병합이라는 근사를 쓴다(턴 2·5), ③ 검증 게이트는 질문에서
온 수치를 환각과 구분한다(턴 3), ④ 기억은 브라우저의 최근 3턴이 전부다(턴 4·6).
