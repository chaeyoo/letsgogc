# 자소서 문항 2 — 중도 포기할 수 있었지만 완수한 경험

> **문항:** 중도에 포기할 수도 있었지만 노력해서 완수한 경험
> ① 그 일을 맡게 된 이유 / 어려웠던 이유
> ② 포기하고 싶던 순간 / 극복하기 위해 한 노력 / 그 과정에서 느낀 점
> ③ 과제 수행 결과 / 이를 통해 느낀 바 (구체적으로)
>
> **소재:** 마크로젠 LIMS — 유럽·싱가포르 법인 독립운영 프로젝트 (팀장 공백으로 인수)
> **관통 가치:** ⭐ **창의도전** (+ 두려움 없는 혁신)
> **분량:** 1000자 (아래 초안 공백 포함 약 950자 — 지원 폼 카운터로 최종 확인)

---

## ✍️ 초안 (완성본 · 필요시 수정)

**① 그 일을 맡게 된 이유 / 어려웠던 이유**

마크로젠에서 유전자 분석 실험실 정보관리 시스템(LIMS)을 개발하던 중, 유럽·싱가포르 법인이 독립적으로 실험을 운영할 수 있도록 시스템을 확장하는 과제를 맡았습니다. 기존에는 해외 법인의 실험 결과 파일이 모두 한국 본사로 모여 분석까지 한국에서 처리했는데, 이를 각 법인이 자체적으로 분석하고 결과를 산출하도록 바꾸는 일이었습니다. 그런데 프로젝트를 이끌던 팀장님이 갑작스럽게 퇴사하시면서, 저는 이 과제를 통째로 떠안게 되었습니다. 단순한 화면 개발이 아니라 실험 결과 파일을 받아 분석하고 결과를 산출하는 전체 프로세스를 아우르는 작업이었고, 낯선 레거시 Java 기반이라 웹 개발자로 입사한 제게는 처음부터 벅찬 일이었습니다.

**② 포기하고 싶던 순간 / 극복 노력 / 느낀 점**

가장 포기하고 싶던 순간은, 제대로 된 인수인계도 없이 방대한 레거시 코드와 생소한 유전자 분석 도메인을 동시에 마주했을 때였습니다. 실험 프로세스를 이해하지 못하니 코드가 무엇을 하는지조차 파악되지 않았습니다. 하지만 저는 물러서는 대신 접근 방식을 바꿨습니다. 먼저 실험실·현업 담당자에게 실험 결과 파일이 어떻게 생성되고 분석되는지를 직접 묻고 전체 흐름을 도식화했습니다. 그 위에서 레거시 코드를 기능 단위로 분해해 이해했고, 웹 개발자를 넘어 Java 개발자로서 백엔드 로직까지 직접 손봤습니다. '코드'가 아니라 '현업의 업무'를 먼저 이해하자 얽혀 있던 문제가 풀리기 시작했고, 낯선 영역도 두려워하지 않고 구조적으로 접근하면 돌파할 수 있다는 것을 배웠습니다.

**③ 과제 수행 결과 / 느낀 점**

결국 유럽·싱가포르 법인이 한국 본사를 거치지 않고 자체적으로 실험 결과를 분석·산출하는 독립 운영 체계를 완성했습니다. 【결과 보강: 예—오픈 시점, 본사 처리 물량/분석 대기시간 감소 등 수치】. 이 경험은 제게 "맡은 범위가 원래 역할을 넘어서더라도, 현업을 이해하고 새로운 방법으로 도전하면 끝까지 완수할 수 있다"는 확신을 남겼습니다. 이는 GC녹십자가 지향하는 **창의도전**의 가치, 그리고 현업에 직접 들어가 문제를 풀어내는 Forward Deployed Engineer의 태도와 맞닿아 있다고 생각합니다.
