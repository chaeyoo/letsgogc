# 자소서 문항 2 — 중도 포기할 수 있었지만 완수한 경험

> **문항:** 중도에 포기할 수도 있었지만 노력해서 완수한 경험
> ① 그 일을 맡게 된 이유 / 어려웠던 이유
> ② 포기하고 싶던 순간 / 극복하기 위해 한 노력 / 그 과정에서 느낀 점
> ③ 과제 수행 결과 / 이를 통해 느낀 바 (구체적으로)
>
> **소재:** 마크로젠 유전자 분석 실험실 정보관리 시스템(LIMS) 풀스택 개발 (레거시 Java)
> **관통 가치:** ⭐ **창의도전** (+ 두려움 없는 혁신) — GC 핵심가치와 연결
> **분량:** 1000자

---

## 🎯 작성 원칙 (포기 순간이 많을 때)
- 포기하고 싶던 순간을 **여러 개 나열하지 말 것.** → **창의도전을 가장 잘 증명하는 한 장면**만 클로즈업.
- "버텨서 완수"보다 **"새로운 방법으로 돌파"** 를 강조 (단순 인내 < 창의적 도전).
- ③ 결과는 반드시 **구체적으로**(수치/증빙/이후 변화).

---

## ✍️ 초안 (1000자 목표 · 【 】는 본인이 채울 부분)

**① 맡게 된 이유 / 어려웠던 이유**

유전자 분석 기업 마크로젠에서 실험실 정보관리 시스템(LIMS)의 풀스택 개발을 맡았습니다. 【맡게 된 이유: 예—입사 초기 담당자 공백/신규 요건 대응 등】. 이 일은 두 가지 면에서 어려웠습니다. 첫째, 시스템이 **오래된 레거시 Java 기반**이라 문서도 부족한 코드를 해석하며 손대야 했습니다. 둘째, 다루는 대상이 **유전자 분석이라는 생소하고 복잡한 실험 도메인**이어서, 실험 프로세스를 이해하지 못하면 기능 하나도 제대로 만들 수 없었습니다.

**② 포기하고 싶던 순간 / 극복 노력 / 느낀 점**

가장 포기하고 싶었던 순간은 【구체적 장면: 예—특정 실험 데이터 흐름이 얽혀 원인 모를 오류가 반복되던 때 / 레거시 코드의 숨은 의존성으로 수정이 계속 실패하던 때】였습니다. 【그때의 심정 한 줄】.

하지만 저는 물러서는 대신 방식을 바꿨습니다. 【극복 노력을 창의도전으로: 예—실험실 담당자에게 직접 실험 프로세스를 묻고 업무 흐름을 도식화해 이해했다 / 레거시 코드를 기능 단위로 분해해 재구조화했다 / 테스트로 원인을 좁혀갔다】. 코드만 보던 시야를 **"현업의 실제 업무를 먼저 이해하는" 방향**으로 돌리자 문제가 풀리기 시작했습니다.

이 과정에서 **낯선 도메인과 오래된 시스템도, 두려워하지 않고 구조적으로 접근하면 돌파할 수 있다**는 것을 배웠습니다.

**③ 결과 / 느낀 점**

결국 【결과: 예—해당 기능/시스템을 기한 내 안정적으로 완수, 실험 데이터 처리 오류 OO 감소, 담당자 업무시간 단축 등 수치·증빙】. 이 경험은 제게 **"어려운 문제일수록 현업을 이해하고 새로운 방법으로 도전한다"** 는 원칙을 남겼습니다. 이는 GC녹십자가 지향하는 **창의도전**의 가치, 그리고 현업에 직접 들어가 문제를 푸는 Forward Deployed Engineer의 태도와 맞닿아 있다고 생각합니다.

3. **② 극복 행동** — 실제로 뭘 했는지 (현업 인터뷰? 코드 재구조화? 테스트 전략? 야근 아닌 '방법의 전환'을 강조)
4. **③ 결과 수치·증빙** — 완료 시점, 개선 지표, 이후 시스템이 어떻게 쓰였는지
